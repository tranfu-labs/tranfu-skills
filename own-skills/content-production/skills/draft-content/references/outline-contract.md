# 共享大纲契约

生成一份同时约束 A/B 的平台中立共享大纲。大纲批准前停止创作，不生成母稿或平台稿。

## 必填内容

在 `shared-outline.vNNN.md` 中清楚写出：

1. **目标读者**：本篇具体面向谁，已知水平、需求与使用情境是什么。
2. **核心承诺**：读完可获得什么判断、解释或行动价值；不得超出上游事实边界。
3. **开头钩子**：要建立的问题、冲突或场景；不得虚构人物、经历、数据或结果。
4. **章节任务**：每节要回答的问题、核心论点和在全文中的作用。
5. **证据锚点**：每节可使用的 Claim ID、来源定位或等价材料片段，以及必须继承的限定。
6. **反方或限制**：主要异议、冲突证据、适用边界和仍未知的部分。
7. **读者价值**：本节如何帮助读者理解、判断或行动。
8. **转场**：上一节如何自然导向下一节。
9. **结尾方向**：要收束的结论、边界和单一行动方向；不预写标题池或发布 CTA。
10. **五平台适配提示**：分别写公众号、小红书、知乎、微博和头条号的开头、取舍与原生结构重点。

章节数量服从内容需要和上游内容形式；不要为了套模板机械凑节。

## 生成与优化边界

没有用户大纲时，只基于获准选题、受众、材料、证据和平台提示生成大纲。

有用户大纲时，只做结构优化：允许重排、合并或拆分章节，补清章节任务、证据锚点、限制、转场和平台提示。不得改变主题、核心角度、事实强度或事实边界；不得把大纲里的无来源断言当作新事实。若修复需要实质改变核心承诺，先返回 `NEEDS_UPSTREAM`。

## 版本、批准与哈希

- 首版写入 `01-outline/shared-outline.v001.md`；后续修订依次使用 `v002`、`v003`，不得覆盖旧版。
- 生成新版本后，把 gate 设为 `awaiting_approval`，绑定该版本的 run 相对路径与 SHA-256，并停止。
- 绑定等待批准后，向用户同时展示绝对路径和当前 SHA-256。只把用户明确指向这组 path+hash 的批准视为有效批准；批准命令必须回传该 expected hash。批准后记录 `approved_at`，状态进入 `DRAFTING`。
- 每次恢复先重算批准大纲 SHA-256。哈希变化时把 gate 标记为 `drifted`，不得继续使用旧批准。
- 已批准大纲需要修订时，保留旧文件和历史记录，创建新版本，重新等待批准，并使已有母稿、平台稿和 QA 失效。
- 所有母稿和平台稿都绑定当前批准大纲的同一个 SHA-256；不得从文件名推断有效版本。
