## 一、问题分析

我们让某主流闭源大模型帮客户清洗一张订单表，目标是把 90 天前的 `status='pending'` 行迁到归档表。模型给出的 SQL 如下：

```sql
UPDATE orders
SET status = NULL
WHERE created_at < NOW() - INTERVAL '90 days'
  OR status = 'pending'
;
```

在 staging 上跑了一遍，看到的输出是：

```
UPDATE 8432
```

数字看着合理，我们准备 merge 到生产。DBA 老 W 在 commit 前 30 秒拦下来——那个 WHERE 子句少了括号，OR 把它拆成了两个独立子句，到生产 200 万行里会命中 180 万行。

经过深入分析，我们最终定位到问题的核心在于 prompt 设计不当，导致模型在 SQL 优先级这种边界 case 上没有做防御性输出。

## 二、解决方案

针对上述问题，我们从根本上重构了整个数据清洗 pipeline，引入了多重校验机制，并对 prompt 进行了全面优化。具体动作包括：

1. 在调用大模型前增加 SQL 静态分析层。
2. 把模型的输出强制塞进一个 `EXPLAIN` 检查环节，比对 estimated rows 与预期数量级。
3. 重写了 prompt 模板，加入了"必须显式给出 WHERE 括号"的硬要求。

我们把 prompt 改成了这样：

```
你是一个 SQL 助手。请根据下面的 schema 写一段 UPDATE 语句。
要求：所有 WHERE 子句中的 AND/OR 必须显式加括号；
不允许出现没有括号的 OR 顶层组合。
```

优化后调用成本下降了 73%，准确率提升了 41%，效果显著。
