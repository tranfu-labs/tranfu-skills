## 这次对话到底在做什么（人话版导览）

> 给非作者读者的导览。看完知道：用户为什么找 AI、AI 用了什么方法、目前走到哪一步、产出物各是什么、下一步打算做什么。

### 1. 用户的真实目标

用户想写一个**系列文章**，把"工作 5 年左右、非计算机行业"的人，从 0 一路带到"会用 skill"、再到"会发布自己的 skill"。

但他这次找 AI **不是要写正文**，而是要把三件**前置工作**做对：

1. **文章风格** — 写出来读起来是什么感觉
2. **单篇形式** — 一篇文章长什么样、有哪些固定结构
3. **系列目录** — 整个系列怎么分篇、先讲什么后讲什么

他还顺手丢了一篇自己已经写好的样章 `docs-1.md`，里面用"军师"类比 LLM、用编号小节讲 Agent/MCP/RAG —— 这等于已经给 AI 一个**风格锚点**。

### 2. AI 用了什么方法

AI 没直接动手写目录，而是**先调用了本地的 `goal-driven-decomposition` skill**。这个 skill 的核心思想是：

> 在动手做之前，先用一组**文档**把"用户真正要的是什么、给谁看、长什么样"钉死，下游每一个干活的 sub-agent 都拿这些文档当唯一参照系，不再向用户反复确认。

具体走的协议：

1. **surface** — 把用户原话一字不改地存档（`user_goal_surface.md`），作为后续所有判断的"审计镜"
2. **true_goal** — 在不偏离原话的前提下，提炼出可被下游引用的目标声明（`00-true-goal-r1.md`）
3. **persona** — 起一个 sub-agent 把"目标读者"写成一张可以**角色扮演**的卡（`00-用户画像-r1.md`），下游任何 agent 都能"假装自己是这个人"去判断"这一段她读得下去吗"
4. （未进行）DoD + GoalTree + doc-relay

### 3. 中途的两次方向修正

这一段对话最有看头的是**两次 push-back**，能看到 AI 是怎么被用户"拨回正轨"的：

| # | AI 第一版的判断 | 用户的反应 | 修正结果 |
|---|---|---|---|
| 1 | persona 用"互联网 in-house 法务"做台面职业 | "法务？专有名词太多，其它专业人看不懂" | 换成**公众号小编 / 内容编辑**（每个动作普通人 3 句话能听懂） |
| 2 | 系列粗估 5-7 篇平铺 | "按最终目标阶段性来，基础 4-5 篇 + 高手 4-5 篇，每阶段有收尾产物" | persona 卡里写死硬约束：**必须两阶段，每阶段开头明示'读完你将得到 X'** |

第 2 个修正特别关键 —— 它把"系列结构"从一个**美学问题**变成了"**阶段性交付物**"的工程问题，后面所有目录设计都得围着两个具体产物转（一个粗但能用的 skill + 一个打磨好可发布的 skill）。

### 4. 下一步会发生什么

会话停在用户说"看一下再 OK"。下一步：

1. 用户审完 persona r1
2. 并行起两个 sub-agent：
   - **DoD agent** —— 给"风格 / 形式 / 目录"三个维度各定义清楚"做到什么程度算完事"
   - **GoalTree agent** —— 把"两阶段、每阶段一个具体收尾产物"钉成一棵目标树
3. 进 **doc-relay** —— 一棒一棒接力出"风格手册 / 单篇模板 / 系列目录"三份正式交付物

整套流程的好处：每一步交出去的都是**文档**，不是聊天记录，所以即便换一个新会话、换一个 agent，也能凭这些文档把任务接着推下去。

### 5. 为什么"林姐"这张卡值得单看一眼

哪怕你不关心 skill，`00-用户画像-r1.md` 本身也是一个很好的 **persona 写法样本**。它有几个非常规的设计：

- **modal 的选择有交代** —— 为什么选小编而不选别的，理由写在卡的最顶上
- **"她对什么会皱眉"做成了表** —— 看到 `git clone` / 看到 `.json` / 看到代码块各自的反应一一列出，下游 agent 能直接当 checklist 用
- **自检题** —— 卡的最后写了 7 道问题，"两个不同 agent 拿这张卡作答，如果答案严重分歧，说明卡还太薄"。这是给 persona 卡本身做的回归测试。

**一句话总结**：用户没让 AI 写文章，而是先让 AI 帮自己想清楚"给谁写、写成什么调、整个系列怎么排"，且这套思考的中间产物全部沉淀成了文档，不会随着会话关闭就丢失。
