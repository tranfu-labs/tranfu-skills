# Skill 任务域框定

## 目的

当需要基于原始经验、文档、事故记录、issue 记录、PR 记录或 guardrail 创建新的 Codex skill，或把尚未成型材料转成新 skill 前，先使用这个 skill。

IMPORTANT: 输出 MUST 是一个 **skill 框架决策**，而不是最终的 `SKILL.md`。NEVER 在 framing 步骤中编写最终 `SKILL.md`，unless 用户已经接受框架并明确要求继续进入实现阶段。

## 核心规则

MUST 优先选择用户期望结果所在的抽象轴，而不是事故里偶然出现的词、文件、平台、工具、标签、接口或代码位置。

MUST 始终询问：

```text
如果这条经验不按事故里的名词命名，而是按它要保护的用户结果命名，它应该叫什么？
```

如果候选名称只是描述“改了什么对象”或“在哪个位置改”，MUST 继续上提一层，寻找用户真正想保证的可观察结果。

NEVER 让事故里的排查路径自动成为 skill 的主流程。能转成“怎样一开始写对”的经验，MUST 优先写成正向 authoring / maintenance 工作流；把探针、日志、缓存绕过、回滚等手段放到验证或故障排查分支。

## 工作流

CREATE A TODO LIST FOR THE TASKS BELOW before starting. TODO LIST MUST 覆盖步骤 1-9 和失败路径检查，并在每一步完成后更新状态。

1. 阅读源材料，直到能列出实际失败、规则或工作流中的至少一种；如果材料不足，进入失败路径。
2. 提取原始经验：
   - 现象
   - 可能原因
   - 模型常见错误
   - 正确动作
   - 验证方法
   - 适用条件
   - 不应泛化的地方
3. 识别表层名词：
   - 具名产品、供应商、工具、文件、标签、API、端点、目录、命令、数据结构或代码位置
4. 识别用户期望结果：
   - 用户想保证什么
   - 哪些可观察行为能证明成功
   - 这属于哪类重复任务
5. 把经验按层级拆开：
   - 正常路径：一开始应该如何创建或维护正确产物
   - 验证路径：如何证明产物满足要求
   - 排障路径：失败后才需要的探针、日志、缓存、回滚或对比方法
6. 选择并比较抽象轴。至少生成 4 个候选 skill 容器，分别覆盖：
   - 平台 / 供应商轴
   - 实现对象轴
   - 代码位置轴
   - 用户结果轴
7. 对候选项做决策前比较：
   - 过窄：只会因某一个产品、供应商或事故名词触发
   - 过于实现导向：绑定在修复是怎么完成的，而不是重要结果是什么
   - 过宽：会吸收成功标准不同的相邻任务
   - 过于排障导向：把一次事故中的诊断步骤误当成日常主流程
   - 结果导向：有清晰的用户可见成功标准，也有自然的排除边界
8. 选择重复任务和成功标准最清晰的容器。
9. 决定源经验在该 skill 内应放置的位置：
   - 主工作流 / 正常路径
   - 编写规范 / 维护规范
   - 验证清单
   - 故障排查
   - 条件分支
   - 兼容性说明
   - 负面示例
   - 参考材料

## 失败路径

- 如果源材料缺少任务背景、用户期望结果或可观察成功信号，MUST 最多询问 3 个澄清问题；问题必须只补齐缺失字段。
- 如果用户未补充，或补充后仍无法识别用户结果轴，NEVER 强行推荐 skill 容器；输出阻塞原因、缺失字段和需要用户提供的下一步材料。
- 如果无法生成至少 4 个候选容器，MUST 说明缺少哪个抽象轴的信息，并停止在“候选不足”状态，而不是把相邻任务硬凑进同一容器。
- 如果输入已经是 skill、SKILL.md、已安装 skill，或用户是在审查、优化、更新、修复现有 skill，MUST 停止使用本 skill，并说明这不是新 skill 任务域框定问题。
- 如果用户明确要求创建诊断 / 排障 skill，MUST 仍生成一个正向结果候选作为对照；只有在诊断本身就是用户结果时，才允许选择排障容器。

## 抽象轴选择

候选容器不能只给一个结论。MUST 把至少 4 个候选放在同一张比较里，让命名质量可检查。

优先级通常是：

1. 用户结果轴：描述最终要保证的可观察结果，通常最稳。
2. 重复任务轴：描述未来会反复出现的工作类型，前提是成功标准统一。
3. 实现对象轴：只在用户的目标本身就是维护该对象时才选择。
4. 平台 / 供应商轴：只在任务确实专属于该平台或供应商时才选择。
5. 代码位置轴：通常只作为定位信息，不作为 skill 容器。

如果某个候选名称必须依赖读者记住事故里的专有名词才会触发，它通常不是好容器。

如果候选名称以“排查 / 修复 / probe / debug / workaround”为中心，MUST 再生成一个对应的正向结果候选：如果没有故障发生，用户应如何创建或维护正确产物。NEVER 把排障候选作为最终容器，unless 用户明确要的是诊断工具。

## 候选评分

对每个候选项做定性评分：

| 标准 | 好信号 | 坏信号 |
|---|---|---|
| 结果对齐 | 按用户想要的结果命名 | 按某个事故对象命名 |
| 可重复性 | 会在多个项目或相似任务中复现 | 只对一次过去事件有用 |
| 成功标准 | 可观察、可测试 | 依赖内部实现是否存在 |
| 范围边界 | 能干净地排除相邻工作 | 吸收无关的邻近任务 |
| 触发质量 | 用户请求能自然映射到它 | 需要记住小众产品名 |
| 放置清晰度 | 经验适合作为 skill 内的一层 | 经验意外变成了整个 skill |
| 路径分层 | 主流程、验证、排障分清 | 把失败后的诊断步骤放成主流程 |

优先选择最容易用用户可观察术语描述验证方式的候选项。

## 范围边界

候选容器 MUST 同时说明它管什么和不管什么。

选择候选时，明确列出：

- 包含：哪些输入、失败模式、工作流和验证方式属于同一成功标准。
- 排除：哪些相邻任务虽然碰到同一文件、平台、资源或代码位置，但成功标准不同。
- 放置：源经验应进入主流程、条件分支、兼容性说明、故障排查、验证清单、负面示例还是参考材料。

当一个候选会把多个成功标准不同的任务合并在一起时，继续缩小或换轴。

## 输出格式

返回以下结构：

```text
推荐 skill 容器:
<规范的结果领域 skill 名称>

推荐原因:
- <为什么这是重复任务领域>
- <为什么成功标准清晰>

为什么不是更窄:
- <哪些表层名词会让 skill 过窄>

为什么不是更偏实现:
- <哪个实现对象或代码位置只是手段，而不是真正目标>

为什么不是更宽:
- <哪些相邻任务有不同成功标准，应该留在范围外>

范围边界:
- 包含: <哪些输入、失败模式、工作流和验证方式属于同一成功标准>
- 排除: <哪些相邻任务虽然碰到同一文件、平台、资源或代码位置，但成功标准不同>
- 放置: <源经验应进入主流程、条件分支、兼容性说明、故障排查、验证清单、负面示例还是参考材料>

候选容器比较:
- <候选 A>: <轴>; <主要问题或优势>
- <候选 B>: <轴>; <主要问题或优势>
- <候选 C>: <轴>; <主要问题或优势>
- <候选 D>: <轴>; <主要问题或优势>

路径分层:
- 正常路径: <skill 主体应该教人如何创建或维护正确产物>
- 验证路径: <如何证明产物正确>
- 排障路径: <失败后才使用的探针、日志、缓存绕过、对比或回滚方法>

建议的验收标准:
- <可观察的检查、命令、文件、输出或人工验证>
```

## 验收标准

完成的框架决策 MUST 同时满足以下可检查条件：

- 输出包含 `推荐 skill 容器`、`推荐原因`、`为什么不是更窄`、`为什么不是更偏实现`、`为什么不是更宽`、`范围边界`、`候选容器比较`、`这条经验的放置位置`、`路径分层`、`核心防错机制`、`建议的 skill 触发条件`、`建议的验收标准` 12 个标题。
- `候选容器比较` MUST 至少包含 4 个候选，并且每个候选都标明一个轴：平台 / 供应商轴、实现对象轴、代码位置轴、用户结果轴或重复任务轴。
- 每个候选 MUST 写出一个优势或问题，问题必须属于过窄、过宽、过于实现导向、过于排障导向、成功标准不清或触发质量差之一。
- `范围边界` MUST 同时包含 `包含`、`排除`、`放置` 三行。
- `路径分层` MUST 同时包含 `正常路径`、`验证路径`、`排障路径` 三行。
- 如果缺少以上任一项，NEVER 宣称 framing 完成；补齐缺项或进入失败路径。

## 示例

<example>
输入摘要：材料描述一次发布后，外部消费者仍拿到旧资产；排查中出现过构建目录、缓存失效、下载链接和验证命令。

推荐 skill 容器:
published-artifact-freshness

推荐原因:
- 这是会在发布、分发、验证外部资产时反复出现的结果领域。
- 成功标准可以通过外部消费者拿到的新版本资产、校验值或可见内容来验证。

为什么不是更窄:
- 以某个缓存层、下载链接或事故名词命名，会只在记得这次事故时触发。

为什么不是更偏实现:
- 构建目录和缓存刷新只是实现手段；用户真正要保证的是外部消费者拿到正确版本。

为什么不是更宽:
- 发布审批、版本号策略和回滚演练的成功标准不同，不应合并进同一个 skill。

范围边界:
- 包含: 发布产物生成、外部分发、缓存刷新后的可见性验证。
- 排除: 纯构建脚本重构、发布审批流设计、事故复盘写作。
- 放置: 缓存刷新经验放入验证清单和故障排查，不作为主工作流标题。

候选容器比较:
- vendor-cache-refresh: 平台 / 供应商轴; 过窄，绑定一次事故中的缓存层。
- build-output-directory-maintenance: 实现对象轴; 过于实现导向，只描述产物所在位置。
- release-route-debugging: 代码位置轴; 过于排障导向，触发词依赖诊断过程。
- published-artifact-freshness: 用户结果轴; 结果可观察，边界清晰。

这条经验的放置位置:
验证清单 / 故障排查

路径分层:
- 正常路径: 生成发布产物后，确保外部分发入口指向新版本。
- 验证路径: 从外部消费者入口读取资产，并比对版本、内容或校验值。
- 排障路径: 失败后才检查缓存层、构建目录、路由和回滚手段。

核心防错机制:
- NEVER 把事故中的缓存探针提升为 skill 主流程。

建议的 skill 触发条件:
- 用户要创建、维护、验证外部可消费发布产物的新鲜度，而不只是调试某个缓存问题。

建议的验收标准:
- 外部入口返回的版本、校验值或内容与本次发布产物一致。
</example>

<bad-example>
WRONG:

推荐 skill 容器:
cache-debug-fix

候选容器比较:
- cache-debug-fix: 修复缓存问题。

错误原因:
- 只给 1 个候选，违反至少 4 个候选的验收标准。
- 名称绑定排障动作而非用户结果，违反“NEVER 让事故里的排查路径自动成为 skill 主流程”。
- 没有 `范围边界` 和 `路径分层`，无法检查相邻任务是否被误吸收。
</bad-example>

## 护栏

- 除非任务确实是平台特定的，否则 NEVER 让具名平台、供应商、框架、文件、标签、API 或函数成为 skill 名称。
- 当真实结果跨越缓存、验证、部署、资产、外部消费者或用户可见行为时，NEVER 把实现位置作为 skill 范围。
- NEVER 因为多个任务接触同一个文件或子系统，就创建一个包罗万象的 skill。
- 当存在更干净的结果领域名称时，NEVER 在推荐名称中保留事故词汇。
- NEVER 把一次事故中的探针页、日志查询、缓存绕过、临时路由或验证命令提升成 skill 主流程；unless 用户明确要创建排障 skill。
- 如果源材料来自失败复盘，MUST 反问：正常情况下应该如何写、生成、配置或维护，才能避免再次失败？
- 最终 skill 的第一屏 MUST 服务于推荐容器的核心结果；NEVER 被事故中的排查顺序牵引。
- NEVER 跳过候选比较；过早命名正是这个 skill 要防止的主要失败模式。
- 除非用户在决策后明确要求实现，否则 NEVER 在 framing 步骤中编写最终 `SKILL.md`。
- NEVER 把验证用例、事故示例或期望答案里的专有名词写进本 skill；本 skill 只保留可迁移的抽象和流程。

## 移交给 Skill 创建

当框架决策被接受，或用户明确要求实现后：

1. 使用 `skill-creator` 处理实际的 skill 文件夹。
2. 以推荐容器作为 skill 名称基础。
3. 把原始经验放到已决定的层级。
4. 只有当事故特定名称对未来执行确实必要时，才把它们保留为兼容性说明或示例。
5. 在最终 skill 中保留范围排除项，避免它之后吸收相邻任务。
6. 写最终 `SKILL.md` 前检查：标题、description、第一段和第一节是否都服务于推荐容器的核心结果，而不是被事故中的排查顺序牵引。
