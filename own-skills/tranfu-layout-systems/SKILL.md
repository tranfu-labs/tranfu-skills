# TranFu 布局系统

使用本 Skill 时，在设计实现或评审页面前，先识别用户的主要活动，再判断页面应采用哪一种主布局系统、页面框架模式和页面模式。

## 执行所有权与组合调用

- 本 Skill 是 `decision-only`：MUST 只输出布局决策或布局评审，NEVER 编辑代码、设计文件或品牌资产。
- 页面级实现、组件视觉、响应式细节和 Visual QA 由 `$tranfu-website-design` 负责。
- 作为 `$tranfu-website-design` 的依赖运行时，MUST 返回命名布局产物并结束，NEVER 再次调用 `$tranfu-website-design`。
- 作为顶层 Skill 运行且用户同时要求实现 UI 时，先输出命名布局产物；仅当 `$tranfu-website-design` 尚未处于活动调用链时，才将该产物交给它继续执行。

## 职责边界

本 Skill 负责页面级和屏幕级结构，包括：

- 用户主要活动、页面目标和关键操作。
- 信息架构、内容优先级和 Section 顺序。
- 页面外壳、导航模型和主工作区。
- Page Header、Sidebar、Main Content 与 Auxiliary Panel 的组合。
- 信息密度、操作路径和上下文保留方式。
- 桌面端、平板端和移动端的结构重排。
- 会显著改变页面结构的 Loading、Empty、Error 等必要状态。
- 跨页面同类内容的结构一致性和模块锚点。

本 Skill 不负责：

- Logo 设计和品牌资产创建。
- 颜色、字体、间距、圆角、阴影和图标等具体视觉 Token。
- 卡片、按钮、输入框、弹层等组件的最终视觉样式。
- 与布局无关的纯文案任务。
- 任何代码、设计文件或品牌资产修改；本 Skill 在所有模式下都保持 `decision-only`。

当任务同时要求实现 TranFu UI 时，MUST 先输出布局产物，再按上方调用守卫将其交给 `$tranfu-website-design` 完成视觉、组件和响应式细节落地。

## 分类模型

TranFu 的布局决策分为三层：

1. **主布局系统**：决定产品外壳、全局导航和整体信息组织。
2. **页面框架模式**：决定 Browser Header、Sidebar、Page Header、Main Content 和 Auxiliary Panel 如何组合。
3. **页面模式**：决定当前页面的主要任务、内容结构和操作路径。

三层可以组合使用，不应相互替代。

```yaml
primary_system: app-shell
frame_layout: t-layout
primary_pattern: management-list
```

上例表示：这是一个应用型产品页面，采用顶部导航、侧边栏和主内容区组成的 T 型框架，当前页面主要用于记录管理。

## 工作流

CREATE A TODO LIST FOR THE TASKS BELOW，并在执行过程中更新。

1. **校验目标。** 若既没有页面、文件、截图、运行目标，也没有足以判断页面任务的描述 → 询问一项最必要的信息并退出。
2. **分类模式。** 选择 `create`、`modify`、`refactor`、`review`、`classify` 或 `visual_qa`；若不属于本 Skill → 指向正确工作流并退出。
3. **识别主要活动。** 从已有页面、任务描述和用户行为识别页面目标、关键对象和关键操作；若仍无法识别 → 提出一个简短问题并退出。
4. **选择布局。** 选择一个主布局系统、一个页面框架模式和一个主要页面模式；只有其他模式显著影响结构时才记录为辅助模式。
5. **加载规则。** 读取 `references/layout-contracts.md` 的选择规则和当前任务所需章节；若文件缺失或不可读 → 输出 `TRANFU_LAYOUT_BLOCKER` 并退出。
6. **形成结构。** 定义导航模型、信息密度、桌面结构、移动结构、必要状态、一致性基准和允许偏移。
7. **按模式派发。** `create|modify|refactor|classify` → 子流程「布局决策」；`review|visual_qa` → 子流程「布局评审」；否则 → 输出 `TRANFU_LAYOUT_BLOCKER` 并退出。
8. **处理组合调用。** 若本 Skill 由 `$tranfu-website-design` 调用 → 返回布局产物；若为顶层调用且用户要求 UI 实现、且 `$tranfu-website-design` 尚未活动 → 将布局产物交给它；否则 → 直接返回布局产物。
9. **结束。** 产出 `TRANFU_LAYOUT_DECISION`、`TRANFU_LAYOUT_REVIEW` 或 `TRANFU_LAYOUT_BLOCKER`，更新 TODO 并结束。

### 子流程「布局决策」

1. 使用 `create|modify|refactor|classify` 对应模式填充完整 `TRANFU_LAYOUT_DECISION`。
2. 校验 `primary_system`、`frame_layout`、`primary_pattern`、桌面结构和移动结构均存在。
3. 返回 `TRANFU_LAYOUT_DECISION` 给主流程步骤 8。

### 子流程「布局评审」

1. 只检查实际提供的页面、文件、截图、运行目标或 UI 区域，不根据名称推断视觉事实。
2. 使用 `review|visual_qa` 对应模式填充 `TRANFU_LAYOUT_REVIEW`；未实际观察的视口标记为 `not_run`。
3. 返回 `TRANFU_LAYOUT_REVIEW` 给主流程步骤 8。

# 主布局系统

## `website`

适用于官网、品牌页、产品介绍页、功能页、解决方案页、公司展示页和公开内容页。

主要活动：

- 了解品牌或产品。
- 理解产品能力。
- 建立信任。
- 浏览多个内容区。
- 进入产品、社群、安装或联系路径。

常见结构：

```text
Browser Header
→ Hero
→ 产品证据 / 核心能力
→ 工作流 / 使用方式
→ 场景 / 资源 / 实践
→ CTA
→ Footer
```

规则：

- 页面围绕连续浏览和分段叙事组织。
- Hero 应快速说明页面主题、产品能力和主要行动。
- 真实产品证据应优先于抽象装饰。
- 不同 Section 各自承担一个清晰论点，避免在同一区域堆叠过多同级信息。
- 具体宽度、Viewport Band、Content Rail 和响应式规则由 `$tranfu-website-design` 的响应式参考资料负责。

## `app-shell`

适用于 B 端 / SaaS 产品、后台、CRM、ATS、知识库管理、内部业务系统和持续使用型产品。

主要活动：

- 管理对象和记录。
- 执行业务流程。
- 查看状态和待办。
- 在多个功能模块之间切换。
- 配置和维护系统。

通常包含：

- 全局导航。
- 页面级导航或 Sidebar。
- Page Header。
- Main Content。
- 上下文操作区。
- 可选 Auxiliary Panel。
- 稳定的列表、详情、筛选和状态模型。

规则：

- 首屏应尽快进入可操作状态，而不是展示营销式 Hero。
- 标题区保持紧凑，主要操作靠近当前页面或对象上下文。
- 搜索、筛选和批量操作贴近其控制的数据。
- 详情可使用稳定路由、Drawer、分栏或就地展开，选择依据是任务深度和上下文保留需求。

`b2b-product` 可作为外部别名，但规范内部统一使用 `app-shell`。

## `docs-knowledge-base`

适用于帮助中心、产品文档、API 文档、使用指南、教程和知识库文章。

主要活动：

- 搜索信息。
- 阅读和理解说明。
- 在文档结构中定位内容。
- 复制代码、配置或操作步骤。
- 在相邻主题之间导航。

通常包含：

- 文档导航。
- 搜索。
- 正文阅读区。
- 页内目录或锚点。
- 上一篇 / 下一篇。
- 相关内容和反馈入口。

规则：

- 优先服务搜索、扫描、连续阅读和跨文档跳转。
- 标题层级、当前位置和页内锚点必须稳定。
- 代码块、表格、步骤、警告和示例应有清晰分组。
- 长文不使用营销式卡片墙或无关转化模块打断阅读。

## `landing-page`

适用于活动页、报名页、下载页、预约演示页和具有单一转化目标的页面。

主要活动：

- 快速理解价值。
- 判断是否采取行动。
- 完成一个明确转化目标。

规则：

- 只保留一个主要转化目标。
- 信息顺序围绕“价值—适用对象—证据—行动”组织。
- 行动入口的文案、位置和层级保持稳定。
- 不放置多个相互竞争的主操作。
- 与 `website` 的区别是：`website` 支持多主题浏览和品牌理解；`landing-page` 围绕单一行动建立短而集中的转化路径。

# 页面框架模式

页面框架模式用于定义页面的基础区域和宏观排布，可以与列表、详情、表单、看板和工作台等页面模式组合使用。

页面框架只决定区域如何组织，不直接决定颜色、字体、圆角、组件样式或具体视觉值。

## 框架组成

| 区域 | 是否必需 | 说明 | 常见内容 |
| --- | --- | --- | --- |
| Browser Header | 可选 | 产品或页面的全局顶部区域 | Logo、全局导航、搜索、主操作、用户入口 |
| Sidebar | 可选 | 产品级或页面级导航与上下文入口 | 导航树、目录、搜索、工作区切换、对象列表 |
| Page Header | 可选 | 当前页面的身份、状态和主要操作区域 | 标题、说明、面包屑、状态、Tabs、主要操作 |
| Main Content | 必需 | 承载用户主要任务和核心信息 | 列表、表格、表单、看板、详情、工作区、空状态 |
| Auxiliary Panel | 可选 | 承载辅助信息、上下文操作或实时状态 | 属性、日志、预览、帮助、关联对象、任务状态 |
| Footer | 可选 | 页面底部的补充导航和说明 | 链接、版权、状态说明、法律信息 |

不应为了填满框架而强行加入所有区域，只保留支持用户主要活动所必需的部分。

## `stacked-layout`：上下布局

```text
Browser Header
└── Main Content
    └── Optional Footer
```

适用于：

- 官网、品牌页、落地页和活动页。
- 单一内容流页面。
- 不需要长期 Sidebar 的公开页面。
- 以向下浏览和分段叙事为主的页面。

规则：

- 页面沿纵向自然展开。
- Section 顺序承担主要叙事逻辑。
- 桌面端和移动端通常都保持上下结构。
- 移动端主要调整 Section 顺序、间距、栅格列数和媒体位置。
- 不因桌面构图需要而在移动端保留不可读的并列内容。

## `t-layout`：T 型布局

```text
Browser Header
├── Sidebar
└── Main Content
```

适用于：

- B 端 / SaaS 产品。
- 后台和管理系统。
- 文档知识库。
- 设置中心。
- 多模块长期使用型产品。

规则：

- Browser Header 承载全局身份、工作区和账户操作。
- Sidebar 承载产品模块、页面目录或稳定导航。
- Main Content 承载当前页面的主要任务。
- 桌面端 Sidebar 可以固定或折叠。
- 平板端 Sidebar 可以缩窄、折叠或转为临时面板。
- 移动端通常将 Sidebar 收入 Drawer，不得挤压 Main Content。

## `page-header-layout`：页面头部 + 内容布局

```text
Browser Header
└── Page Header
    └── Main Content
```

适用于：

- 二级页面。
- 列表页、详情页、设置页和表单页。
- 不需要常驻 Sidebar，但需要稳定页面身份和操作区的场景。

Page Header 通常承载：

- 面包屑或返回路径。
- 页面标题与说明。
- 对象状态。
- 主要操作和次级操作。
- Tabs、筛选摘要或时间范围。

规则：

- Page Header 和 Main Content 必须属于同一页面上下文。
- Page Header 不应变成第二套全局导航。
- 页面身份、当前状态和主要操作保持稳定。
- 移动端重排标题、状态和操作。
- 多个操作应收敛为主要操作和更多菜单。
- Tabs 或筛选器空间不足时，可横向滚动、折叠或转为菜单。

## `multi-pane-layout`：多面板综合布局

```text
Browser Header
├── Sidebar
├── Main Content
└── Auxiliary Panel
```

适用于：

- AI 工作台。
- 调试台和执行控制台。
- 复杂详情页。
- 任务监控和运行状态页面。
- 需要同时查看输入、输出、日志或上下文的高密度页面。

规则：

- Sidebar 承载稳定导航或对象列表。
- Main Content 承载用户主要任务，并保持最高优先级。
- Auxiliary Panel 承载属性、日志、预览、状态或关联信息。
- 不因信息较多就默认同时展开所有面板。
- 紧凑桌面端优先折叠 Auxiliary Panel，而不是压缩 Main Content。
- 平板端辅助面板可转为 Drawer、Popover 或上下堆叠区域。
- 移动端通常只保留一个主要工作区；Sidebar 和 Auxiliary Panel 转为 Drawer、独立页面或上下文入口。

## 页面框架选择规则

- 连续浏览、品牌介绍和内容叙事 → `stacked-layout`
- 顶部导航 + 稳定 Sidebar → `t-layout`
- 页面身份与操作头部 + Main Content → `page-header-layout`
- Main Content + 上下文辅助面板 → `multi-pane-layout`

不要因为页面内容多就默认使用多面板布局，也不要因为实现简单就把所有页面做成上下布局。

# 页面模式

## `management-list`

适用于记录、资源、任务、客户、候选人和内容管理。

主要结构：

- Page Header 与主要操作。
- 搜索、筛选和排序。
- 列表或表格。
- 批量操作。
- 分页或持续加载。
- Loading、Empty、Error 状态。

## `console-workbench`

适用于 AI 工作台、任务操作中心、抓取 / 解析 / 发布控制台、调试台、运行监控和工具编排界面。

主要结构：

- 任务或对象上下文。
- 当前状态和主操作区。
- 输入 / 配置区。
- 输出 / 运行状态区。
- 日志、步骤或 Auxiliary Panel。
- 取消、重试和异常恢复路径。

## `dashboard-analytics`

适用于指标页、漏斗分析、趋势监控、报表页和运营 BI 页面。

主要结构：

- 指标摘要。
- 时间范围与筛选条件。
- 图表或趋势区域。
- 分布、对比和解释。
- 明细列表或下钻入口。

## `form-configuration`

适用于设置页、创建 / 编辑表单、参数配置、集成配置和发布配置。

主要结构：

- 页面或对象上下文。
- 分组表单。
- 辅助说明。
- 验证与错误定位。
- 影响摘要或预览。
- 稳定的保存 / 取消操作区。

## `wizard-flow`

适用于新手引导、导入流程、创建项目流程、配置向导和多步骤发布流程。

主要结构：

- 当前目标和步骤进度。
- 当前步骤内容。
- 帮助、要求和验证。
- 上一步 / 下一步。
- 保存、退出和恢复路径。
- 完成确认或结果页。

## `detail-page`

适用于候选人、企业、职位、任务、资源和记录详情。

主要结构：

- 对象身份和状态。
- 主要操作。
- 关键摘要。
- 分组详情。
- 关联对象或历史记录。
- 可选 Auxiliary Panel。

规则：

- 主要信息和次要元数据必须分层。
- 高频操作靠近对象身份或相关内容。
- 危险操作独立隔离。
- 与列表配合时，根据任务深度选择稳定路由、Drawer、分栏或就地展开。

# 所有布局通用规则

## 用户任务与信息优先级

- 必须先识别用户的主要活动、页面目标和关键操作，再选择主布局系统、页面框架和页面模式。
- 每个页面必须有一个明确的主要活动。页面可以同时承载状态、辅助信息和次级操作，但必须保持清晰主次关系。
- 页面标题、对象身份、当前状态和主要操作应放在稳定、可预期的位置。存在连续流程时，还必须明确当前步骤、下一步和返回路径。
- Main Content 必须保持最高的信息与操作优先级；导航、筛选器、日志、属性和 Auxiliary Panel 不得无故压缩或抢占主工作区。

## 内容承载与复杂度

- 内容必须根据关系进行分组，但分组不等于新增卡片。优先使用标题、列表行、Tabs、折叠、间距、浅表面区块或独立页面控制复杂度。
- 避免无意义的卡片嵌套。卡片适用于独立模块、重复内容项或需要明确边界的功能单元；已有面板内部优先使用分组和列表结构。
- 不因信息较多就默认同时展示全部区域。低频内容、辅助信息和上下文操作可以通过折叠、Drawer、Popover、Tabs 或独立页面按需呈现。
- 页面结构应服务真实任务，不使用空容器、巨大留白或装饰区域制造虚假的层级和复杂度。

## 布局稳定性

- 布局必须保持稳定。Loading、Error、筛选展开、日志增长、状态文案变化和异步结果出现时，不应造成 Main Content 不可预测地跳动、遮挡或失去操作位置。
- 会显著改变页面高度或宽度的异步内容，应预留合理空间、限制增长范围，或提供滚动、折叠和分页策略。
- Fixed、Sticky 和 Absolute 区域不得遮挡主要内容、焦点目标、错误信息或移动端操作。
- 布局不得依赖固定页面高度。

## 响应式重排

- 桌面端可以使用并列区域、Sidebar 和 Auxiliary Panel；移动端必须根据任务优先级进行堆叠、折叠、转为 Drawer 或拆分为独立路径。
- 移动端不得只是将桌面布局整体缩小。主操作、当前状态和核心内容应优先保留，低频辅助区域应后置或按需打开。
- 空间不足时，应优先减少列数、折叠辅助面板、收敛间距和重排内容，不得通过压缩 Main Content 或将文字缩小到可读下限以下解决问题。

## 状态与恢复路径

- 页面应根据功能考虑会影响结构的状态，例如 `loading`、`empty`、`error`、`warning` 和完成结果，并为其提供稳定位置和恢复路径。
- 交互组件只需实现与自身功能相关的状态，不机械要求所有组件覆盖全部状态。
- Empty 状态应提供可执行的下一步；Error 状态应说明原因和恢复方式；Loading 状态应保持布局稳定。
- 长任务应提供进度、取消、重试和结果恢复路径。

## 视觉职责边界

- 布局必须符合 TranFu 中性、克制、工程化和真实产品优先的总体方向。
- 具体颜色、字体、卡片、描边、阴影、圆角和组件状态样式，由 `$tranfu-website-design` 及其模块化参考资料负责，本 Skill 不重复定义具体视觉值。
- 禁止将官网或落地页的 Hero 式构图直接套用于高密度 B 端、操作台、数据看板、表单和详情页，除非页面的主要目标确实是品牌展示或营销转化。
- 禁止加入与用户主要活动无关、会抢夺产品内容注意力的装饰性布局区域。

# 参考资料路由

`references/layout-contracts.md` 是本 Skill 的必需参考资料：

- 做任何布局判断前，MUST 读取其中的「选择规则」和当前主布局系统、页面框架、页面模式对应章节。

# 失败路径

- 目标与任务描述均缺失 → 询问一项最必要的信息并退出。
- 无法识别用户主要活动 → 提出一个简短问题；仍无法识别时输出 `TRANFU_LAYOUT_BLOCKER`，代码为 `unclassifiable_activity`。
- 必需参考资料缺失或不可读 → 输出 `TRANFU_LAYOUT_BLOCKER`，代码为 `missing_layout_reference`。
- 页面、代码、截图和运行结果对结构事实相互冲突 → 输出 `TRANFU_LAYOUT_BLOCKER`，代码为 `evidence_conflict`，列出冲突证据。
- `$tranfu-website-design` 不可用 → 仍返回完整布局产物，并在 `handoff` 字段记录 `status: not_run`；NEVER 自行编辑 UI。
- 未实际观察的截图、运行目标或视口 → 标记为 `not_run`；NEVER 推断为通过。
