# 多平台写作流程

让已批准的证据经过一次大纲确认，再进入两份母稿和十份完整平台稿。

## 什么时候用它

**准备共享大纲**

我已有批准通过的题目和调研包，或等价的授权材料，希望先得到一份受证据约束的大纲，在完整写作前审阅。

**生成双版稿件**

我已确认大纲的准确路径和 SHA-256，希望生成两份平台中立母稿，同时确保 B 版风格不会影响 A 版。

**适配五个平台**

我需要公众号、小红书、知乎、微博和头条号的双版本完整稿件，每份都直接从对应母稿适配。

**不接**

它不处理调研、选题、一次性写稿、标题池、审校、配图、排版、发布或发布后分析。

## 它会产出什么

**完整写作只有一个人工门禁：必须按准确路径和 SHA-256 批准共享大纲。**

- **批准大纲**：写入有版本号的共享大纲，并停下来等待明确批准
- **双版母稿**：生成 A、B 两份母稿，严格隔离两边的额外输入
- **平台初稿**：生成五个平台乘以两个版本，共十份完整适配稿
- **确定性核验**：检查输入快照、哈希、必需文件、标题、占位符和交接状态
- **文件改动**：独立模式只写入 `03-内容创作/<run-id>/`
- **交接结果**：以 `READY_FOR_PROOFREAD` 收尾，并返回路径与验收数据
- **绝不会做**：搜索事实、默认批准大纲、审校、配图、排版或发布

## 前置条件与边界

**前置条件**

需要一份有效且已通过的选题方案与完成的调研记录，或明确的题目、受众、大纲和授权材料。必需的参考文件及脚本必须可读。

**相邻能力分工**

| 需求 | 交给 |
|---|---|
| 调研与证据整理 | **collect-sources** |
| 审查完成后的稿件 | **proofread-content** |
| 运行完整生产流程 | **content-production** |

**不接的场景**

- 普通文案或单平台写作
- 只讨论大纲、不需要完整双版流程
- 标题、图片、排版、发布或排期任务

**容易混淆的边界**

- A、B 两版共享事实和大纲，但 A 版绝不读取 B 版风格文件
- 每个平台稿都从对应母稿产生，不会从另一个平台稿二次改写
- 总控模式会把结果交回总控，不创建独立的嵌套运行
