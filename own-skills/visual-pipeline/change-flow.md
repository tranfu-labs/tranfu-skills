# change-flow

任何对显示信息段 / 显示框架段 / 风格段文档或 HTML 的改动, 走这里.

**全程中文**: 文档与对话中不使用 S1/S2/P0/P1/R1 等英文缩写, 用 "主场景 / 必看字段 / 头部区 / 入口态" 等中文.

## 一、归属判定 (必做第一步)

收到修改请求**不要直接改**, 先归属 — 这条修改本质是哪一段的决策错了?

| 修改信号 | 归属 |
|---|---|
| 加/减字段; 改用户场景; 改优先级; 改反场景 | 显示信息段 |
| 加/减/挪区域; 改状态机; 改区域间联动 | 显示框架段 |
| 调色板; 换对标产品; 加候选; 切选定版 | 风格段 (改候选) |
| 加/减组件类型 (从显示框架段推出来的) | 风格段 (改组件清单 + 各候选样例库) |
| 改某个组件的样式 (按钮长什么样、列表行布局) | 风格段 (改各候选样例库) |
| 抠按钮圆角/间距/hover/icon 选型 | 风格段 (改选定版的整体组合 section) |
| 用户说"不对 / 不是我要的"且指向上游决策 (用户/场景/字段) | **退回最上游的相关段** |

归属错 = 修不动. 动手前必须**显式说出归属判定**, 让用户能纠正:
> "我判断这是显示信息段 (用户场景/字段优先级) 的修改, 退到 `01-display-info.md` 改. 对吗?"

## 二、退回最上游 + 下游同步

定位到某段后:

1. 先改该段的**文档**, 再改该段的 HTML (如果该段有 HTML)
2. **逐段向下推进**: 后续段必须同步检查并更新
3. 风格段同步 = 看选定版是否还合适; 不合适按下文第四节切换

不允许跳改 (改显示信息段后直接改风格段, 跳过显示框架段同步).

## 三、跨段合并禁止

"我顺便把信息段和风格段一起改" — 不行. 一次只动一段, 改完同步下游, 再开下一次.

**例外**: 上游段的改动直接、机械地引发下游段的纯翻译式变动 (例: 加字段后显示框架段区域表加一行容纳它), 这种翻译式同步可在同一次完成. 但仅限翻译, **不带新决策**.

## 四、风格段选定版的特殊规则

`03-selected.html` 是 ground truth. 切换选定:

1. `mv 03-selected.html 03-alt-<旧调性名>.html` (旧选定归档)
2. `mv 03-alt-<新调性名>.html 03-selected.html` (新选定上位)
3. `03-style.md` changelog 节加一行: `YYYY-MM-DD: 选定从「调性 A」切到「调性 B」, 原因 ...`

**禁止**: 悄悄改 `03-selected.html` 的视觉决策 (色板/调性) 而不切候选 + 写 changelog. 像素级抠不算切, 仍在同一选定版内迭代, 不需要 changelog.

## 五、工程不可行 (代码层做不出选定版)

→ 退回 **风格段**, 设计层调整, 不要让代码层自作主张简化或漂移. 否则 `03-selected.html` 和最终代码会分叉.

## 六、用户连续否定 (兜底信号)

同一产物**连续两次**被用户否定决策 (不是细调), 强制暂停所有工具调用, 反问:

> "刚才两次都在否定 [决策 X / 决策 Y]. 是不是 [某段] 没锁? 我退回 `0N-*.md` 改?"

这是**最后防线**. 常用 = 前面归属做错了, 应回头检视为什么没在第一次否定时就归属对.

## 七、不允许的操作

- 直接改 HTML 不改文档
- 改文档但不同步下游
- 在代码/HTML 里现编上游决策 (字段、场景、风格)
- 用 "先继续改, 同时反思" 绕过归属判定
- 在任何产物里使用 S1/P0/R1 等英文缩写
