# Skill 内容准入评估

判断一段素材够不够格沉淀成可复用的 skill——只出**通过 / 打回**, 打回时绝不补假流程, 只列缺什么。

## 什么时候用它

**事故复盘要不要固化**:

我刚复盘完一个线上事故, 手里有一份纠偏做法, 想让 skill 判一下这是可复用流程还是一次性事实, 再决定要不要沉淀成规则。

**经验教训要不要复用**:

项目里踩了一次坑, 已经写好一段总结, 想让 skill 说清楚它够不够写成 skill——够就走下游流程, 不够就直接告诉我缺什么。

**文档片段要不要升级**:

某段项目文档 / `AGENTS.md` / issue 讨论感觉可以固化成能力, 又不太确定, 想先过一遍准入判定, 别急着动笔。

**批量筛素材**:

手上一堆事故记录 / 反馈 / 经验片段, 想让 skill 逐条筛出哪些值得沉淀, 剩下的直接归档。

**不接**:

内容已经是 skill / `SKILL.md` / 已安装 skill → **skill-improve-workflow**; 起 skill 的英文目录名 (小写加连字符) → **skill-domain-framing**; 起中英文显示名 → **skill-name-generation**; 判断通过后写 `SKILL.md` → **skill-create-workflow**。

## 它会产出什么

**打回时绝不补写假流程, 只列缺什么——最反常识的一点。**

- **结论**: 只有两类——**通过** (六项准入全过) / **打回** (任一硬条件缺失)
- **六项判断**: 可重复 / 触发条件 / 可执行流程 / 验证方式 / 边界和反例, 每项都必须能从输入摘出至少 1 条证据
- **打回附缺失清单**: 列出输入里缺哪几项 + 需要用户补什么信息, 让下一轮回来时能一次过
- **通过附提炼建议**: 给下游 skill 编写者一份触发条件 / 工作流 / 反例 / 验收标准的草稿要点
- **绝不会做**: 帮打回的内容强行编工作流凑数, 也不会替不合格内容决定去留 (删了还是留着, 由你自己判)

## 前置条件 / 边界

**前置**:

需要一段可以贴过来的内容——事故复盘 / 经验总结 / 文档片段 / 反馈记录都行。内容太短 / 只是背景说明会被直接打回。

**相邻 skill 分工**:

| 动作 | 交给 |
|---|---|
| 判断通过后写 `SKILL.md` 骨架 | **skill-create-workflow** |
| 起 skill 的英文目录名 (小写加连字符) | **skill-domain-framing** |
| 起中英文显示名 | **skill-name-generation** |
| 判断通过后生成 README | **skill-readme-generation** |
| 内容已经是 skill, 想审 / 改进 | **skill-improve-workflow** |

**不接的场景**:

- 输入本身已经是 skill / `SKILL.md` / 已安装 skill 的内容
- 只是一段代码 / 命令想「变成 skill」——没有触发场景, 没有边界
- 想让 skill 把不合格内容改写成合格 (那是 skill 作者的活)

**微妙边界**:

- 高风险场景 (法务 / 财务 / 合规 / 生产发布) 可以放宽「可重复」次数, 但流程和验证必须齐全
- 只是一次性事实 / 单次偏好 / 情绪反馈 → 一律打回, 没有「以后再看」的软通过
