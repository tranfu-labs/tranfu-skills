# strategy-first-development

## 介绍

把复杂产品/工程开发从“马上写代码”改成“先对齐战略、产品形态、预期体验、成熟方案、技术栈和验证矩阵”的 agent 工作流。适合在新项目、MVP、重要功能、架构升级、部署方案或 AI 产品探索开始前使用。

这个 skill 是复杂开发前的门禁，不是创建/优化 skill 的元工作流。创建、审查、修复已有 skill 时应使用对应的 skill 工作流。

## 什么时候用

当你希望 agent 先和你讨论清楚方向，而不是直接实现时使用。典型场景包括：

- 新产品/新 MVP 还没确定战略目标和产品形态；
- 需要先看参考截图、竞品页面或生成几个预期页面方向；
- 需要搜索 GitHub 成熟项目和官方框架，避免重复造轮子；
- 需要确定技术栈、模块边界、架构和工作流程；
- 需要把部署、测试、验收方式在实现前写进计划。

不要用于单行命令、小文案、翻译、已明确范围的 bug 修复、纯代码 review，或“创建/优化 skill”这类元任务。

## 触发示例

```text
用 strategy-first-development 帮我先规划这个产品，不要马上写代码。
```

```text
我想做一个高考出分后选学校和专业的网页，先不要写代码，先定产品形态、预期页面、成熟项目和技术栈。
```

```text
这个项目开始前先和我讨论方案，生成几个预期页面让我选，再决定技术栈和架构。
```

```text
不要重复造轮子，先查成熟框架和 GitHub 项目，再给我实现计划。
```

## 期望输出

skill 会推动 agent 输出：

- strategic goal：为什么做、做给谁、成功标准是什么；
- product/project shape：产品形态和项目形态；
- expected experience：预期截图、页面方向、CLI/API 样例或 workflow；
- mature project findings：GitHub/官方生态成熟项目调研；
- chosen stack：技术栈和被拒绝方案；
- architecture plan：模块、边界、数据流、工作流程；
- verification matrix：测试、构建、浏览器/API、部署检查。

关键硬约束：

- 实现前必须有显式计划或 checklist；
- 网络可用时必须搜索成熟项目/框架；
- 不得在没有理由时手写成熟领域的核心引擎；
- 交付前必须报告验证命令和结果。
