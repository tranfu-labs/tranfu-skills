# 09. 视觉可信一致感 / Visual Trust and Consistency

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**（怎么截图 / 怎么点开状态都在 `00`），只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：favicon / app icon / logo 归 `01`；可交互元素有没有即时反馈归 `03`；icon-only 控件要不要文本 / tooltip 归 `10`；窄视口重排挤坏归 `11`。

## 一、解决什么问题

用户能否把这一页**读成一个连贯、可信、可操作的产品**，而不是多个系统拼起来的半成品。具体到五件事：同类元素能否被读成同一系统、相同状态能否不必重学、视觉层级能否对上信息重要性、分组关系能否被看出来、装饰与功能的边界能否分清——再加上单个控件本身**装不装得下、像不像成品**。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做：

- 翻出同类元素的各可见状态（选中 / 当前 / disabled / error / success / 危险 / loading），用于横向比对；翻开的状态逐个登记进 `states_opened`。
- A / B 类横向比对用脚本盘点输出的**样式向量**（`controls[].style`：背景色 / 文字色 / 边框 / 圆角 / 字号 / 字重 / 内边距 / 行高）。该字段缺失时按 `00` 的对账规则记缺口，A / B 类输出 `blocker`。
- 对 `closeupTargets` 的每一条（关键 CTA 的操作定义见 `00` 1.3），用 `prepareCloseup` / `resetCloseup` 做**裁剪放大重采**特写，供 G 类判定标签装不装得下（不靠 `getClientRects` 行数，那对块级控件恒为 1）。
- 保留双视口整页截图，供「层级」「分组」两类做感知判定。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓。命中下面任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**误读含义 / 状态 / 层级 / 可点性 / 完成度**"才定罪；否则按各分类的**释放理由**放过。

两道闸都假设证据在场。某类要求的证据若不存在（特写没截 / 样式向量缺字段 / 状态没翻到），该类输出 `blocker` 并点名缺什么——**缺证据 = blocker，不是已满足**；不得用整页截图脑补替代该类专用证据。

样式比对用到的轴，按"显眼程度"分两档（这把"不强制间距刻度 / 精确圆角 / 阴影 / token"从口号变成了释放规则）：

- **显眼轴**（单轴离群即嫌疑）：背景填充色系、形状（胶囊 / 方）、有无边框、图标 填充 / 线性、尺寸级别。
- **细微轴**（需 ≥2 轴同时离群，或已破坏分组，才算嫌疑）：精确圆角值、阴影值、内边距差一档、字重差 `<100`。

## 四、形式目录 + 判定

**按类记账**：本维度的产出单位是**类**，不是文档。A–G 每类各给一条结论——`actionable` / `already_satisfied` / `blocker` / `not_applicable` 之一，附一句证据锚点（哪张截图 / 哪组样式向量 / 哪条特写）。标准模式填进阶段 4 判断输出的 `class_coverage`（`17` 定义）；`local sequential fallback` 在结论里逐类列出。一条发现绝不关闭整个维度。

### A. 同类元素是否像同一个系统

形式：

- A1 主 / 次 / 危险按钮各自像不同组件库（填充、形状、边框、尺寸不成体系）。
- A2 同种卡片 / 列表项的圆角、阴影、留白、分隔在同页一会儿一种。
- A3 同类状态标签（badge / chip）的配色体系或形状不统一。
- A4 其它同类元素读起来像多个系统拼接。

判定：把同类元素按职能归桶 → 取桶内最常见样式 → 离群项按显眼 / 细微轴规则挂嫌疑 → 在截图定位离群项，确认它读起来"像另一套系统"且无职能 / 语境理由 → 定罪。释放：不同职能、不同密度、不同功能区语境造成的差异。

### B. 相同状态是否一致表达

形式：

- B1 选中 / 当前项在相近场景一会儿描边、一会儿变底色。
- B2 disabled / loading / error / success 在不同模块表现不稳定，用户每到一页都得重学。
- B3 同一危险语义（删除等）的视觉强度忽轻忽重。
- B4 其它相同状态被做成可分辨度过低或互相混淆。

判定：收集全页同一状态语义的所有实例（环节二已翻出各状态）→ 样式向量比对，显眼轴离群 = 嫌疑 → 截图确认相近场景下表达确实不一致、需要用户重学 → 定罪。释放：不同语境下合理的强度差异。

### C. 视觉层级是否匹配重要性

形式：

- C1 页面标题、区块标题、按钮、状态标签、辅助说明都在抢视觉重量。
- C2 真正的主操作不突出，次要元素反而更扎眼。
- C3 辅助信息（说明 / 时间戳 / 元数据）视觉重量盖过关键数据。
- C4 其它让用户一眼看不出"什么最重要"的层级问题。

判定（感知协议）：在整页截图上做**前三注视点**测试——说出眼睛最先落到的前 3 个元素及顺序 → 和声明的信息优先级（主操作 > 页面标题 > 区块标题 > 辅助）比对 → 顺序不符 = 嫌疑 → 确认这会让用户误判什么最重要 → 定罪。释放：不同功能区可有不同强调节奏，只要关键项仍最先被看到。

### D. 分组关系是否可读

形式：

- D1 哪些控件属于同一个筛选 / 设置区，看不出来。
- D2 某个按钮作用于哪个列表 / 对象，不清楚。
- D3 说明文字归属哪个输入框，靠用户猜。
- D4 其它让用户搞错"控件作用对象"的分组问题。

判定（感知协议）：对一个控件，在截图上凭**邻近 + 共享样式**说出它"读进"哪个组 → 和功能分组（同容器 / aria-group / 实际作用对象）比对 → 视觉分组 ≠ 功能分组 = 嫌疑 → 确认用户会搞错作用对象 → 定罪。释放：分组不靠间距，但有其它清晰线索（标题 / 边框 / 标签）。

### E. 图标风格 / 语义是否打架

形式：

- E1 同一功能区混用线性 / 填充、不同粗细。
- E2 图标语义与文字标签含义不符或互相打架。
- E3 图标颜色强度过载，状态和动作分不清。
- E4 其它让图标既不像状态也不像动作的混乱。

判定：同功能区图标取 {填充·线性、粗细、颜色强度、语义} → 风格离群或语义与标签冲突 = 嫌疑 → 截图确认用户会分不清状态 / 动作或误读含义 → 定罪。释放：不同功能区可有不同图标族。（icon-only 是否需要文本 / tooltip 归 `10`，不在此判。）

### F. 装饰与功能边界

形式：

- F1 装饰性 chip / badge / card / icon 做得像能点（有按钮样式或 `cursor:pointer`，却无动作）。
- F2 真实操作入口做得像静态装饰（无可点线索）。
- F3 可点与不可点元素在视觉上无法区分。
- F4 其它让用户误点装饰、或错过真入口的边界问题。

判定（双向）：① 对 `cursor:pointer` / `role=button` / 带事件监听的元素，看截图上有没有可点线索；② 对显眼且看起来可点的 chip / badge / card / icon，看它实际是否可交互。任一方向错配 = 嫌疑 → 确认用户会误点装饰或错过真入口 → 定罪。释放：约定俗成的可点样式（链接色等）不算装饰冒充。

### G. 控件标签是否装得下（换行 / 裁切 / 溢出）

嫌疑来源：`closeupTargets` 全量 + 盘点里 `wrapSuspect` / `overflow.clipX` / `overflow.clipY` 命中项。

形式：

- G1 按钮文字意外换成两行 / 多行，而容器没为多行设计（高度或对齐被撑乱）。
- G2 标签 / chip / 菜单项文字被裁切，关键信息丢失（有省略号但读不全）。
- G3 文字溢出容器、撑破对齐，或盖到相邻元素。
- G4 其它让控件读起来未完成 / 不像成品的标签排版。

判定（放大视觉协议）：对 `closeupTargets` 的每一条，**逐个**看其放大重采特写——不靠 `getClientRects().length`（它对 `button` 等块级控件恒为 1，内部换行测不出）→ 标签是否被迫换成两行 / 多行、被裁切、或溢出撑破对齐 = 嫌疑 → 在特写上确认它读起来像未完成 / 挤坏、且**不是有意的多行设计** → 定罪。佐证：`wrapSuspect` 命中**换行嫌疑**；`overflow.clipX` / `overflow.clipY` 命中**裁切**（换行是撑高、溢出标志不一定触发，故定罪以视觉特写为准）。特写缺失的条目 = 本类 `blocker`，不是放过。释放：本就是多行设计的控件、有意截断且有完整 tooltip（tooltip 可达性归 `10`）。注意：**只在窄视口下才换行**的归 `11`；本类只主责**默认 / 桌面视口**下的标签贴合。

## 五、边界 / 非目标

- 不强制使用某个间距刻度，不规定统一圆角 / 阴影 / 边框 数值或具体视觉 token。
- 不把 token、CSS 变量、组件 API、设计系统实现、主题配置、组件抽象写成 MUST。
- 不要求所有页面完全一样：不同功能区可有不同密度、节奏、强调，只要用户仍能识别相似对象、状态、层级、关系。
- 不要求做大规模重设计；本维度通过局部收敛状态、层级、分组、图标语义、装饰 / 功能边界、标签贴合来修正。
- 不把"更漂亮 / 更现代 / 更高级"列成目标；必须落到用户能否更容易判断**含义、关系、状态、重要性、可操作性、完成度**。
- 不主责 favicon / app icon / logo（→ `01`）、即时交互反馈（→ `03`）、icon-only 文本需求（→ `10`）、窄视口重排（→ `11`）。
