# 03. 交互状态的用户反馈感 / Interaction Feedback and Action Safety

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**，只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：动作完成后的成功 / 失败 / 处理中结果反馈、撤销可见性、动效分寸归 `12`；按钮 / 确认框 / 错误 / 状态文案是否准确表达对象 / 数量 / 影响 / 可撤销归 `13`；任何状态造成的布局位移归 `04`；数据 loading / 空 / 无结果 / 错误 / 权限态本身归 `05`；表单输入 / 校验 / 提交信心归 `07`；纯 hover 可达 / 触摸目标 / 键盘 / 对比度归 `10`；同类元素 / 状态的视觉一致性归 `09`。本维度只主责**操作前与操作瞬间**：可操作性、即时操作反馈、当前 / 选中态、不可用态、动作风险与优先级、破坏性动作的提交保护。

## 一、解决什么问题

用户在点之前能不能判断**什么可操作**、当前**处在哪个状态**，点下去**有没有被接收**、**什么时候不能继续**，以及高风险动作**会不会被误提交**。缺这些，界面看起来像静态文本、像坏了、像没反应，用户只能靠猜，破坏性动作还可能一键造成难恢复的后果。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做；凡需副作用才能到达的状态记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`）：

- **可操作性盘点**：枚举按钮 / 链接 / tab / 菜单项 / 分页 / 可点击卡片 / 表格行 / 图标按钮，记录是否带可点线索（`cursor:pointer`、`role=button`、按钮样式、链接色、事件监听）。
- **触发 hover / focus / pressed 重盘**：对可交互元素触发 hover、Tab focus、`:active`（mousedown 但不松开，不产生副作用），盘点是否出现 pressed / active / focus 反馈。
- **当前 / 选中态盘点**：翻出当前 tab、当前 nav、当前筛选、当前页、当前选中项、当前展开区域，采集它们与同类未选中项的样式向量差（背景 / 文字 / 边框 / 下划线 / 图标），供"差异够不够分辨"判定。
- **disabled / loading 盘点**：盘点静态可达的 disabled、loading、不可用控件（`disabled` 属性、`aria-disabled`、置灰样式、`pointer-events:none`），记录它们和可点态的视觉区分。
- **破坏性动作入口盘点**：枚举删除 / 移除 / 清空 / 覆盖 / 发布 / 批量操作等入口，记录其视觉强度、是否紧邻常用操作、是否在密集列表 / 表格中、是否带可见的确认 / 保护线索（不真正触发）。
- **动作语义分桶**：把页面动作按 主 / 次 / 危险 / 取消 / 返回 / 清空 归桶，采集各桶的样式向量，供"风险与优先级有没有混淆"判定。

凡 pressed 反馈、loading 接收态、确认 modal 等只有真正提交 / 删除才能到达的，**不执行副作用**，记 `blocker` 缺口。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户在操作前 / 操作瞬间**判断不了可操作性、辨不清当前状态、感知不到动作被接收、分不清能否继续，或被诱导误提交破坏性动作**"才定罪；否则按各分类的**释放理由**放过。够不到的状态一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 可操作性不明（看不出能点）

形式：

- A1 按钮 / 链接 / tab / 菜单项做得像静态文本或装饰，无可点线索。
- A2 可点击卡片 / 表格行 / 图标按钮没有任何"可交互"信号，用户得靠试。
- A3 分页 / 展开收起入口混在普通文本里，看不出是控件。
- A4 其它让用户靠猜哪里能点的可操作性问题。

判定：用探查盘点的可操作性信号 → 对每个候选控件看截图上有没有可点线索（按钮样式 / 链接色 / `cursor:pointer` / 图标按钮形态）→ 全无 = 嫌疑 → 确认用户会把它读成静态文本 / 装饰、不知道能点 → 定罪。释放：约定俗成的可点样式（链接色、明显的图标按钮）已足够；纯装饰元素本就不该可点。（装饰冒充可点 / 真入口像装饰的双向错配也可由 `09` 装饰与功能边界覆盖，本类只主责"该可点的看不看得出能点"。）

### B. 即时反馈缺失（点了没回应）

形式：

- B1 点击 / 切换 / 选择无任何 pressed / active / selected 反馈，像没点中。
- B2 提交 / 加载触发后无 loading / busy，用户不知道是否已接收。
- B3 反馈延迟太久或太弱，用户重复点击。
- B4 其它让用户感觉"动作没被接收"的情况。

判定：用探查触发 hover / focus / pressed → 看动作触发瞬间有没有可感知反馈（pressed / active / selected / loading / busy）→ 缺失 = 嫌疑 → 确认用户感觉操作没回应、会重复触发 → 定罪。释放：已有清晰即时反馈（按钮态变化、列表立即更新等）。注意：pressed / loading 等多数需真正点击才到达，够不到则记 `blocker` 缺口，不猜。

### C. 当前 / 选中态不明确

形式：

- C1 当前 tab / 当前 nav / 当前页只靠极弱颜色差，扫一眼分不出。
- C2 当前筛选 / 当前选中项 / 当前展开区域没有清晰的选中表达。
- C3 选中态与 hover 态难以区分，用户搞不清"选了没"。
- C4 其它让用户靠记忆才能知道当前在哪的状态问题。

判定：用探查采集的选中 / 未选中样式向量差 → 差异是否足够一眼分辨（不止极弱颜色差）→ 不足 = 嫌疑 → 截图确认用户扫一眼判断不出当前在哪个 tab / 页 / 筛选 / 选中项 → 定罪。释放：选中态有明显且稳定的视觉表达（底色 / 描边 / 下划线 / 图标等）。注意：同一选中语义在不同模块表达不一致归 `09`，本类只主责"当前态够不够分辨"。

### D. 不可用态像坏了（disabled / loading 不可辨）

形式：

- D1 disabled 看起来仍像可点，点了没反应，像界面坏了。
- D2 loading 中控件未置忙，用户能重复提交、不知系统是否接收。
- D3 权限不足 / 条件未满足的控件和正常可点态毫无区分。
- D4 其它让用户分不清"暂时不能点 / 永远不能点 / 正在处理 / 需先完成别的动作"的情况。

判定：用探查盘点的 disabled / loading 控件 → 它和可点态有没有可辨的视觉区分、用户能不能理解"为什么不能继续" → 无区分或误读成"坏了" = 嫌疑 → 截图确认用户分不清不可用的性质 → 定罪。释放：disabled / loading 有清晰且约定俗成的表达。注意：loading / 空 / 错误 / 权限态**本身的设计**归 `05`，其文案清晰度归 `13`；本类只主责"控件是否表达出不可用、不可用是何种性质"。不强制每个 disabled 都配 tooltip。

### E. 风险与优先级混淆（主 / 次 / 危险动作分不清）

形式：

- E1 删除 / 提交 / 保存 / 取消 / 返回 / 清空筛选全用同一种按钮样式和强度。
- E2 危险动作（删除等）视觉强度和普通动作一样，看不出风险。
- E3 主操作不突出，次要 / 危险动作反而更扎眼，引导误点。
- E4 其它让用户感知不到动作风险或优先级的混淆。

判定：用探查的动作分桶样式向量 → 主 / 次 / 危险 / 取消 桶之间风险与优先级是否被表达出来 → 全同一强度、或危险动作不被标识 = 嫌疑 → 截图确认用户分不清哪个是主操作、哪个有风险，可能误点 → 定罪。释放：风险与优先级有可感知的强弱区分（危险色 / 次级样式 / 主 CTA 突出等）。注意：动作文案是否准确表达后果归 `13`；同一危险语义跨页强度不一致归 `09`。

### F. 破坏性动作缺乏提交保护

形式：

- F1 不可逆删除、批量删除 / 覆盖 / 发布 / 权限变更直接一键执行，无任何确认或保护。
- F2 影响钱 / 权限 / 安全 / 发布 / 账号 / 生产数据、或影响其他用户 / 团队 / 共享资源的操作无保护。
- F3 危险图标按钮紧邻常用操作或出现在密集列表 / 表格中，极易误触，且无确认 / 保护。
- F4 其它高影响、难预览、难恢复操作缺乏防误提交保护的情况。

判定：用探查盘点的破坏性动作入口 → 按风险 / 可恢复性判断它是否属于"MUST 保护"场景（不可逆 / 批量 / 影响钱权安全发布账号生产数据 / 影响他人或共享资源 / 难理解难恢复 / 易误点危险图标）→ 是该类却无任何提交保护（确认 modal / 二段式提交 / 输名确认 / 提交前 review / 选定目标后才启用 / 危险入口附近清晰的对象与影响范围）= 嫌疑 → 确认误操作会轻易造成难恢复后果 → 定罪。

释放：低成本、单项、可恢复操作（删除筛选条件、移除临时标签、关闭可重开面板、可撤销 archive / remove 等）**不该**被二次确认打断——这类反而要避免过度确认（确认疲劳）。但若产品依赖 undo / 恢复入口 / 可重开路径来降低风险，**恢复反馈是否可见、及时、可理解归 `12`**，本维度只判断"是否避免了过度确认"。确认与保护是手段，不写成"所有危险操作都必须二次确认"；风险被理解、误操作不轻易酿成难恢复结果，才是 MUST。

## 五、边界 / 非目标

- 不把"每个元素必须写某个 CSS selector"写成 MUST；不要求所有交互都有动画；不要求重写组件库。
- 不把 ARIA 属性本身当本维度 MUST；输入方式（hover / 键盘 / 触摸 / 缩放 / 对比度）可用性归 `10`，这里只关心用户可感知反馈。
- 不要求每个 disabled 都配 tooltip；依场景而定。
- 不写成"所有危险操作都必须二次确认"；二次确认是手段，风险保护才是 MUST。
- 不把内部状态管理、组件抽象、样式架构本身当目标；只检查用户能看到和感受到的结果。
- 路由：动作完成后的结果反馈 / undo 可见性 / 动效分寸 → `12`；动作 / 确认 / 错误文案准确性 → `13`；状态造成的布局位移 → `04`；数据 loading / 空 / 无结果 / 错误 / 权限态本身 → `05`；表单输入 / 校验 / 提交信心 → `07`；多记录扫描与动作范围 → `06`；导航范围与返回路径 → `08`；纯 hover 可达 / 触摸目标 / 键盘 / 对比度 → `10`；视口 / 容器重排 → `11`；视觉一致性 → `09`；favicon / app icon / logo → `01`；主题 → `02`。
