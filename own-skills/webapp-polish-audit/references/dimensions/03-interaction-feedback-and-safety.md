# 03. 关键交互动作的风险与保护 / Action Risk and Destructive Protection

这是一条**审查维度**：证据由 `../pipeline/16-page-exploration-and-capture.md` 统一采集，本文件只写判定标准——① 解决什么问题；② 向采集环节申请的额外探查；③ 形式目录 + 判定。

本维度只主责两件事：界面里**哪个动作是危险的**能否被用户读出（风险与优先级），以及**危险动作有没有防误提交保护**（破坏性动作提交保护）。只关注可能造成难恢复后果的动作。

## 一、解决什么问题

用户在操作界面时：

1. 能否分清"这是主操作 / 次操作 / 危险动作"，会不会因为强弱不分而误点危险动作。
2. 危险动作（不可逆删除 / 批量删除 / 覆盖 / 发布 / 权限变更 / 影响钱权账户或生产数据的动作）有没有防误提交保护（确认框 / 二段式提交 / 输名确认 / 提交前 review 等）。

## 二、向探寻引擎申请的额外探查（→ `16` 第 2 步）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做）：

- **破坏性动作入口盘点**：枚举删除 / 移除 / 清空 / 覆盖 / 发布 / 批量操作等入口（文本匹配"删除 / 移除 / 清空 / 覆盖 / 发布 / publish / delete / remove / discard" + 图标匹配 trash / x / close 类），记录其视觉强度、是否紧邻常用操作、是否在密集列表 / 表格中、是否带**可静态检测到**的确认 / 保护线索（`aria-controls` 指向 dialog / 父或 sibling 存在 `<dialog>` 或 `[role=dialog]` / 属性含 `data-confirm` / `data-modal` / `data-require-confirmation` 等）。
- **动作语义分桶**：把页面动作按主 / 次 / 危险 / 取消 / 返回归桶，采集各桶的按钮 `{ text, backgroundColor, color, size, position, container_role }` 样式向量，供"风险与优先级有没有混淆"判定。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"用户会**误点危险动作**，或危险动作被一键触发造成**难恢复后果**"才定罪；否则按各分类的**释放理由**放过。够不到的破坏性动作 confirm 弹层一律按 §四·B 的授权门槛处理，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

> **本维度的判定读 JSON，不看图。** 证据是采集环节落盘的 DOM 结构、可见文本、样式向量、几何与状态属性。判断者不开浏览器——判不了就输出 `blocker` 并写清缺什么，由主代理决定重采还是动用 SKILL.md §0 的「受限截图」例外。**判断者自己去页面上看一眼是越权。**

### A. 风险与优先级混淆（主 / 次 / 危险动作分不清）

形式：

- A1 删除 / 提交 / 保存 / 取消 / 返回 / 清空筛选全用同一种按钮样式和强度。
- A2 危险动作（删除等）视觉强度和普通动作一样，看不出风险。
- A3 主操作不突出，次要 / 危险动作反而更扎眼，引导误点。
- A4 其它让用户感知不到动作风险或优先级的混淆。

判定：用探查的动作分桶样式向量 → 主 / 次 / 危险 / 取消桶之间风险与优先级是否被表达出来 → 全同一强度、或危险动作不被标识 = 嫌疑 → 确认用户分不清哪个是主操作、哪个有风险，可能误点 → 定罪。释放：风险与优先级有可感知的强弱区分（危险色 / 次级样式 / 主 CTA 突出等）。

**证据与判定模式：`脚本+文本判`**。脚本按主 / 次 / 危险 / 取消归桶采集样式向量分布（背景色 / 文字色 / 尺寸 / 边框 / 位置 / 容器角色），输出每桶的 `{ count, dominant_colors, size_range, has_outline, container_context }` JSON。LLM 读 JSON 判"桶间强弱是否有可感知的区分"——不需要截图，不需要看图判断者。启发式：若危险桶的 backgroundColor 与主桶相同（无红 / 警戒色），且 size 不小于主桶 → 高疑；若危险桶有明显警戒色（reds, oranges）或 outline-only 次级样式 → 低疑。走 SKILL §4 的双判断者并行决胜结构。

### B. 破坏性动作缺乏提交保护

形式：

- B1 不可逆删除、批量删除 / 覆盖 / 发布 / 权限变更直接一键执行，无任何确认或保护。
- B2 影响钱 / 权限 / 安全 / 发布 / 账号 / 生产数据、或影响其他用户 / 团队 / 共享资源的操作无保护。
- B3 危险图标按钮紧邻常用操作或出现在密集列表 / 表格中，极易误触，且无确认 / 保护。
- B4 其它高影响、难预览、难恢复操作缺乏防误提交保护的情况。

判定：用探查盘点的破坏性动作入口 → 按风险 / 可恢复性判断它是否属于"必须保护"场景（不可逆 / 批量 / 影响钱权安全发布账号生产数据 / 影响他人或共享资源 / 难理解难恢复 / 易误点危险图标）→ 是该类却无任何可静态或半动态检测到的提交保护 = 嫌疑 → 确认误操作会轻易造成难恢复后果 → 定罪。

释放：低成本、单项、可恢复操作（删除筛选条件、移除临时标签、关闭可重开面板、可撤销 archive / remove 等）**不该**被二次确认打断——这类反而要避免过度确认。确认与保护是手段，不写成"所有危险操作都必须二次确认"；风险被理解、误操作不轻易酿成难恢复结果，才是必须。

**证据与判定模式：`脚本先过滤 + 半动态验证（需授权）`**。分两档：

**静态阶段**（无授权也跑，覆盖模式化实现）：脚本对每个破坏性按钮查静态 confirm signal：

- button 的 `aria-controls` 是否指向 `[role=dialog]` 或 `<dialog>` 元素
- button 父容器 / 直接 sibling 是否有 `[role=dialog]` / `<dialog>` 元素（含 hidden 态）
- button 属性是否含 `data-confirm` / `data-modal` / `data-require-confirmation`
- button 所在 form 是否含 `<input type="hidden" name="confirmation">` 等模式

命中任一 → `already_satisfied`。全无 signal → 进入半动态门槛。

**半动态阶段**（需授权）：仅当以下三条件**全部满足**才启用：

1. `audit-state.json.allow_destructive_click_simulation === true`（由用户显式授权，见 SKILL §0 例外条款）；
2. 目标 URL 主机名含 `staging.` / `.test.` / `localhost` / `127.0.0.1` 之一（生产环境强制静态）；
3. 拦截层已就位：`fetch` / `XMLHttpRequest.prototype.send` / `navigator.sendBeacon` / `WebSocket.prototype.send` / `window.location` setter / `beforeunload` 全部 hook 为 no-op。

半动态流程：

1. Pre-snapshot：`[role=dialog]` 数、URL、aria-live 区域文本、拦截层调用计数。
2. 模拟 `button.click()`，等 500ms。
3. Post-snapshot：同上字段。
4. Diff 判定：
   - 新增 `[role=dialog]` 且文本含"确认 / 删除 / 无法撤销 / permanent / cannot be undone / discard" 关键词 → `already_satisfied`；
   - 无弹层但 aria-live 出现"已删除 / removed / deleted" 或拦截层记录到 fetch/WebSocket 试图发出破坏性 endpoint → `actionable`（客户端 optimistic 已触发或服务端已被调用，无保护）；
   - 无弹层也无变化 → `actionable`（无保护）。

未授权 / 生产环境 → 全无静态 signal 的按钮记 `suspected_no_protection`（视作 blocker），附建议"请人工确认这些破坏性动作是否有保护，或在 staging 环境启用半动态验证"。

## 五、边界 / 非目标

- 不要求所有危险操作都必须二次确认；确认疲劳有害。目标是"风险被理解、误操作不轻易酿成难恢复结果"，手段可以多样。
- 不主责按钮**文案**是否清晰表达对象 / 数量 / 影响 / 可撤销（→ 归 `13`）。本维度只审"哪些动作看起来是危险的" + "危险动作有没有保护机制"，不审文案措辞。
- 不主责非破坏性动作的即时反馈（pressed / loading / 当前态等）；本维度只关注可能造成难恢复后果的动作。
- 不要求为每个按钮加提示气泡；依场景而定。
