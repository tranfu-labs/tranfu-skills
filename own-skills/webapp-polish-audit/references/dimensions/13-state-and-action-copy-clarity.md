# 13. 关键状态与动作的可理解性 / State and Action Copy Clarity

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**，只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：操作前的即时可操作性 / 操作反馈归 `03`，动作完成后的结果反馈机制归 `12`；任何状态导致的布局移动归 `04`；数据 loading / 空 / 无结果 / 错误 / 权限 / 部分数据等**状态本身的存在与续接**归 `05`，表单特定输入 / 校验 / 保留 / 提交信心归 `07`；多记录扫描和动作范围归 `06`，导航范围和返回路径归 `08`；favicon / app icon / logo 归 `01`，主题归 `02`，同类元素 / 状态的视觉一致性归 `09`。本维度只主责这些状态与动作的**文案清晰度**——关键 CTA 标签、空/错误/权限/loading/结果文案、破坏性确认文案、动作范围措辞、跨页同义动作的一致命名。

**重要（只读约束）**：本维度的文案多数靠可见文本就能审，但破坏性确认框、提交/删除后的结果文案常需要副作用才能到达——这类按 `00` 环节二 2.4 记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`。**缺口是诚实结论，不是放过。** 只读够得到的（页面上已渲染的 CTA 标签、GET 来的空 / 错 / 权限态文案、已有的 loading 文案）正常审。

## 一、解决什么问题

文字能不能让用户**理解状态、动作、影响范围和下一步**：按下一个按钮前知道它真正会做什么；看到错误时知道怎么恢复；面对空 / 无结果时知道是没开始、没权限、筛太窄还是没加载完；做高影响动作时知道作用对象的数量与范围、是否立即生效、是否可撤销、是否通知他人或影响生产数据；看到的状态文案和系统真实状态一致；跨页同义动作不靠用户猜后果是否相同。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做（凡需副作用才能到达的状态，写「声明了但只读被阻挡」缺口）：

- **盘点关键 CTA 标签**：采集保存 / 发布 / 删除 / 发送 / 授权 / 付款 / 批量更新 / 导出 / 邀请 / 确认等动作按钮的可见文本，连同其触发的真实后果线索（按钮所在区块、关联对象、是否走副作用）。
- **盘点空 / 无结果 / 错误 / 权限 / loading 文案**：把页面上已渲染、或 GET 即可到达的空状态、无结果、错误信息、权限不足、加载中 / 处理中文案读出来；够不到的（如真实提交后的失败 / 超时 / 部分成功文案）记 `blocker` 缺口。
- **盘点破坏性确认文案**：危险 / 高影响动作的二次确认（删除 / 清空 / 停用 / 批量变更）的措辞——能静态够到（如点开即弹的确认 modal）就读文本；需真触发副作用才弹出的，记「声明了但只读被阻挡」缺口。
- **盘点动作范围措辞**：高影响动作文案里有没有写清作用对象（一项 / 当前页 / 全部筛选结果 / 整个账户）、是否立即生效、是否可撤销、是否通知他人、是否影响生产数据。
- **盘点跨页同义动作命名**：横向采集删除 / 移除 / 归档 / 关闭 / 取消 / 撤销 / 停用 / 禁用等动词在不同页面的用词，供一致性比对。
- **盘点状态文案与真实状态**：对够得到的状态，把显示文案（`Saved` / `Saving` / `Failed` / 「没有数据」/「马上完成」）和实际系统状态对上，看是否矛盾；够不到的态记 `blocker` 缺口。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**误判动作后果 / 状态含义 / 影响范围，或不知道怎么恢复、下一步做什么**"才定罪；否则按各分类的**释放理由**放过。够不到的破坏性确认 / 结果文案一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 关键动作文案不表达真实后果

形式：

- A1 用 `OK` / `Submit` / `Continue` / `Done` 当关键 CTA，实际却在发布 / 付款 / 删除 / 发送邀请 / 批量改数据。
- A2 按钮文案泛化到看不出会写数据、影响他人、花钱还是只是关闭弹层。
- A3 同一动作的标签比真实后果"轻"——例如 `Save` 实际会立即发布上线。
- A4 其它让用户按下前猜不到关键动作会做什么的标签。

判定：用探查盘点的 CTA 标签 + 真实后果线索 → 标签是否用模糊词掩盖了保存 / 发布 / 删除 / 发送 / 授权 / 付款 / 批量更新等实际后果 → 掩盖 = 嫌疑 → 在截图 / 可见文本上确认按下前确实读不出后果（够不到真实触发态则记 `blocker` 缺口）→ 定罪。释放：低风险、上下文明确的动作可以用短标签，只要关键后果没被模糊词掩盖。

### B. 错误信息不帮助恢复

形式：

- B1 只给 `Something went wrong` / `Invalid` / `Failed` 这类通用失败，用户不知道因为什么。
- B2 错误不说明是网络 / 输入 / 权限 / 筛选条件 / 服务端暂时不可用，用户分不清该做什么。
- B3 错误暴露内部异常 / 接口名 / 堆栈 / 无意义错误码，对用户毫无恢复价值。
- B4 其它让用户看完错误仍不知能不能重试、改哪里、换什么条件、联系谁的文案。

判定：到达该错误态（GET 来的错 / 权限态够得到就读，需真触发失败 / 超时才出现的记 `blocker` 缺口）→ 文案是否至少给出一条恢复线索（重试 / 改哪里 / 换条件 / 联系谁 / 稍后再试）→ 没有 = 嫌疑 → 确认用户确实无从恢复 → 定罪。释放：错误已清楚指向可执行的恢复路径，不必暴露技术细节。

### C. 空 / 无结果状态不指下一步

形式：

- C1 空状态只有一句欢迎语或一张插画，不说明为什么空、接下来做什么。
- C2 分不清是还没创建过、没有权限、筛选太窄、数据被清空，还是仍在加载。
- C3 无结果只写「没有数据」，不提示放宽筛选 / 换条件 / 新建入口。
- C4 其它让用户面对空 / 无结果时判断不了原因和下一步的文案。

判定：到达空 / 无结果态（多数 GET 即可够到）→ 文案能否让用户判断空的**原因**（未开始 / 无权限 / 筛太窄 / 数据清空 / 仍在加载）并给出**合理下一步** → 不能 = 嫌疑 → 截图 / 可见文本确认用户会卡在"不知为何空、不知做什么" → 定罪。释放：原因和下一步已清楚，不强求配插画 / 鼓励语 / 营销式说明。

### D. 危险 / 高影响动作不说明影响范围

形式：

- D1 破坏性确认只问「确定吗」，不说作用对象的数量与范围（一项 / 当前页 / 全部筛选结果 / 整个账户）。
- D2 不说明是否立即生效、是否可撤销。
- D3 不说明是否通知他人、是否影响生产数据 / 公开可见性 / 账户范围。
- D4 其它让用户在高影响动作前算不清"会波及多大范围"的确认文案。

判定：到达确认文案（点开即弹的 modal 够得到就读，需真触发副作用才弹出的记「声明了但只读被阻挡」`blocker` 缺口）→ 文案是否写清对象范围 / 数量 / 是否立即生效 / 是否可撤销 / 是否通知他人 / 是否影响生产数据 → 关键项缺失 = 嫌疑 → 确认用户会误判影响范围 → 定罪。释放：影响范围、可撤销性、波及对象已在文案中讲清。

### E. 状态文案与真实状态矛盾

形式：

- E1 按钮写 `Saved` 但其实仍在保存，或 `Saving` 卡住后不更新。
- E2 错误说「没有数据」但真实是权限不足；空态文案与实际状态对不上。
- E3 loading 文案说「马上完成」但任务可能很久，制造错误预期。
- E4 其它显示文案与实际系统状态对不上、让用户误以为已完成 / 没数据 / 可继续操作的情况。

判定：把够得到的状态文案与实际系统状态对上（够不到的态记 `blocker` 缺口）→ 文案是否与真实状态一致 → 矛盾 = 嫌疑 → 截图确认用户会因此误判已完成 / 无数据 / 可继续 → 定罪。释放：文案如实反映当前状态，不对结果或时长做无依据承诺。

### F. 同类动作 / 状态命名不一致

形式：

- F1 删除 / 移除 / 归档 / 关闭 / 取消 / 撤销 / 停用 / 禁用混用，含义相同却到处换词。
- F2 同一动作在不同页面换不同标签，用户无法判断后果是否相同。
- F3 含义不同的动作却用同一个词，掩盖了后果差异（如"删除"与"归档"都叫"移除"）。
- F4 其它让用户靠猜"这两处是不是同一回事"的命名。

判定：用探查横向采集的跨页动词 → 含义相同的是否用词一致、含义不同的是否明确区分 → 同义换词或异义同词 = 嫌疑 → 截图 / 可见文本确认用户会因此误判后果是否相同 → 定罪。释放：用词差异对应真实的语义差异，或在各自语境中后果已无歧义。

## 五、边界 / 非目标

- 不要求文案"优美 / 有品牌感 / 更像营销"；清楚、准确、可恢复优先。不规定固定语气、人格、幽默风格、品牌 voice 或 microcopy 风格。
- 不要求所有按钮都写成长句；低风险、上下文明确的动作可以短，但关键后果不能被模糊词掩盖。
- 不要求所有错误都暴露技术细节；错误应帮助用户恢复，而不是泄露内部异常、接口名、堆栈或无意义错误码。
- 不要求所有空状态都配插画 / 鼓励语 / 营销式说明；用户能判断原因和下一步才是目标。
- 不把 i18n、术语库、文案系统、内容治理流程、组件抽象或错误码架构写成本维度的 MUST。
- **数据 loading / 空 / 无结果 / 错误 / 权限 / 部分数据等状态本身的存在与续接归 `05`；本维度只主责这些状态出现后的文案清晰度。**
- **动作完成后的结果反馈机制（toast / banner / modal / inline / 动效形式是否合适、是否遮挡下一步）归 `12`；本维度只主责反馈文案是否准确表达对象 / 数量 / 影响 / 可撤销。** 即时可操作性 / 操作反馈归 `03`。
- 表单特定输入 / 校验 / 保留 / 提交信心归 `07`；多记录扫描和动作范围归 `06`；导航范围和返回路径归 `08`；任何状态导致的布局移动归 `04`；同类元素 / 状态的视觉一致性归 `09`；favicon / app icon / logo 归 `01`；主题归 `02`。
