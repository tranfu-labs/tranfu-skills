# 13. 破坏性动作与高影响动作的文案清晰度 / Destructive Action Copy Clarity

这是一条**审查维度**：证据由 `../pipeline/16-page-exploration-and-capture.md` 统一采集，本文件只写判定标准——① 解决什么问题；② 向采集环节申请的额外探查；③ 形式目录 + 判定。

本维度只主责两件事：高影响 CTA 按钮的文案是否**掩盖真实后果**，以及破坏性 / 高影响动作的确认文案是否**写清影响范围**。

## 一、解决什么问题

文字能不能让用户在按下高影响按钮前**理解真正会做什么**：

1. 关键 CTA（发布 / 付款 / 删除 / 发送 / 授权 / 批量更新 / 导出）的按钮文案是否被 "OK / Submit / Continue / Done" 这类泛化词掩盖了真实后果。
2. 高影响动作（删除 / 清空 / 停用 / 批量变更 / 影响钱权账户生产数据的动作）的确认文案是否写清：作用对象数量 / 是否立即生效 / 是否可撤销 / 是否通知他人 / 是否影响生产数据。

**只读约束**：破坏性动作的 confirm 弹层通常需真触发才能到达。按本文 §四·B 的授权门槛处理，不猜测、不放过。

## 二、向探寻引擎申请的额外探查（→ `16` 第 2 步）

引擎在池化执行时，为本维度额外做：

- **盘点关键 CTA 标签**：采集所有 `button` / `[role=button]` / `[type=submit]` 的 `{ text, aria-label, form_action, adjacent_form_fields, sibling_dialog_id }`。
- **盘点破坏性确认文案（静态可达部分）**：找 `[role=dialog]` / `[aria-modal=true]` / `<dialog>` 元素（含 hidden），提取其 title + body 文本 + primary / danger 按钮文本。
- **盘点动作范围措辞**：从上述文案中提取"对象数量 / 时效 / 可撤销 / 通知 / 影响面" 关键词。

需真触发副作用才能到达的 confirm 弹层按 §四·B 的授权门槛处理。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"用户按下前**误判动作后果**，或对确认框**误算影响范围**"才定罪；否则按各分类的**释放理由**放过。够不到的破坏性 confirm 弹层按 §四·B 授权门槛处理。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

> **本维度的判定读 JSON，不看图。** 证据是采集环节落盘的 DOM 结构、可见文本、样式向量、几何与状态属性。判断者不开浏览器——判不了就输出 `blocker` 并写清缺什么，由主代理决定重采还是动用 SKILL.md §0 的「受限截图」例外。**判断者自己去页面上看一眼是越权。**

### A. 关键动作文案不表达真实后果

形式：

- A1 用 `OK` / `Submit` / `Continue` / `Done` 当关键 CTA，实际却在发布 / 付款 / 删除 / 发送邀请 / 批量改数据。
- A2 按钮文案泛化到看不出会写数据、影响他人、花钱还是只是关闭弹层。
- A3 同一动作的标签比真实后果"轻"——例如 `Save` 实际会立即发布上线。
- A4 其它让用户按下前猜不到关键动作会做什么的标签。

判定：用探查盘点的 CTA 标签 + 真实后果线索 → 标签是否用模糊词掩盖了保存 / 发布 / 删除 / 发送 / 授权 / 付款 / 批量更新等实际后果 → 掩盖 = 嫌疑 → 确认按下前确实读不出后果 → 定罪。释放：低风险、上下文明确的动作可以用短标签，只要关键后果没被模糊词掩盖。

**证据与判定模式：`脚本+文本判`**。脚本采所有 `button` / `[role=button]` / `[type=submit]` 的 `{ text, aria-label, form_action, adjacent_form_fields, sibling_dialog_id }`，输出 JSON。LLM 读 JSON 用启发式判：

- 高疑：text 属于 `{ ok, submit, continue, done, confirm, 确定, 提交, 继续, 完成 }` 且 form_action 或 button context 指向破坏性 endpoint（`/publish` / `/delete` / `/pay` / `/invite` / `/remove` 等）或所在 form 含 payment / delete 相关字段；
- 低疑：text 明确含破坏性动词（"发布 / 删除 / 付款 / 邀请 / publish / delete / pay / invite"）；
- 中疑：其它，交由 LLM 综合上下文判断。

不需要截图。走 SKILL §4 的双判断者并行决胜结构。

### B. 危险 / 高影响动作不说明影响范围

形式：

- B1 破坏性确认只问"确定吗"，不说作用对象的数量与范围（一项 / 当前页 / 全部筛选结果 / 整个账户）。
- B2 不说明是否立即生效、是否可撤销。
- B3 不说明是否通知他人、是否影响生产数据 / 公开可见性 / 账户范围。
- B4 其它让用户在高影响动作前算不清"会波及多大范围"的确认文案。

判定：到达确认文案 → 文案是否写清对象范围 / 数量 / 是否立即生效 / 是否可撤销 / 是否通知他人 / 是否影响生产数据 → 关键项缺失 = 嫌疑 → 确认用户会误判影响范围 → 定罪。释放：影响范围、可撤销性、波及对象已在文案中讲清。

**证据与判定模式：`脚本先过滤 + 半动态验证（需授权）`（与 03.B 共享半动态触发条件）**。分两档：

**静态阶段**：脚本对每个 `[role=dialog]` / `<dialog>` 元素（含 hidden），提取 title + body 文本 + primary / danger 按钮文本，判是否包含：

- 对象数量表述：数字 + 单位（"3 items" / "2 项" / "5 条记录"），或明确范围词（"当前页" / "全部" / "所有筛选结果"）
- 时效表述："立即生效" / "24 小时内" / "immediately"
- 可撤销表述："可撤销" / "无法撤销" / "permanent" / "cannot be undone" / "undo available"
- 通知 / 影响：涉及他人时是否说明（"通知 3 名成员"）

关键项缺失 → 嫌疑。LLM 读 JSON 综合判定。

**半动态阶段（需授权，与 03.B 共用触发条件）**：仅当 SKILL §0 例外条款满足（`audit-state.json.allow_destructive_click_simulation === true` 且 URL 非生产环境）时，通过 03.B 描述的点击 + 比对流程触发弹层 → 抓取弹层文本 → 走静态阶段判定。

未授权 / 生产环境 → 无法到达的 confirm 弹层记 **`pending_authorization`**：写明发现了哪个破坏性 CTA、打算怎么验（拦截写请求后模拟点击抓弹层文案）、不验的代价（分不清「没有确认」与「确认存在但静态不可见」）、怎么授权（staging 重跑 + 写明允许模拟破坏性动作）。

## 五、边界 / 非目标

- 不要求文案"优美 / 有品牌感 / 更像营销"；清楚、准确、可恢复优先。
- 不要求所有按钮都写成长句；低风险、上下文明确的动作可以短，但关键后果不能被模糊词掩盖。
- 不主责错误信息文案 / 空状态文案 / loading 文案清晰度（非破坏性范畴）。
- 不主责跨页同义动作命名一致性（非破坏性范畴）。
- 不主责按钮**视觉**风险表达（→ 归 `03.A`）；本维度只审文案。
