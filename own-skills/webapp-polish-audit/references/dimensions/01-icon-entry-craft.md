# 01. 图标与产品入口质感 / Icon and Entry Craft

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**（怎么截图 / 怎么取 head link 都在 `00`），只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：界面内部功能图标的一致性归 `09`；icon-only 控件看不看得懂归 `10`；title / meta / Open Graph / theme-color / manifest 都不进入本维度（见第五节）。本维度只主责 **favicon / app icon / 产品图标 / logo mark 这些产品入口图标的可见完成度**。

## 一、解决什么问题

用户打开浏览器标签页、桌面 / PWA 启动入口或产品第一屏时，最先被肉眼感知到的是入口图标。这一维要回答的是：这个图标看起来**像个收尾过的产品，还是像没改完的脚手架**——有没有还挂着框架默认 favicon、是不是一块锐利方块、在 16px/32px 小尺寸下糊不糊、在暗色标签栏 / 暗色 shell 下能不能认出、没有品牌资料时有没有克制地处理而不是硬编一个过重的 logo。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做；需要副作用才能到达的状态记成「声明了但只读被阻挡」缺口，挂本维度名下）：

- **取入口图标资源清单**：从 head 盘点 `<link rel="icon">` / `apple-touch-icon` / SVG favicon 的实际引用路径与来源文件，连同浏览器标签页里**真正渲染出来**的 favicon 一起采集（源文件和实际渲染可能不一致）。
- **缩到小尺寸重采**：把入口图标渲染 / 缩放到 16px 与 32px，采集小尺寸下的可读截图。
- **深浅背景各采一次**：分别在浅色背景与暗色背景（暗色浏览器栏 / 暗色 app shell / 系统深色模式）下采集图标截图，看轮廓、主体、对比是否都成立。
- **暗色资源到达方式**：盘点是否声明了 light/dark asset 或 SVG/CSS `prefers-color-scheme` 分支；切到深色模式看暗色分支是否实际生效。需要副作用才能切换的，记「声明了但只读被阻挡」缺口。
- **比对框架默认指纹**：把采集到的 favicon 与已知框架默认图标（Vite、React、Next 等）做指纹比对，标出疑似未替换项。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**把入口读成未完成 / 脚手架，或在小尺寸·暗色下认不出图标**"才定罪；否则按各分类的**释放理由**放过。够不到的状态（如暗色切换被只读阻挡）一律记 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 框架默认图标未替换

形式：

- A1 标签页 favicon 仍是 Vite 的闪电 / React 原子 / Next 默认图标。
- A2 head 的 `<link rel="icon">` 仍指向脚手架生成的默认资源路径（如 `/vite.svg`、`/favicon.ico` 原封未动）。
- A3 `apple-touch-icon` / PWA 入口图标沿用模板默认。
- A4 其它让产品入口读起来像没改过的脚手架项目的默认图标。

判定：用探查的框架默认指纹比对 → 标签页实际渲染或 head 引用命中已知默认图标 = 嫌疑 → 截图确认这是产品真实入口、且确为未替换的框架默认（不是有意复用）→ 定罪。释放：图标已是产品自有资源，只是恰好风格相近。注意：这属于**图标质感**问题，不是 metadata 问题——不要因为 title / meta 缺失而在此判败。

### B. 容器型图标是生硬方块 / 无处理

形式：

- B1 图标是锐利尖角的生硬方形色块，直接贴满容器、无任何收边。
- B2 容器型图标既无圆角、也无 mask / 内边距 / 轮廓处理，边缘硬切。
- B3 图标主体顶满边界、没有安全内边距，缩小后糊成一坨。
- B4 其它让容器型入口图标读起来未经处理、不像成品的形态。

判定：盘点入口图标的容器形态 → 是锐利方块且无圆角 / mask / 内边距 / 轮廓任一处理 = 嫌疑 → 截图确认它读起来像未收尾的色块、不是有意的方形品牌造型 → 定罪。释放：方形是该品牌有意的视觉造型、或已有清晰收边处理。

### C. 深浅背景下辨识度不足

形式：

- C1 图标只在白底好看，放到暗色浏览器栏 / 暗色 shell 上糊掉或边界消失。
- C2 深色模式下图标主体与背景对比不足，几乎看不出轮廓。
- C3 声明了要适配深浅模式，但未提供 light/dark asset、也没用 `prefers-color-scheme`，深色下直接失效。
- C4 其它让图标在某一种背景下辨识度明显不足的情况。

判定：用探查在浅色与暗色背景各采的截图比对 → 任一背景下轮廓 / 主体 / 对比不成立 = 嫌疑 → 确认在常见暗色入口下用户确实认不出图标 → 定罪。若暗色分支需副作用才能切换、只读到不了，记 `blocker` 缺口而非放过。释放：深浅背景都有足够对比，或已用 light/dark asset / `prefers-color-scheme` 正确处理且实际生效。

### D. 小尺寸不可读

形式：

- D1 favicon 在 16px / 32px 下依赖复杂细节、细线网格，缩小后糊成一片。
- D2 图标里塞了长文字 / 完整单词，小尺寸后读不出。
- D3 用了低对比渐变 / 多层渐变，缩到 16px 后主体辨不清。
- D4 其它让入口图标在 16px / 32px 下不可读的细节过载。

判定：用探查缩到 16px / 32px 的重采截图 → 主体是否仍可辨、关键形状是否还在 = 嫌疑 → 确认小尺寸下用户认不出这是什么图标 → 定罪。释放：小尺寸下主体清晰、形状可辨，细节减省是有意为之。

### E. 无品牌资料时编造过重的 logo

形式：

- E1 没有任何品牌输入，却凭空编出复杂 logo / 吉祥物 / 含义过重的品牌系统。
- E2 入口图标堆了过多元素，超出"克制简单图形或字母标"应有的分寸。
- E3 直接拿锐利方块加一个字母草草交付，当作"简洁图标"（注意与 B 的区别：B 看形态收边，E 看在无品牌资料下的取舍是否克制、可信）。
- E4 其它在缺品牌资料时反而降低可信度的图标取舍。

判定：核对有无品牌资料输入 → 在无输入的前提下，图标是否做成了过度复杂的 logo / 品牌系统，或草率到不可信 = 嫌疑 → 确认这个取舍会降低产品可信度（用户读到"编的"或"没做完"）→ 定罪。释放：用克制的简单图形或字母标、与产品语境相称的处理。

## 五、边界 / 非目标

- `title`：太基础，不进入本维度。
- `meta description`：偏 SEO，不进入本维度。
- `theme-color`：有价值，但不是最高置信通用 MUST，依赖移动浏览器 / PWA 场景，不进入本维度。
- Open Graph：只对公开分享页面重要，不是通用 MUST，不进入本维度。
- `manifest`：只有 PWA / installable app 才重要，不进入本维度。
- 不评价完整品牌策略、品牌故事、营销文案或 logo 体系；本维度只看入口图标的可见完成度。
- 不主责界面内部按钮 / 导航 / 状态 / toolbar / 列表行操作里的**功能图标一致性**（→ `09`）。
- 不主责 icon-only 控件**看不看得懂 / 要不要文本 / `aria-label` / tooltip**（→ `10`）。
- 不把图标资源管线、构建配置、`<link>` 注入方式、PWA 实现写成本维度目标；MUST 是入口图标在用户眼里读起来已收尾、可识别。
