# 12. 操作结果的确认与动效分寸 / Result Feedback and Motion Proportion

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**，只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：操作前的可操作性、disabled、pressed、危险提交保护归 `03`；结果 / 确认 / 错误文案是否准确表达对象、数量、影响范围、是否可撤销归 `13`。本维度只主责**动作被触发之后**的结果确认、处理中反馈、恢复路径和动效分寸。

**重要（只读约束）**：本维度天然想去触发保存 / 删除 / 提交，但本 skill 只读、禁副作用。所以多数"成功 / 失败 / 处理中"态需要副作用才能到达——这类按 `00` 环节二 2.4 记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`。**缺口是诚实结论，不是放过。** 只读够得到的（已有 toast、导航 loading、GET 来的空 / 错态、骨架屏）正常审。

## 一、解决什么问题

动作触发后，用户能不能**减少结果不确定性而不被打断**：知道成功 / 失败 / 仍在处理；反馈强度匹配影响范围；反馈与动效不挡住下一步；可撤销动作的恢复路径看得见；长任务持续可感知；动效服务理解而非制造延迟。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做：

- **列动作**：盘点页面上有结果的动作（保存 / 创建 / 删除 / 复制 / 导出 / 上传 / 发送 / 刷新 / 生成 / 批处理）。
- **声明需要的反馈态**：对每个动作声明要到达的态——处理中 / 成功 / 失败·超时 / 部分成功 / 长任务进度 / 撤销入口。
- **到达只读可达的**：已有 toast、导航 loading、骨架屏、GET 来的空 / 错态——到达并截图。
- **够不到的记缺口**：需要真去保存 / 删除 / 上传等副作用才能触发的态——不触发，记「声明了但只读被阻挡」缺口，挂本维度名下。
- **盘点反馈层几何**：toast / banner / loading overlay / progress / 动画层出现时的位置与层级，记录是否压住保存 / 撤销 / 关闭 / 错误信息 / 主 CTA，尤其是否压住刚操作的区域。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**猜不到结果、错过结果、被反馈打断，或被动效拖慢 / 误导**"才定罪；否则按各分类的**释放理由**放过。够不到的反馈态一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 结果状态缺失（不知道成 / 败 / 处理中）

形式：

- A1 提交 / 保存 / 上传后只闪一下 loading，完成后没有明确的成功 / 失败态。
- A2 失败 / 超时静默，界面停在成功样式或毫无变化。
- A3 复制 / 局部刷新等动作完成后无任何可感知结果。
- A4 其它让用户靠猜动作是否成功的情况。

判定：到达该动作的结果态（只读够得到就看，够不到就记 `blocker` 缺口）→ 有没有明确区分成功 / 失败 / 仍在处理 → 缺失 = 嫌疑 → 截图确认用户只能靠猜 → 定罪。释放：结果已由 inline 提示 / 按钮态 / 列表变化等清楚表达。

### B. 反馈强度不匹配影响范围

形式：

- B1 删除数据 / 批量变更 / 生产数据操作只给一闪而过的小提示。
- B2 复制一个字段、切换轻量设置却弹出巨大 modal，打断任务。
- B3 高影响动作反馈过轻且无恢复路径。
- B4 其它反馈强度与动作影响明显错配的情况。

判定：对比动作影响范围与反馈强度 → 高影响却过轻、或轻动作却过重 = 嫌疑 → 截图确认用户会错过结果或被无谓打断 → 定罪。释放：反馈强度与影响范围匹配。

### C. 反馈遮挡下一步关键操作

形式：

- C1 toast / banner / loading overlay 挡住保存 / 撤销 / 关闭 / 错误信息 / 主 CTA。
- C2 反馈盖住刚刚操作的区域。
- C3 动画层拦住后续点击。
- C4 其它让反馈层挡住下一步的情况。

判定：用探查盘点的反馈层几何 → 与关键操作的位置是否重叠 → 重叠 = 嫌疑 → 截图确认确实挡住下一步 → 定罪。释放：反馈避开关键操作区与刚操作区。

### D. 撤销 / 恢复路径不可见

形式：

- D1 用 undo 替代二次确认，但撤销入口停留过短、位置隐蔽、文案不清。
- D2 撤销不说明作用于哪个对象 / 动作。
- D3 其它让用户来不及理解或恢复的情况。

判定：到达可撤销动作后（够不到则记 `blocker` 缺口）→ 撤销入口是否可见、可理解、可及时使用、对象清楚 → 否 = 嫌疑 → 确认用户会来不及恢复 → 定罪。释放：恢复路径明显、停留充分、对象清楚。

### E. 长任务无持续反馈

形式：

- E1 上传 / 导出 / 生成 / 批处理 / 保存大量数据只在点击瞬间闪一下，之后界面像卡住。
- E2 长任务没有进度，也没有"仍在处理"的感知。
- E3 其它让用户以为系统卡死的长任务。

判定：到达长任务的处理中态（够不到则记 `blocker` 缺口）→ 是否持续可感知（进度 / 状态文本 / 持续动画）→ 否 = 嫌疑 → 确认用户会误判为卡死 → 定罪。释放：有持续的处理中感知，哪怕不是精确进度。

### F. 动效制造延迟或干扰判断

形式：

- F1 长转场 / 复杂动效让用户干等。
- F2 频繁动效让列表增删看不清，或让人误判操作是否完成。
- F3 其它为了"精致"而拖慢或遮蔽结果的动效。

判定：观察转场 / 展开 / toast / loading / 列表增删动效 → 是否人为延迟关键操作、或让结果看不清 → 是 = 嫌疑 → 截图或逐帧确认 → 定罪。释放：动效服务状态理解、不增加等待、不遮蔽结果。

## 五、边界 / 非目标

- 不要求所有动作都 toast；反馈可以是 inline 提示、状态文本、按钮态、banner、progress、modal、列表状态变化等任何可感知形式。
- 不要求所有反馈都有动画；没有动画但结果清楚，通常优于有动画却增加等待或误判。
- 不规定 toast / banner / inline / modal 哪个是唯一正确形式；形式取决于影响范围、风险、持续时间和下一步任务。
- 不把具体 animation duration、easing、transition 属性或动画库写成 MUST。
- 不要求都有复杂进度条；MUST 是用户能持续知道处理状态。
- 不主责操作前的可操作性 / disabled / pressed / 危险二次确认（→ `03`）、反馈文案是否准确表达对象 / 数量 / 影响 / 可撤销（→ `13`）。
- 不把请求状态管理、toast provider、队列实现、动画组件抽象当作本维度目标。
