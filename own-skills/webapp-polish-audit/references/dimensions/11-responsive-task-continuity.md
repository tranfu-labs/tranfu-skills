# 11. 响应式下的任务连续性 / Responsive Task Continuity

这是一条**审查维度**：证据由 `../pipeline/16-page-exploration-and-capture.md` 统一采集，本文件只写判定标准——① 解决什么问题；② 向采集环节申请的额外探查；③ 形式目录 + 判定。

边界：可交互元素有没有即时反馈 / 可操作性归 `03`；输入方式（鼠标 / 键盘 / 触摸）、感知条件（缩放 / 对比度 / 暗色）、纯图标含义归 `10`。状态导致的布局移动、同类元素的视觉一致性不在主流程五维度覆盖范围内。本维度主责**视口 / 容器尺寸变化后关键任务能否继续被找到、理解、安全操作并完成**（A–E 类），**以及内容装不下导致的截断 / 挤坏**（F 类，不限视口——桌面上主按钮孤字折行与窄屏上金额被截断是同一缺陷）。

## 一、解决什么问题

用户在桌面以外的尺寸下还能不能把核心任务做完。窄屏、小窗口、分屏、容器变窄、移动端弹出键盘时，关键操作（保存 / 提交 / 关闭 / 返回 / 搜索 / 筛选 / 分页 / 主 CTA / 行操作 / 批量操作 / 详情入口）会不会消失、被挤出屏幕、被遮挡或藏进难发现的横向区域；表单 label 与输入框、列表项与操作、表格列与行、错误与字段、价格与单位在重排后会不会断开关系；宽表格横向滚动后用户能不能看出还有列、还够得到行操作；弹层在小屏下会不会困住用户；同一控件在桌面与移动端语义会不会漂移；为塞下内容会不会把关键判断信息截断到失去含义。

## 二、向探寻引擎申请的额外探查（→ `16` 第 2 步）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做；需要副作用才能到达的状态记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`）：

- **窄视口 / 小窗口重盘点**：在窄屏（`16` 通用采集已切的 `<=390x844` 档）和更小的容器宽度下重做元素盘点，供本维度逐类比对；视口 / 容器控制不可用则如实记缺口。
- **横向溢出排查**：盘点页面级与容器级的 `scrollWidth > clientWidth`，定位哪些是宽表格 / 对比视图 / toolbar / 横向列表造成的溢出，记录右边界外还藏了什么列 / 操作。
- **关键操作可达性复核**：在窄视口下逐个核对保存 / 提交 / 关闭 / 返回 / 搜索 / 筛选 / 分页 / 主 CTA / 行操作有没有被挤出屏幕、被 sticky header/footer 或 cookie bar 遮挡、或退到横向滚动才够得到。
- **宽表格 / 对比视图重排盘点**：盘点宽表格、数据矩阵、看板、时间线、图表在窄屏下的重排结果（横向滚动 / 折叠 / 换行 / 隐藏列 / 转卡片），记录关键列、行操作、危险操作是否仍可见或可达。
- **弹层小屏几何**：对弹层 / drawer / menu / dropdown / date picker / popover，在窄视口下记录其高度是否超出视口、关闭 / 确认 / 取消 / 当前选项 / 错误信息是否被挤出可视区。
- **移动键盘遮挡复核**：键盘弹出会改变可视高度——只读够得到就在窄视口下复核底部提交 / 取消 / 关闭 / 当前输入 / 字段错误是否被遮挡；够不到键盘真实弹出态的，记「声明了但只读被阻挡」缺口。
- **桌面 / 窄屏语义对照**：对同一控件（局部筛选 / 行操作 / 详情入口等）对照两档视口的措辞与作用范围，记录语义是否漂移。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户在窄视口 / 小窗口 / 容器变窄 / 移动键盘下**找不到关键操作、误解信息关系或操作范围、看不出数据还有更多、被弹层困住，或失去判断依据而做不完核心任务**"才定罪；否则按各分类的**释放理由**放过。够不到的状态（如键盘真实弹出）一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

> 本维度全部 class 为 `脚本直判`，判据是确定性数值/存在性比较——各操作卡里的「不要截图」是**正常路径**的约束。极端情况下主代理仍可走 SKILL.md §0 的「受限截图」例外，但那需要先跑完脚本并说清缺什么。

### A. 窄视口下关键操作消失或不可达

**目标**：找出窄视口 / 小窗口下，保存 / 提交 / 关闭 / 返回 / 搜索 / 筛选 / 分页 / 主 CTA / 行操作 / 批量操作 / 详情入口等关键操作是否被挤出屏幕、被遮挡或整体消失——这类操作一旦不可达，用户在移动端就做不完核心任务。

形式（不是穷举，命中同类即可挂嫌疑）：

- A1 保存 / 提交 / 关闭 / 返回 / 主 CTA 在窄屏或小窗口下被挤出屏幕，需横向滚动才能找到。
- A2 搜索 / 筛选 / 分页在移动端跑到不可达位置，或被 sticky header/footer、cookie bar 遮挡。
- A3 行操作 / 批量操作 / 详情入口在窄屏下整列消失，没有替代入口。
- A4 其它让关键操作在窄视口下需要横向猜测才能找到的情况。

**定罪线**：窄视口下关键操作落在视口边界外、被遮挡或整体消失，且确认它是关键任务路径、用户确实找不到或够不到该操作（无可见替代入口）才定罪。释放：窄屏下移入收纳菜单 / 更多入口 / 底部 bar 等仍可见可达的合理收纳。

**证据与判定模式：`脚本直判`**。脚本在窄视口下取所有可点元素矩形，判断是否越界或被遮挡，并检测同容器内的收纳入口。纯几何 + 存在性 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **用 `resize_page` 到 375×812（或断言 `window.innerWidth === 375`）** —— 因为判定依赖真实窄视口下的几何数据，视口不对，后续所有矩形都不可信。
- **一次性 `evaluate_script` 遍历可点元素，批量拿 rect + text + aria-label** —— 因为批量采集能保证同一时刻的几何一致，避免中途布局变化（如动画、懒加载）导致前后矩形对不上。
- **溢出标记用 rect 边界比较（`right>innerWidth` 等），不用目测** —— 因为数值判据可复核、可回归；"挤没挤出屏幕"靠肉眼因人而异，不可复核。
- **先用 text/aria-label 关键词圈定"关键动作"，再判几何** —— 因为越界本身不是缺陷，只有关键操作越界且无替代入口才是；先圈定范围能避免把普通按钮的越界也算进来，抬高查准。
- **检测同容器内是否有收纳入口（`aria-haspopup` / `aria-controls` / 汉堡按钮）** —— 因为这是释放条件的自动化版本：越界但收进了菜单仍然可达，不构成缺陷。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `text` / `aria-label` / `rect` / `outOfViewport` 判据（`right`/`left`/`bottom` 越界）/ `isCriticalAction` / `hasFallbackEntry` / 遮挡来源（sticky header/footer/cookie bar）。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 窄视口档位 | 375×812（常见窄屏档位） | 本 skill 经验值，代表常见最窄主流机型，仅作嫌疑信号；实际断点应以该页面为准 |

**不要这样做**（踩过的坑，硬边界）

- **不要按屏幕分段截图** — 长页判溢出/裁切靠几何，不需要 13 段。
- 不要派判断者。
- 不要在 `window.innerWidth !== 375` 时跑 mobile 检测。

### B. 重排后信息关系断开

**目标**：找出窄屏重排后，表单 label 与输入框、列表项与操作、表格列与行、错误信息与字段、价格与单位之间的邻近关系是否被拆散——关系一旦断开，用户会搞错这次操作 / 错误 / 单位归属于哪个对象。

形式（不是穷举，命中同类即可挂嫌疑）：

- B1 表单 label 与输入框在窄屏重排后上下错位、对不上是哪个字段。
- B2 列表项与其操作、表格列与行在重排后断开，分不清操作作用于哪条。
- B3 错误信息与字段、价格与单位在重排后被拆到看不出归属。
- B4 其它让用户在窄屏下搞不清"信息属于哪里"的重排断裂。

**定罪线**：label / 操作 / 单位 / 错误与其对应对象的邻近关系被重排破坏，且用户无法靠位置 / 标题 / 边框等线索判断归属，会搞错字段 / 对象 / 单位，才定罪。释放：重排后另有清晰线索（就近标题 / 边框 / 显式标签 / 行内绑定）使关系仍可读。

**证据与判定模式：`脚本直判`**。脚本配对 `<label>` 与其输入框，算垂直间距与轴线对齐，并检查错误信息与字段的矩形邻接。纯几何 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **`resize_page` 到 375** —— 因为重排断裂只在窄视口下出现，桌面宽视口下 label 与 input 通常同行，不会触发这类缺陷。
- **用 `label[for]` / label 包裹关系找对应 input，不按视觉邻近猜配对** —— 因为 DOM 关联是权威的语义配对来源；几何邻近只用来验证"重排后是否还挨着"，反过来当配对依据会把两个恰好挨着的无关字段错配。
- **垂直距离与一行高（约 24px）比较、检查水平是否同轴** —— 因为这是"错位"的可复核几何定义：距离超过一行高或不同轴，用户就很难靠视觉判断这个 label 对应哪个输入框。
- **错误信息与字段的邻接关系用 rect 相邻判断，不只信 `aria-describedby`** —— 因为该属性未必总被正确使用，几何邻接是用户实际感知到的线索，脚本判据应贴近用户体验而非只信任标记是否存在。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`labelSelector` / `inputSelector` / `verticalGap` / `aligned` / `splitByReflow` / 就近标题或边框等兜底线索。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 错位判据 | 垂直距离 ≥ 一行高（约 24px） | 本 skill 经验值，非标准；该页面通用行高明显不同时以实际行高为准 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 判"看得出归属"。
- 不要派判断者。
- 关系判断靠 DOM 结构 + 几何位置，不猜视觉。

### C. 横向滚动让用户误判范围或够不到操作

**目标**：找出宽表格 / 对比视图横向滚动后，用户能不能看出还有更多列、够不够得到关键列 / 行操作 / 危险操作——看不出范围或够不到操作，用户会误判数据缺失或做不了该做的操作。

形式（不是穷举，命中同类即可挂嫌疑）：

- C1 宽表格只加了横向溢出，无任何"右边还有列"的提示，用户误以为数据缺失。
- C2 关键列 / 当前行 / 行操作藏在横向滚动右侧，难发现。
- C3 危险操作（删除等）被推到横向滚动深处或难发现位置。
- C4 其它让用户看不出滚动范围、或在横向滚动中够不到 / 误触关键操作的情况。

**定罪线**：容器命中横向溢出且无滚动提示 / 列边界 / 关键列保留，或行操作 / 危险操作落在溢出区，且用户确实看不出还有更多列、或够不到行操作、或误判数据缺失、或难以安全触发危险操作，才定罪。释放：横向滚动本身允许，只要有可感知的滚动线索、关键列 / 行操作仍可理解并安全触发。

**证据与判定模式：`脚本直判`**。脚本对溢出容器算 `scrollWidth > clientWidth`，并检测滚动线索（渐隐遮罩 / 指示图标 / `scroll-snap-type`）与溢出区内的危险操作。数值 + 存在性 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **`resize_page` 到 375** —— 因为横向溢出多数在窄屏下才触发，宽视口下表格通常能完整展示。
- **遍历 `table` / `[role=table]` / overflow-x 容器，算 `scrollWidth > clientWidth`** —— 因为这是横向溢出的确定性判据，比目测"是不是被切掉"更可靠、可回归。
- **先检测滚动线索（渐隐遮罩 / 指示图标 / `scroll-snap-type`），不是有溢出就直接算缺陷** —— 因为横向滚动本身允许，只有用户感知不到"还有更多"或够不到关键列时才算缺陷；先找线索能自动过滤掉大量设计如此的正常横向滚动。
- **检查溢出区内是否含危险操作 / 行操作 selector，单独标记** —— 因为溢出区里藏的是普通信息列还是危险操作，严重度完全不同，需要分开报告而不是笼统一条溢出。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`containerSelector` / `scrollWidth` / `clientWidth` / `hasScrollHint` / `dangerActionsInOverflow` / 关键列是否落在溢出区内。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 横向溢出判据 | `scrollWidth > clientWidth` | 存在性判据，非数值阈值——这本身就是"存在溢出"的定义，无需另设门槛 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 判"用户看不看得出还有列"。
- 不要派判断者。
- 有 scroll hint 就放过；本 class 不硬要求"绝不允许横向滚动"。

### D. 弹层在小屏下困住用户或被键盘遮挡

**目标**：找出弹层 / drawer / menu / date picker 在小屏或移动键盘弹出下会不会困住用户——关闭 / 确认 / 取消按钮跑出可视区，或当前选项 / 错误信息看不见摸不着，用户就出不来、做不完。

形式（不是穷举，命中同类即可挂嫌疑）：

- D1 弹层 / drawer / menu / date picker 在小屏下高度超出视口，关闭 / 取消 / 确认按钮被挤出可视区。
- D2 弹层里的当前选项 / 错误信息在窄屏下不可见或不可达。
- D3 移动键盘弹出后提交 / 取消 / 当前输入 / 字段错误被键盘或底部 UI 遮挡。
- D4 其它让用户在小屏弹层 / 键盘弹出态下出不来或做不完的情况。

**定罪线**：窄视口弹层的关闭 / 确认 / 取消 / 当前选项 / 错误信息超出视口或被遮挡，且用户确实无法关闭 / 确认 / 取消或看不到关键信息而被困住，才定罪；键盘遮挡态只读够不到则记 `blocker` 缺口、不猜测。释放：弹层在小屏下可滚动到达且关键控件 / 信息始终可见可触发。

**证据与判定模式：`脚本直判`**。脚本取弹层及其关闭 / 确认 / 取消按钮在窄视口下的矩形，判断是否超出视口。纯几何 → 主代理直接判 → 键盘真实弹出态属副作用，记 `blocker` 不猜测。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **按行为找，不按 role 枚举** —— 因为浮层的实现方式千差万别：`role=dialog`、`role=listbox`、`role=menu`、`aria-haspopup`，还有大量纯 `div + onClick` 什么都不标的。枚举 role 是白名单思路，永远在追加清单、永远漏下一个——实测教训就是曾经只找 `[role=dialog]`，纯 div+onClick 的浮层全漏。正确做法是：可点的就点，点完看多出了什么。
- **候选取所有可点元素（`button`/`a`/`[role=button]`/`[onclick]`/`cursor: pointer`），不预筛 role** —— 因为预筛 role 等于先假设了"弹层长什么样"，会把不遵循 ARIA 规范但确实是弹层的实现排除在候选之外。
- **触发点击有多个来源，按可用性取第一个能用的** —— 因为不同运行时对 `evaluate_script` 的写权限限制不同，写死一种方式在受限运行时会整个卡死：
  1. 浏览器工具提供的点击能力（如 click / computer 一类交互动作）—— 不依赖页面执行域的写权限，通常最可靠，优先试。
  2. `evaluate_script` 里 `el.click()` 或 `dispatchEvent` 构造点击事件 —— 需要页面执行域可写。
  3. 静态推断（DOM 里已存在但 hidden 的，临时 `el.hidden = false` 取 rect 后复原）—— 只作兜底，不作首选。
  三条都试过、全不通才记 `blocker`，并写明每条各自的失败原因。
- **写请求拦截钩子只是"可能触发写操作的点击"的前置条件，不是所有点击的通行证** —— 因为纯展开菜单、tab 切换、语言切换这类点击本身不发请求、不改数据，风险等同于 `10.B` 的 Tab 走查；这类点击即使装不上拦截钩子也可以直接点。只有命中破坏性词表（提交/删除/购买等）的触发控件才必须先装上拦截钩子，装不上就跳过该控件、记 `pending_authorization`，不能因此把整个 class 记成 `blocker`。
- **逐个点击前后各拍一次 DOM 快照，取 diff** —— 因为"点出了什么"只有对比前后状态才能确定；只看点击后的静态 DOM，无法区分"这是刚出现的浮层"还是"页面本来就有的内容"。

  ```js
  const before = snapshot();                 // 记录所有可见元素的 selector + rect
  el.click();
  await new Promise(r => setTimeout(r, 300));
  const after = snapshot();
  const appeared = diff(after, before);      // 新出现或从不可见变可见的元素
  ```

- **`appeared` 非空即算点出浮层，不管有没有 role** —— 因为判定依据是"用户看到了新内容"，role 只是锦上添花的元数据，不该成为判定的门槛；对新出现的容器取几何，检查是否 `rect.bottom > innerHeight`、内部关闭 / 确认 / 取消是否落在视口外、当前选中项是否可见。
- **顺带把 `appeared` 里可点元素的矩形喂给 `10.C`** —— 因为展开态的选项此前从未被量过（折叠时 `display:none` 拿不到有效矩形），这是唯一能拿到这批矩形的时机，错过这次点击就要重新点一遍。
- **复原后再测下一个（按 Esc，或再点一次触发控件，或点击浮层外部，确认 DOM 回到 `before` 状态）** —— 因为弹层状态会互相污染：上一个没关掉，下一个的 `before` 快照就不干净，diff 出来的 `appeared` 会把两层混在一起。
- **输出时记录 `hasRole`** —— 因为这个字段本身就是用来观察"按 role 枚举会漏掉多少"，是验证"按行为找"这个方法论有效性的数据来源。
- **键盘弹出遮挡只读到不了就记 `blocker`，不猜测** —— 因为键盘真实弹出是运行时副作用态，静态修改无法模拟真实键盘高度，猜出来的数据比没有数据更危险——它会被当作已验证的结论。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`triggerSelector` / `appearedContainerSelector` / `hasRole` / `rect` / `overflowsViewport` / `closeInViewport` / `confirmInViewport` / `currentSelectionInViewport` / `optionRects`。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 弹层溢出视口 | `rect.bottom > innerHeight` | 存在性判据，非数值阈值——底部超出可视区即挂嫌疑 |
| 点击后快照等待 | 约 300ms | 本 skill 经验值，覆盖大多数 CSS 过渡 / JS state 更新耗时；页面动画明显更慢时以实测耗时为准 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 判"是否被遮挡"——矩形几何就是结论。
- 不要派判断者。
- **不要只找静态 DOM 里的 dialog 就收工**：条件渲染的弹层不点开永远是 `dialogs: []`，那不是「没有弹层」而是「没查到」。
- **命中破坏性词表的触发控件，点击前必须先装写请求拦截**；拦到写请求立即停止、复原并记 `pending_authorization`。低风险点击（展开菜单 / tab 切换 / 语言切换等）不受此限制，即使拦截钩子装不上也可以直接点。
- 键盘弹出遮挡属真副作用态，只读到不了记 `blocker`，不猜。
- **不要因为 `evaluate_script` 不能写就整个 class 记 `blocker`** —— 实测教训：某次运行里 11.D 以「页面 evaluate 全局不可扩展，写请求拦截无法安装，条件弹层无法安全点击测量」记了 blocker，而同一次运行里 `press_key`（10.B）、`resize_page`（10.E）都正常工作，说明受限的只是 `evaluate_script` 的写权限，浏览器工具层的交互能力仍然可用。先把点击的三条来源都试一遍，全不通才记 `blocker`，并写明每条各自的失败原因。

### E. 响应式变化改变任务语义

**目标**：找出同一控件在桌面与窄屏之间措辞或作用范围有没有漂移——局部当成全局、单行当成批量、本地当成外部，用户会在窄屏下误判操作后果。

形式（不是穷举，命中同类即可挂嫌疑）：

- E1 桌面上的"筛选当前列表"在移动端措辞 / 行为变得像全局搜索。
- E2 桌面上作用于单行的行操作，在移动端读起来像批量操作。
- E3 桌面上的本地详情入口，在移动端变得像跳到外部导航。
- E4 其它让同一控件在桌面与窄屏间作用范围 / 含义漂移的情况。

**定罪线**：桌面 / 窄屏语义对照中同一控件的措辞或作用范围发生漂移，且用户会在移动端误判操作范围（局部当全局、单行当批量、本地当外部），才定罪。释放：两档视口下作用范围与语义一致，仅布局形态不同；纯图标兜底不算漂移。

**证据与判定模式：`脚本直判`**。脚本在两档视口下分别取同一稳定选择器的文案与可访问名，做比对。字符串比对 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **两档视口都用同一份脚本采集 `{stableSelector, textContent, ariaLabel}`** —— 因为只有同一选择器在两个视口下的文案才有可比性；`stableSelector` 优先用 `[data-testid]` / `[id]`，退化到 tag+className+index，是为了保证比对的是同一个元素，而不是布局变化后误配对的两个元素。
- **desktop 用 1280×720，mobile 用 375×812** —— 因为这是常见的桌面与移动主流分辨率组合，能覆盖大多数响应式断点两端的形态差异。
- **用 DOM label 文本比对，不跨视口比对截图** —— 因为语义漂移是文字 / aria-label 内容的变化，字符串比对精确且可复核；截图比对既测不出"作用范围"变了，也容易被布局差异干扰产生假阳性。
- **只在漂移指向不同作用范围（局部 vs 全局 / 单行 vs 批量 / 本地 vs 外部）时才定罪** —— 因为纯图标兜底（desktop 留文字、mobile 只留图标）是常见的空间压缩手段，不代表语义变了；这道闸把"格式变化"和"语义漂移"分开。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `desktopLabel` / `mobileLabel` / `semanticDrift`（是否变化且非纯 icon fallback）/ 变化后指向的作用范围类别。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 视口对照档位 | desktop 1280×720 / mobile 375×812 | 本 skill 经验值，代表常见桌面与移动主流分辨率；该页面实际断点不同时以该断点为准 |

**不要这样做**（踩过的坑，硬边界）

- 不要跨视口比对截图，用 DOM label 比对。
- 不要派判断者。
- 纯图标兜底 (desktop text, mobile 只保留图标) 不算漂移，只算格式变化。

### F. 内容装不下：截断或挤坏

**目标**：找出因为塞不下而让用户**读不全关键信息**、或让控件**显得未完成**的地方。

不限视口、不限语言——桌面上主按钮孤字折行、窄屏上金额被截断、英文文案把 padding 挤没，是同一个缺陷的不同表现，不因出现在哪块屏幕或哪种语言而改变归属。

形式（不是穷举，命中同类即可挂嫌疑）：

- F1 关键数值 / 金额 / 对象名称被截断到读不全。
- F2 状态 / 错误原因 / 确认后果被压成只剩图标或省略号，失去判断依据。
- F3 操作对象被截断，用户分不清这次操作作用于谁。
- F4 控件标签被迫折行，末行只剩孤字 / 极短残句，读起来像没做完。
- F5 文字撑满容器、余量被吃光，或溢出撑破对齐、盖到相邻元素。
- F6 其它因塞不下而读不全、或让控件显得未完成的压缩。

**定罪线**：无 `title` / 提示气泡 / 展开机制等可达补全，**且**信息本身是判断依据（金额 / 状态 / 对象 / 后果）或该控件是关键操作。补全本身的可达性归 `10`。

**证据与判定模式：`脚本直判`**。全部判据是几何与文本的确定性比较 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **用 Range 测真实行数** —— 对元素本身调 `getClientRects()` 对 `button`、`td` 等块级 / inline-block / table-cell 控件**恒为 1**，测不出内部折行。

  ```js
  const range = document.createRange();
  range.selectNodeContents(el);
  const rects = [...range.getClientRects()].filter((r) => r.width > 0 && r.height > 0);
  const lineCount = rects.length;
  const lastRatio = lineCount >= 2 ? rects.at(-1).width / rects[0].width : 1;
  ```

- **只对叶子文本元素测行数** —— 元素含子元素时，`selectNodeContents` 拿到的是**各子元素盒子的矩形数**（`button > span + svg` 得到 3），那不是行数。真正被折行的永远是承载文本的那一层。**但这只约束「在哪测行数」，不是候选集门槛**——见 `16` 的测量通用原则第二条。
- **量文字到容器边框的实际距离，左右分开** —— 因为用户看到的「挤不挤」就是这个距离，而**它跟「宽度够不够」不是一回事**。实测：`a.home-product-cta` 英文版按「内容盒宽 − 文字宽」只差 −0.8，看着像"差一点点"；但文字左边距边框只有 **0.6px**（padding 声明 16px，文字整体左移了 15px），右边却空 30.6px。宽度类指标测不出文字画在哪。
  - 取法：文字矩形（在承载文本那层建 Range）与容器矩形（约束宽度那层）的 `left` / `right` 差值。左右必须分开——`Start training` 左 0.6 / 右 30.6，取平均或取和都会把问题抹平。
  - 跨层配对关系（文字取自哪个 selector、容器取自哪个）要进证据，见 `16` 通用原则第一条。
- **采集范围要宽于判据** —— 判据只决定「报什么」，不该决定「采什么」。**实测教训**：顶格的按钮不折行、不裁切、无省略号，三类嫌疑一个都不命中，于是连数据都没留下；等加上余量判据时，只能重跑整轮审计才能验证。采得宽，新判据可以在历史 probe 上直接回溯。
- **同类元素比分布** —— 同页所有按钮的余量中位数 36px、而某几个是负值，**离群本身就是信号**，比任何固定阈值都贴合这个页面。分布也能发现我没预设过的缺陷形态。
- **跨 locale 比分布，不只比单个元素** —— 实测 tranfu.com：中文余量中位 36px、英文 25.7px，中文 1 个负值、英文 6 个。**即使每条单独看都够不上定罪线，中位数下滑与负值数量翻倍本身就是结论**：布局宽度只按一种语言调过。分布差异达 2 倍时单独产出一条 finding。
- **跨 locale 并列测同一 selector** —— 中文「开始免费试用」6 字与英文 `Start Free Trial` 16 字符，在同一按钮宽度下表现完全不同。**一个控件只按一种语言的文案宽度设计过**，是这类缺陷最常见的成因。任一语言命中即定罪，某语言通过不能替另一语言背书。
- **两档视口都跑** —— 桌面那次不能省：主按钮孤字折行只在宽视口下出现。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `locale` / `viewport` / `text` / `textLength` / 真实行数 / 末行首行比 / `scrollWidth` 与 `clientWidth` / 左右 padding / 文字实际宽度 / **余量**（内容盒宽 − 文字宽）/ `whiteSpace` / `textOverflow` / `display` / 有无补全 / 是否关键信息。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 孤字折行 | 末行/首行 < 0.6 | 有意的多行排版不会把末行留成孤字——这个比值把「挤坏」和「设计如此」分开 |
| 真裁切 | 溢出 ≥ 4px | 亚像素舍入与字体度量误差通常 1–3px（实测页脚版权行 202 vs 205 就是噪声） |
| 长文案释放 | `textLength > 20` | 按钮标签通常 ≤10 字；30–40 字的卡片链接标题折两行是正常排版。`isControl` 区分不了「控件标签」与「控件里承载的长标题」，这条才是那道闸 |
| 文字贴边 | 单侧边距 **< 8px** | 8px 约半个字符宽，是文字的呼吸下限。低于它就是贴边——不必等到负数，**刚好卡住本身就是隐患**（字体渲染在不同设备有差异、文案随时可能改） |

> 实测分界（tranfu.com，同一批控件两语言对照）：
>
> | 控件 | 左距 | 右距 | |
> | --- | --- | --- | --- |
> | 中文 `开始训练` | 15.9 | 45.6 | ✓ |
> | 中文 `查看全部` | 16 | 16 | ✓ auto 宽度，天然合格 |
> | 中文 `加入社群` | 24 | 24 | ✓ |
> | 英文 `Start training` | **0.6** | 30.6 | ✗ 贴边 |
> | 英文 `OfferPilot Interview` | 24 | **−4.9** | ✗ 已画到边框外 |
> | 英文 `English` | 32 | **6** | ✗ 偏紧 |
>
> **中英分界清清楚楚，零误报**——中文与 auto 宽度按钮最小 15.9，全部通过。
>
> 严重度按边距大小分档：`−4.9`（溢出边框）> `0.6`（几乎贴死）> `6`（偏紧）。负数与正数的区别是严重度，**不是「算不算问题」**。

其余释放：`white-space: pre-line`、`min-height` 已容纳多行、`line-clamp` 且有展开、有可达补全。

**不要这样做**（踩过的坑，硬边界）

- **不要截图判「看着挤不挤」** —— 行数、宽度比、余量像素就是结论。
- **不要让判据决定采集范围** —— 见上文顶格的实测教训。
- **不要把明细压成汇总** —— 实测踩过：整份 probe 427 字节，11 个嫌疑 0 条明细，无法复核等于没有证据。汇总统计只能作附加字段。
- **不要对元素本身数行数**，也不要对含子元素的元素建 Range —— 两者拿到的都不是行数。
- **不要把 `scrollWidth > clientWidth` 直接当裁切** —— `display: inline` 的元素两者恒等（没有真正的滚动盒），3px 以内是舍入噪声。
- **不要对段落 / 列表项判孤字折行** —— `P` / `LI` / `BLOCKQUOTE` / `DD` / `TD` / `ARTICLE` 的折行是正常排版。孤字折行只对控件成立。
- **不要报视觉隐藏元素的裁切** —— `sr-only` 的典型实现就是 `width:1px + overflow:hidden + nowrap`，必然假命中；按 `rect ≤ 1px` / `clip: rect(0,0,0,0)` / `clipPath: inset(50%)` / 类名含 `sr-only|visually-hidden|screen-reader-text` 剔除。
- **不要只测一种语言** —— 英文嫌疑数是中文 3 倍时，差异本身就是结论。
- **不要只报「疑似偏小」** —— 每条 finding 必须带具体数值与 selector。
- **不要用「宽度够不够」代替「文字画在哪」** —— 踩过的错：按内容盒宽算余量只差 −0.8，实际文字左边距边框 0.6px。宽度类指标对文字位置偏移完全不敏感。
- **不要给贴边设负向噪声门槛** —— 踩过的错：设 `< −1px` 才算，于是 `Start training`（0.6px）整个漏掉。噪声门槛是从**溢出像素的测量误差**推出来的，套到边距上没有依据。
- **不要引入「宽度是否写死」这类成因判断** —— 踩过的错：为了不误报 auto 宽度按钮，想先判断宽度是不是固定的。但那要枚举 `width` / `flex-shrink` / `max-width` / 网格固定列宽…永远列不全，而**换成量边距后 auto 按钮天然合格（16 / 24px），根本不需要这个前置条件**。
- **不要静默跳过隐藏态元素** —— 实测发现同名 CTA 有 8 个 `rect=0`（在未激活的 tab 面板里），**只有当前激活的那个被测到**，其余产品的按钮从未检查过。要么切 tab 后测（属低风险交互），要么记 gap 说明哪些没测——「测不到」和「没问题」不是一回事。

> 判据可用 `node scripts/fit-check.mjs` 在对照页 `scripts/fixtures/fit.html` 上复验（7 用例）；带 URL 参数可扫真实页面。改判据后跑一次能立刻发现退化——本类的判据被实测推翻过四次才定下来。

## 五、边界 / 非目标

- 不规定具体断点、viewport 阈值或容器宽度；文中数值（如 `<=390x844`）只是**嫌疑信号**，不当硬性必须 / token。
- 不规定必须 mobile-first，不规定用 grid / flex / container query / 媒体查询 / CSS overflow / sticky column 或特定组件库实现。
- 不要求所有桌面功能在手机上等价高效；本维度的必须是关键任务不能断。
- 不要求复杂数据表在手机上一定变成卡片；横向滚动也可以，只要用户理解范围且能安全操作。
- 不把"所有页面都要完美适配所有设备"写成无限范围；检查重点是常见尺寸变化下的核心任务连续性。
- **换行 / 挤坏不分视口，一律归本维度 F 类**：桌面与窄屏下的标签装不下是同一缺陷，判据见 F 类（`nowrap` 裁切、孤字折行、省略号截断）。但**折行本身不是缺陷**——只有末行 < 首行 × 0.6 的孤字折行才定罪，正常多行排版一律释放。
- 不主责仅悬停、键盘焦点、触摸目标、低对比、页面放大或纯图标控件含义——输入 / 感知（键盘 / 触摸 / 缩放 / 对比度）归 `10`。
- 即时可操作性 / 操作反馈归 `03`；数据状态的文案清晰度归 `13`。动作完成后的结果反馈、状态导致的布局移动、数据 loading / 空 / 无结果 / 错误 / 权限态本身，均不在主流程五维度覆盖范围内。
- 不把内部代码结构、响应式 token、breakpoint 命名、组件抽象或测试工具选择当作本维度目标。
