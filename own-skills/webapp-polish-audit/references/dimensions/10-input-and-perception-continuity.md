# 10. 不同操作方式下不阻断任务 / Input and Perception Continuity

这是一条**审查维度**：证据由 `../pipeline/16-page-exploration-and-capture.md` 统一采集，本文件只写判定标准——① 解决什么问题；② 向采集环节申请的额外探查；③ 形式目录 + 判定。

边界：窄屏重排、移动端键盘遮挡、横向滚动、桌面/移动语义漂移归 `11`；可交互元素有没有即时反馈（悬停/pressed/disabled/loading 有没有反应）归 `03`。同类元素/状态的视觉一致性不在主流程五维度覆盖范围内。本维度只主责**输入方式**（鼠标/键盘/触摸）和**感知条件**（缩放/对比度/暗色）下，关键任务会不会被阻断。

## 一、解决什么问题

关键任务（删除/编辑/提交/保存/关闭/筛选/分页/导出/刷新/更多菜单/详情入口等）在常见输入方式和感知条件下，能不能被**发现、理解、触发、完成**——不被悬停依赖、键盘陷阱、过小触摸目标、低对比、缩放遮挡、纯图标语义这六类挡住。

## 二、向探寻引擎申请的额外探查（→ `16` 第 2 步）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做；需要副作用才能到达的状态记成「只读被阻挡」缺口）：

- **键盘走查**：用 Tab 走一遍主流程，记录焦点顺序、焦点是否可见；对弹层 / drawer / menu / dropdown / date picker / command palette / wizard，记录能否 Tab 出来、Esc / 关闭后焦点落到哪。
- **关掉悬停重走**：把只在悬停 / 行悬停 / 卡片悬停 / 工具栏浮层才出现的操作翻出来，逐个看有没有非悬停替代路径。
- **放大重盘点**：在 200%（或系统大字号）下重做元素盘点与几何计算，看关键控件 / 错误信息 / 当前焦点是否被遮挡或挤出（用矩形重叠面积判，不截图）。
- **触摸目标尺寸**：盘点关闭 / 删除 / 分页 / checkbox / 更多菜单 / 表格行操作 / 移动 toolbar 的命中区尺寸。
- **状态表达方式**：盘点错误 / 选中 / 危险 / 成功 / 禁用，记录是否只靠颜色 / 低对比，有没有形状 / 图标 / 文本 / 位置等冗余线索。
- **纯图标控件**：盘点删除 / 导出 / 筛选 / 更多 / 关闭 / 刷新等纯图标按钮，有没有文本 / `aria-label` / 提示气泡 / 上下文。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户在常见输入 / 感知条件下**发现不了、理解不了、触发不了或完不成关键任务**"才定罪；否则按各分类的**释放理由**放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

> 本维度全部 class 为 `脚本直判`，判据是确定性数值/存在性比较——各操作卡里的「不要截图」是**正常路径**的约束。极端情况下主代理仍可走 SKILL.md §0 的「受限截图」例外，但那需要先跑完脚本并说清缺什么。

### A. 悬停依赖阻断关键任务

**目标**：找出只在悬停时才出现、且没有非悬停替代路径的关键操作——触摸屏和键盘用户永远触发不了悬停，这类操作对他们等于不存在。

形式（不是穷举，命中同类即可挂嫌疑）：

- A1 删除 / 编辑 / 更多 / 详情入口只在悬停出现，无非悬停替代。
- A2 行 / 卡片操作只在悬停浮现，触摸屏与键盘用户够不到。
- A3 工具栏 / 浮层动作仅悬停可见。
- A4 其它把关键任务锁死在悬停里的情况。

**定罪线**：仅悬停可见且同容器内无等价非悬停路径（可见按钮 / 菜单入口 / 行内动作 / 详情页），且确认这是关键任务、触摸 / 键盘用户确实做不了，才定罪。释放：纯装饰性悬停，或有等价的非悬停路径。

**证据与判定模式：`脚本直判`**。脚本遍历可点元素并比对 `mouseover` 前后的计算样式与子元素可见性（或解析样式表里的 `:hover` 规则），输出 `appearsOnHoverOnly` / `hasFallbackPath`。存在性判断，确定性 → 主代理直接组装 finding → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **按行为找，不按样式表枚举** —— 因为扫 stylesheet 里的 `:hover` 规则是白名单思路：JS 驱动的 hover 显隐（`onMouseEnter` 改 state）根本不在样式表里，会整个漏掉——实测教训就是曾经只扫样式表 `:hover` 规则，JS 驱动的 hover 显隐全漏。正确做法是真的 hover 一下看变化。
- **一次性遍历可点元素及常见 hover 宿主容器（列表项 / 卡片 / 表格行）** —— 因为悬停显隐的触发对象不只是按钮本身，很多时候是整行 / 整卡在 hover 后才浮出操作，只扫可点元素会漏掉这类容器级 hover。
- **触发悬停有多个来源，按可用性取第一个能用的** —— 因为不同运行时对 `evaluate_script` 的写权限限制不同，写死一种方式在受限运行时会整个卡死：
  1. 运行时 / 浏览器工具提供的 hover 能力（如 hover 一类交互动作）—— 不依赖页面执行域的写权限，通常最可靠，优先试。
  2. `evaluate_script` 里 `dispatchEvent(new MouseEvent('mouseover', {bubbles: true}))` 构造事件 —— 需要页面执行域可写；`mouseover` 是纯浏览器事件，不发请求、不改数据，无需拦截保护，风险为零。
  3. 静态推断（解析样式表里的 `:hover` 规则、读 hidden 元素的声明样式）—— 只作补充信号，不作唯一来源。
  三条都试过、全不通才记 `blocker`，并写明每条各自的失败原因。
- **无论用哪种触发方式，都前后各拍一次子树快照比对**（每个后代的 `display` / `opacity` / `visibility` / `pointer-events` / rect）—— 因为唯一能确认"这个元素是不是仅悬停可见"的方法，就是真的触发 hover 后观察变化；只看触发后的静态 DOM 无法确定"这是 hover 才有的"还是"本来就在"。
- **触发后用对应方式复原（移开鼠标 / `mouseout` 事件）** —— 因为不复原会让下一个元素的 hover 状态被污染，比对出的"hover 后才可见"可能混入前一次的残留。
- **样式表扫描只作补充信号，不作唯一来源** —— 因为纯 CSS `:hover` 规则能覆盖的只是一部分实现方式，把它当唯一来源就是重蹈"只查白名单"的覆辙。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `text` / `appearsOnHoverOnly` / `hasFallbackPath`（同容器内是否有等价非 hover 入口）。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 悬停可见判据 | 无独立数值阈值 | 存在性判断——是否仅悬停可见（前后子树可见性变化）、是否存在替代路径，均为布尔判据 |

**不要这样做**（踩过的坑，硬边界）

- 不要派采集者 / 判断者 / 验收者子代理。
- 不要 `take_screenshot` 判"悬停显示什么"。
- 不要真点击悬停元素（避免副作用）。
- 不要走"每张截图一个判断者"分发。
- **不要因为 `evaluate_script` 不能写就整个 class 记 `blocker`** —— 实测教训：某次运行里 10.A 以「运行时无法构造 MouseEvent，且样式表扫描不能排除 JS 驱动 hover-only 显隐」记了 blocker，而同一次运行里 `press_key`（10.B）、`resize_page`（10.E）都正常工作，说明受限的只是 `evaluate_script` 的写权限，浏览器工具层的交互能力仍然可用。先把 hover 的三条来源都试一遍，全不通才记 `blocker`，并写明每条各自的失败原因。

### B. 键盘被困或丢失位置

**目标**：找出键盘走查中会把用户困住或让用户丢失当前位置的地方——弹层里 Tab 进得去出不来、关闭后焦点乱跑或消失、焦点本身不可见，键盘用户就没法安全操作。

形式（不是穷举，命中同类即可挂嫌疑）：

- B1 弹层 / drawer / menu / date picker 里 Tab 进得去出不来。
- B2 关闭浮层后焦点跑到页首或消失。
- B3 焦点不可见，用户看不出当前在哪。
- B4 其它让键盘用户卡住或迷失的情况。

**定罪线**：浮层里 Tab 进得去出不来、关闭浮层后焦点跑到页首或消失、或焦点不可见，且确认会困住或迷失键盘用户，才定罪。释放：无浮层、无键盘陷阱的简单页面。

**证据与判定模式：`脚本直判`**。脚本用 `press_key` 走 Tab 序列，逐步记录 `document.activeElement` 与 outline 宽度，输出 `tabSequence` / `trappedContainers`。序列是否离开弹层、outline 是否可见均为数值/布尔判据 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **循环按 Tab 20-40 次，每次记录 `document.activeElement` 的 stable selector + rect + outline 宽度** —— 因为键盘走查的核心证据就是焦点序列本身，只有连续记录才能判断"有没有走出弹层"和"焦点顺序是否合理"。
- **不要因为写请求拦截钩子装不上就跳过 Tab 走查** —— 因为按 Tab 键只移动焦点，不发请求、不改数据，本就不需要拦截保护。拦截是「点击可能触发写操作的控件」时才需要的前置条件，别把它扩大成所有交互的通行证。
- **打开弹层本身按低风险交互处理，默认允许** —— 因为要走查弹层内的 Tab 序列就必须先把弹层打开；装得上写请求拦截就先装上作为额外保险，装不上则退到静态 hidden 态取几何（同 `11.D` 的兜底方式），两条都不通才记 `blocker`。
- **判断"困住"用 Tab 序列是否离开弹层容器，而非固定次数** —— 因为不同弹层内可聚焦元素数量不同，用"连续超过该弹层内可聚焦元素总数仍未离开"作为判据，比拍一个固定数字更贴合每个弹层的实际情况。
- **用 outline 宽度判断焦点是否可见** —— 因为用户判断"当前在哪"完全依赖视觉可见的焦点指示；`outline: none` 之类样式把它抹掉又没有替代样式时，键盘用户就彻底丢失位置感。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`tabSequence`（selector / inModal / focusVisible）/ `trappedContainers` / outline 宽度 / 关闭浮层后焦点落点。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 单次 Tab 走查次数 | 20–40 次 | 本 skill 经验值，覆盖大多数页面主流程的可聚焦元素数量；页面可聚焦元素明显更多时以实际数量为准 |
| focusVisible 判据 | outline 宽度 ≥ 2px | 本 skill 经验值，用于判断焦点框是否有实际可见宽度；页面用非 outline 手段（如背景色变化）表达焦点时以实际效果为准 |
| trapped 判据 | 连续 Tab 次数超过该弹层内可聚焦元素总数仍未离开 | 存在性判据，非固定数值——用弹层自身元素数标定，不用统一次数 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 判 focus 位置。
- 不要派判断者。
- **不要因为写请求拦截钩子装不上就跳过 Tab 走查**：按 Tab 键只移动焦点，不发请求、不改数据，本就不需要拦截保护。拦截是「点击可能触发写操作的控件」时才需要的前置条件，别把它扩大成所有交互的通行证。
- 打开弹层属**低风险交互**，默认允许：装得上拦截就先装，装不上则改用静态 hidden 态取几何（见 11.D）；两条都不通才记 `blocker`。

### C. 触摸目标过小

**目标**：找出关闭 / 删除 / 分页 / checkbox 等关键操作的命中区是否小到容易被触摸误点——命中区不是视觉大小，是用户手指实际能点中的范围。

形式（不是穷举，命中同类即可挂嫌疑）：

- C1 关闭 / 删除 / 分页 / checkbox 命中区过小，易误点。
- C2 更多菜单 / 行操作 / 移动 toolbar 目标过密、相互挤压。
- C3 危险操作和相邻操作贴太近，触摸易误触。
- C4 其它在触摸下需要精细点击才能完成的关键操作。

**定罪线**：命中区明显偏小（约 `<40–44px` 量级，仅作嫌疑信号，不当硬性数值），且确认是关键 / 危险操作、触摸下确实容易误点，才定罪。释放：低频、低风险且有撤销路径的小目标。

**证据与判定模式：`脚本直判`**。脚本取每个可点元素的 `getBoundingClientRect`，与 44px 门槛比较。纯数值比较 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **遍历常见可点元素类型（`button`/`a[href]`/`[role=button]`/checkbox/radio/`[role=tab]` 等）** —— 因为这些是触摸误点后果最大的交互类型，覆盖它们能抓住绝大多数触摸目标过小的关键场景。
- **量的是命中区，不是图标** —— 因为用户点击的是整个可点元素的矩形范围，不是内部的 `svg` / `span` 图标；只测图标尺寸会系统性低估命中区，把实际够大但图标画小的按钮也误判成缺陷。取可点元素自身的 `getBoundingClientRect()`；若可点元素本身尺寸为 0（纯包裹容器），才向下找第一个有实际尺寸的子元素，并记录取自哪层（见 `16` 的测量通用原则第一条）。
- **`::before` / `::after` 或负 margin 扩出的命中区不计入，但命中时在证据里注明** —— 因为 `getBoundingClientRect()` 本来就取不到伪元素扩出的可点范围，这是已知的测量偏差方向（只会低估不会高估），如实记录比强行修正更可靠。
- **用 44×44 作为嫌疑门槛** —— 因为这是 WCAG 2.5.5 的标准值，不是本 skill 拍的；48×48 是 Material Design 更严格的要求，44 是两者中更宽松、误报率更低的门槛，适合作嫌疑级信号。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `text` / `width` / `height` / `actionable`（`width<44 || height<44`）/ 命中区取自哪一层（自身或向下找到的子元素）/ 是否有伪元素扩展命中区。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 触摸目标下限 | 44×44px | WCAG 2.5.5 标准值 |
| 更严格参考 | 48×48px | Material Design 经验值，比 WCAG 更严 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 判"看着大不大"。
- 不要派判断者。
- 不要报"疑似偏小"却不给具体 px 数字。

### D. 状态只靠低对比或颜色

**目标**：找出只靠颜色或过低对比来表达的关键状态（错误 / 成功 / 选中 / 禁用）——色弱、暗色环境、视觉疲劳下这些状态会读不出来。

形式（不是穷举，命中同类即可挂嫌疑）：

- D1 错误 / 成功只用红 / 绿，色弱场景不可辨。
- D2 选中 / 禁用只靠淡边框或微弱底色。
- D3 关键文本对比过低，暗色 / 投影 / 疲劳下读不出。
- D4 其它让关键状态在常见感知条件下不可辨的情况。

**定罪线**：状态只靠颜色 / 低对比、无形状 / 图标 / 文本 / 位置等冗余线索，且确认状态确实分不出，才定罪。释放：颜色之外另有清晰冗余线索。

**证据与判定模式：`脚本直判`**。脚本算文字色与背景色的对比度并检测图标 / 文本前缀等冗余线索，与 4.5 / 3.0 阈值比较。纯数值比较 → 主代理直接判 → 无需看图兜底（模拟色弱不在本 skill 范围）。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **筛选范围用 role / aria-invalid / class 关键词圈定状态元素** —— 因为要先圈定"这是在表达状态"的元素，再去测对比度；否则全页面测对比度会产生大量与状态无关的噪音数据。
- **背景色必须跨层求，不能只取元素自身 `backgroundColor`** —— 因为元素自身背景极常见是 `transparent` / `rgba(...,0)`，直接拿来算等于跟透明比，数值完全不可信。沿祖先链上溯找第一个非透明背景，找不到则用页面根背景兜底；元素或祖先有半透明背景时按 alpha 合成。把背景色取自哪个 selector、上溯了几层记进证据（见 `16` 的测量通用原则第一条）。

  > 实测（tranfu.com，248 个带文字元素）：**195 个（79%）自身背景透明**，真背景需上溯 1–3 层。只取自身背景的话，这条判据对四分之三以上的元素产出的都是无效数值。
- **用 WCAG 对比度公式（相对亮度 → `(L1+0.05)/(L2+0.05)`）而非目测颜色深浅** —— 因为对比度是可复核的数值判据，"看着够不够深"因显示器、环境光而异，不可复核也不可回归。
- **检查图标 / img / text prefix 等冗余线索，未达阈值但有冗余线索则释放** —— 因为色弱用户的问题根源是"只靠颜色区分状态"，只要还有另一条非颜色的感知通道，状态依然可辨，不构成本类缺陷。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `ratio` / `threshold` / `hasIcon` / `hasTextPrefix` / 背景色取自哪一层及上溯层数 / `actionable`。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 正文对比度下限 | 4.5:1 | WCAG 标准值 |
| 大字号（≥18pt 或 14pt bold）对比度下限 | 3.0:1 | WCAG 标准值 |
| 背景透明比例 | 实测 79%（248 个元素中 195 个） | 本 skill 实测（tranfu.com），说明跨层求背景色不是边缘情况而是常态 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 模拟色弱。
- 不要派判断者。
- contrast 未达阈值但有图标 / text prefix / 形状冗余 → 释放，不报。

### E. 缩放或大字号下遮挡 / 不可达

**目标**：找出放大到 200% 或大字号下，fixed / sticky 元素是否会遮挡关键控件、错误信息或当前焦点——低视力用户依赖缩放 / 大字号阅读，遮挡会让他们完成不了任务。

形式（不是穷举，命中同类即可挂嫌疑）：

- E1 fixed toolbar / drawer / toast / cookie bar 放大后挡住提交 / 错误 / 当前焦点。
- E2 关键控件放大后跑出可操作范围。
- E3 错误信息放大后被遮挡。
- E4 其它在缩放 / 大字号下让关键控件不可达的情况。

**定罪线**：200% 缩放或大字号下关键控件 / 错误信息 / 当前焦点被 fixed / sticky 元素遮挡或挤出可操作范围，且确认会阻断任务，才定罪。释放：放大后仍可滚动到达、无遮挡。

**证据与判定模式：`脚本直判`**。脚本在 200% 缩放下取固定 / 粘性元素与关键控件的矩形，算重叠面积。纯几何计算 → 主代理直接判 → 无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **按半视口宽 → 放大根字号 → `zoom` 的顺序尝试，不要停在第一个失败上就记 `blocker`** —— 因为三条路可靠性递减、被运行时拦截的概率递增：半视口宽只是改变视口尺寸，从不被拦截；根字号是常规样式写入，几乎总能成功；`zoom` 属性在部分运行时会被拦截写入。按可靠性从高到低试，才能在大多数页面上拿到数据，而不是一遇到 `zoom` 被拦就放弃整个 class。

  1. **半视口宽（首选，最可靠）**：`resize_page` 到 `640×720`。CSS 像素减半 ≈ 200% 缩放的布局效果，且不改页面任何样式。
  2. **放大根字号**：`evaluate_script` 设 `document.documentElement.style.fontSize = '32px'`（默认 16px 的两倍）。`rem` 布局下这就是大字号场景的主要效果；这属于允许的只读探查——只改渲染样式，不导航、不提交、不写数据，与「翻开静态可达隐藏面」同级。
  3. **`zoom`（末选）**：`document.documentElement.style.zoom = 2`。部分运行时会拦截对 `documentElement.style` 的写入，被拦就换上面两条，不要在此打转。

  三条都不可用才记 `blocker`，并写明各自失败原因。
- **采数覆盖 fixed/sticky 元素 + 主内容区 + 当前焦点元素** —— 因为遮挡的施害方是 fixed/sticky 元素，受害方是用户正在操作或需要看到的内容（主内容区、当前焦点），两者都要拿到矩形才能算重叠。
- **用矩形重叠公式（`Math.max(0, min(right)-max(left)) * Math.max(0, min(bottom)-max(top))`）算重叠面积，不目测遮挡** —— 因为重叠面积是可复核的数值判据，"看着挡没挡住"在缩放 / 大字号这种非常规布局下尤其容易看走眼。
- **完成后必须恢复 `fontSize` / `zoom` 并切回原视口** —— 因为这些修改会持续影响页面渲染，不恢复会污染后续 class 采集到的几何数据，导致后面的判定跟着一起错。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`fixed_selector` / `overlapping`（`target_selector` + `area`）/ 当前实际生效的探查路径（半视口宽 / 根字号 / zoom）/ 三条路各自的失败原因（若有）。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 半视口宽 | 640×720 | 本 skill 经验值，对应 1280×720 标准桌面视口的一半宽度，模拟约 200% 缩放的布局效果 |
| 大字号 | 根字号 32px（默认 16px 的两倍） | 本 skill 经验值，对应约 200% 大字号场景 |
| zoom 倍数 | 2（200%） | 本 skill 经验值，对齐前两条路径的缩放比例 |

**不要这样做**（踩过的坑，硬边界）

- 不要 `take_screenshot` 判"是否被遮挡"——重叠面积就是结论。
- 不要派判断者。
- **改完必须恢复**：`fontSize` / `zoom` 置空、视口切回原尺寸，否则污染后续 class 的几何数据。
- 不要因为 `zoom` 被拦就直接记 `blocker`——那是三条路里最不可靠的一条，先试半视口宽和根字号。

### F. 纯图标语义不明

**目标**：找出没有任何文本 / `aria-label` / 提示气泡的纯图标关键操作——用户必须靠猜才能知道这个图标是干什么的。

形式（不是穷举，命中同类即可挂嫌疑）：

- F1 删除 / 导出 / 筛选 / 更多 / 关闭 / 刷新等纯图标按钮无文本 / 标签 / 提示气泡。
- F2 图标语义不通用，用户需要猜含义。
- F3 其它让用户靠猜才能用关键操作的纯图标控件。

**定罪线**：纯图标控件完全没有文本 / `aria-label` / 提示气泡 / 上下文等语义来源，且确认用户必须靠猜才能使用该关键操作，才定罪。释放：空间紧张但语义通用且风险低（如熟悉的搜索放大镜）。

**证据与判定模式：`脚本直判`**。脚本筛出无文字且含图标子元素的控件，检查可访问名来源（`aria-label` / `aria-labelledby` / `title` / 视觉隐藏文本）。存在性判断 → 主代理直接判 → 本类只判「完全无名」，不判「名字好不好」，故无需看图兜底。

---

**推荐怎么找**（每条都附理由；理由不成立时可自行换法，换了要在 probe 里说明）

- **遍历 `button`/`a`/`[role=button]`，用"无文本 + 含图标子元素"过滤纯图标控件** —— 因为只有先圈定"纯图标"（没有伴随文字）的控件，才谈得上"语义不明"；带文字的图标按钮不在本类范围内。
- **检查 accessible name 的完整来源链（`aria-label`/`aria-labelledby`/`title`/视觉隐藏文本），不只看可见文本** —— 因为可访问名不一定是可见文本，很多纯图标按钮靠 `aria-label` 或视觉隐藏的 `sr-only` span 提供语义，只查可见文本会把这些实际上有名字的按钮误判为缺陷。
- **只判"完全无 name"，不判"name 好不好"** —— 因为"图标语义通不通用"是主观判断，脚本无法可靠判定；"完全没有可访问名"是存在性判据，客观、可复核，把判定范围收窄到这条能避免引入不可靠的主观判断。

**值得量的东西**（不是必填清单，够复核即可；遇到清单外的信号自行补）

`selector` / `iconType` / `hasAccessibleName` / accessible name 来源（`aria-label`/`aria-labelledby`/`title`/视觉隐藏文本）/ `actionable`。

**参考值**（实测校准，非普适真理；页面分布明显不同时以分布为准）

| 信号 | 参考值 | 本质 |
| --- | --- | --- |
| 纯图标判据 | `textContent` trim 后为空，且含 svg / `[class*=icon]` / `<i>` / 图标 font 的 `:before`/`:after` | 存在性判据，非数值阈值 |
| 定罪判据 | accessible name 完全缺失 | 存在性判据；"name 好不好"不在本类判定范围 |

**不要这样做**（踩过的坑，硬边界）

- 不要派判断者子代理判"图标看不看得懂"。
- 不要 `take_screenshot`。
- 有 accessible name（即使不通用）也放过；本 class 只判"完全无 name"，不判"name 好不好"。

## 五、边界 / 非目标

- 不把具体 ARIA 属性、`role`、`tabindex`、focus trap 库或屏幕阅读器 API 写成必须；这些是可能的实现手段。
- 不要求做完整 WCAG 审计，也不把本维度扩展成合规报告。
- 不要求每个图标按钮都强制显示文字；空间、熟悉度、上下文、风险会影响方案。
- 不禁止所有悬停 reveal；只有当关键任务没有非悬停替代路径时才违反。
- 不要求对所有辅助技术达到完美支持；但关键任务不能在常见输入 / 感知条件下明显阻断。
- 不主责窄屏 / 移动端键盘 / 横向滚动 / 响应式重排（→ `11`）、即时交互反馈（→ `03`）。视觉一致性不在主流程五维度覆盖范围内。
- 不把组件抽象、设计系统 token、lint 规则或测试框架写成本维度目标。
