# 02. 深浅模式体验等价性 / Light-Dark Experience Equivalence

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**（怎么切主题 / 怎么截图 / 怎么盘点都在 `00`），只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：单一主题内的即时可操作性 / hover / pressed / disabled / loading 反馈归 `03`；同类元素 / 状态的视觉一致性归 `09`；favicon / app icon / logo 归 `01`；`theme-color` / manifest / meta / OG 不在本维度。本维度只主责**产品已暴露 light/dark mode、主题切换、system theme 跟随或多主题展示时，各主题是否等价、可读、可辨、可操作，且切换不丢任务上下文**。本维度不要求产品新增 dark mode。

## 一、解决什么问题

产品如果给了浅色 / 深色（或多主题），用户在任一主题下都该拿到同等质量的体验：文字 / 按钮 / 边界 / 图标 / 表格 / 输入框读得清、关键状态（选中 / 禁用 / 危险 / 成功 / 警告 / loading / error / hover / pressed）含义认得出、主题特定的 logo / 插图 / 透明图不消失不脏边、深色不是"只把背景刷黑"而层级 / 边界 / 可点性垮掉。而且切主题时，正在填的表单、选好的筛选、当前 tab、滚动位置不能被冲掉。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做；需要副作用才能到达的状态记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`）：

- **切换浅 / 深主题并重截**：通过页面自带的 theme toggle、或只读可达的主题入口，切到每个可达主题，对同一核心流程双视口整页重截，逐主题各得一份。toggle 不存在 / 切换需登录或写偏好等副作用才能持久时，记缺口。
- **system theme 跟随**：若产品声明跟随系统主题，在只读可达范围内观察 `prefers-color-scheme` 浅 / 深两侧的呈现；只读够不到系统级切换时，记缺口。
- **跨主题状态重盘点**：对环节一翻出的同一状态语义（selected / disabled / danger / success / warning / loading / error / hover / pressed），在每个主题下重做盘点，带上**样式向量**（背景色 / 文字色 / 边框 / 圆角 / 字号 / 字重 / 内边距），供逐主题横向比对。
- **对比主题特定资源**：盘点 logo / 图标 / 插图 / 空状态图 / 透明 PNG·SVG 在每个主题下的实际呈现（是否消失、糊、脏边、对比不足），逐主题留截图配对。
- **切换前先布上下文，再切**：在有未提交输入 / 已选筛选 / 非默认 tab / 已滚动位置时触发只读可达的主题切换，切换后重盘点这些上下文是否仍在。注意：只布只读可达的上下文，切换本身若触发整页 reload / 重建以外的写副作用（如保存偏好到服务端）才算到达，则记缺口。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户在某个主题下**读不清 / 辨不出状态 / 操作不了**，或**切换主题丢失当前任务**"才定罪；否则按各分类的**释放理由**放过。够不到的主题 / 状态一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 某一主题下读不清 / 操作不了

形式：

- A1 切到 dark 后正文 / 辅助文字对比骤降，暗色下读不出。
- A2 按钮 / 输入框 / 表格 / 卡片的边界在某主题下消失，分不出可操作区。
- A3 图标 / 分隔线 / 阴影在某主题下融进背景，看不见。
- A4 其它让某一主题明显比另一主题难读 / 难操作的情况。

判定：用探查的逐主题重截 → 在每个主题下看正文 / 辅助文字 / 按钮 / 图标 / 表单 / 表格 / 导航 / 浮层 / 边界 / 可点区域是否清楚 → 任一主题明显读不清或操作不了 = 嫌疑 → 截图对比确认该主题确实让用户看不清文字 / 按钮 / 边界 / 图标 / 表格 / 输入框 → 定罪。释放：两主题都达到可读可操作，差异不影响发现 / 理解 / 触发任务。

### B. 跨主题状态含义不一致

形式：

- B1 某状态（selected / danger 等）在一个主题用明确状态色，另一主题只靠轻微明暗变化表达，含义变模糊。
- B2 disabled / loading / error / success 在某主题下与正常态几乎不可分辨。
- B3 同一危险语义在两主题强度落差大，一边醒目一边几乎无感。
- B4 其它让关键状态在某主题下辨不出含义的情况。

判定：用探查的跨主题状态重盘点 → 对每个状态语义比对两主题的样式向量与可辨性 → 某主题下该状态与正常态分不开、或含义与另一主题不一致 = 嫌疑 → 截图（含暗色）确认用户在该主题下认不出选中 / 禁用 / 危险 / 成功 / 警告 / loading / error / hover / pressed → 定罪。释放：两主题各自都能被辨认，表达手段不同但含义一致。

### C. 主题切换丢失任务上下文

形式：

- C1 切主题后正在填的表单被重置 / 清空。
- C2 切主题后已选筛选丢失、跳回首页、或当前 tab 被复位。
- C3 切主题通过整页 reload / 重建完成，滚动位置或当前工作流明显错乱。
- C4 其它让主题切换冲掉用户当前进度的情况。

判定（功能测试）：用探查"切换前先布上下文，再切" → 切换后重盘点表单值 / 筛选 / tab / 滚动位置 / 当前工作流是否保留 → 任一被冲掉 = 嫌疑 → 确认这会让用户丢失当前任务上下文 → 定罪。释放：切换为纯样式切换，上下文完整保留。需副作用才能持久切换、只读够不到的，记 `blocker` 缺口。

### D. 主题特定资源只适配一个主题

形式：

- D1 logo / 图标在某主题下消失或与背景同色看不见。
- D2 插图 / 空状态图在某主题下糊成一片或边缘脏。
- D3 透明 PNG / SVG 在某主题下对比不足、显出脏底或噪点。
- D4 其它只为单一背景设计、换主题即破损的视觉资源。

判定：用探查的资源逐主题截图配对 → 对每个 logo / 图标 / 插图 / 空状态图 / 透明图，看是否在某主题下消失、糊、脏边、对比不足 → 是 = 嫌疑 → 截图确认该资源在该主题下用户可见破损 → 定罪。释放：资源在两主题都清晰可辨，或已按主题提供了对应版本。第三方嵌入内容无法完全主题化时，记录用户可见缺口、确认不破坏主要任务即可，不强求完美主题化。

### E. 深色只是"背景变黑"，层级 / 边界 / 可点性垮掉

形式：

- E1 dark 下只是底色翻黑，原本靠浅底分隔的卡片 / 区块层级全平掉。
- E2 边界 / 分隔线 / 投影没跟着主题调整，可操作区与背景糊在一起。
- E3 可点击区域在 dark 下失去可点线索，看不出哪里能点。
- E4 其它深色模式下层级 / 边界 / 可点性 / 状态反馈不再清楚的情况。

判定：用探查的深色重截 → 看层级、边界、可点击区域、状态反馈在深色下是否仍清楚 → 只是背景变黑而这些线索丢失 = 嫌疑 → 截图确认用户在深色下读不出层级 / 分不清边界 / 找不到可点入口 → 定罪。释放：深色经过完整适配，层级 / 边界 / 可点性 / 状态反馈与浅色等价清楚。

## 五、边界 / 非目标

- 不强制每个产品都新增 dark mode；本维度只在产品已暴露 light/dark mode、theme toggle、system theme 跟随或多主题展示时才检查。
- 不把 `theme-color`、manifest、meta、OG 放进本维度。
- 不把"必须用 token / CSS variable / semantic color"写成 MUST；这些是实现手段，不是用户层体验。
- 不要求做一套新的 theme architecture，也不要求大规模重构。
- 不要求所有第三方嵌入内容都完美主题化；但用户可见的破损应记录为缺口（→ D 的释放说明）。
- 不主责单一主题内的即时可操作性 / hover / pressed / disabled / loading 反馈（→ `03`）、相同状态的视觉语义一致性（→ `09`）。
- 不主责 favicon / app icon / logo（→ `01`）、视觉一致性（→ `09`）、动作文案（→ `13`）。
- 数值阈值（如对比度量级）一律作嫌疑信号，不当硬性 MUST / token。
