# 05. 数据状态的可继续性 / Data State Continuity

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**（怎么截图 / 怎么点开状态都在 `00`），只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

**重要（只读约束）**：本维度天然想去触发错误 / 权限 / 刷新失败 / 部分数据等态，但本 skill 只读、禁副作用。所以这些态多数需要副作用或特定后端数据才能到达——这类按 `00` 环节二 2.4 记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`。**缺口是诚实结论，不是放过。** 只读够得到的（GET 来的空 / 无结果 / 错 / 权限态、骨架屏、导航 loading）正常审。

边界：任何状态切换造成的布局跳动归 `04`；即时可操作性 / 操作反馈归 `03`，动作完成后的结果反馈归 `12`；表单内校验 / 提交信心归 `07`；这些数据状态的**文案是否准确清晰**归 `13`。本维度只主责**数据区域在非完整成功态下，状态本身是否存在、是否可被区分、是否给出继续路径**。

## 一、解决什么问题

用户打开一个由数据加载产生的区域（表格 / 列表 / 详情页 / 搜索结果 / 筛选结果 / 资源页）时，能不能**始终知道当前是什么状态、并且还能继续**：是在加载、没有数据、条件没匹配、接口坏了、权限不够，还是页面没做完——这几种必须可区分，不能糊成一片空白或死界面；状态变化后（删除、刷新失败、部分数据返回、分页到边界）也不能把用户留在无解释的空页或清掉他正看的旧内容。本维度只管状态**存不存在、续不续得上**，不管这些状态的文案写得准不准（那归 `13`）。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做（以下凡只读可做的正常到达；凡需副作用或特定数据才能到达的，记「声明了但只读被阻挡」缺口）：

- **列数据区域**：盘点页面上由数据加载产生的区域（表格 / 列表 / 详情 / 搜索结果 / 筛选结果 / 资源面板 / 提交后的结果区）。
- **声明要到达的态**：对每个数据区域声明要到达的态——loading / 空 / 无结果 / 错误 / 权限不足或不可用 / 部分数据 / 刷新失败 / 重试。
- **到达只读够得到的**：骨架屏、导航 loading、初次进入的 loading；以及后端本就 GET 回来的空态 / 无结果态 / 错误态 / 权限态——到达并截图、盘点。
- **只读可做的条件态**：在搜索框 / 筛选器里输入只触发 GET（不写数据）的条件，到达"无结果"态并截图，确认搜索词 / 筛选条件是否保留。
- **够不到的记缺口**：要靠删除 / 提交 / 断网重试 / 制造刷新失败 / 构造部分数据等副作用或特定数据才能触发的态——**不触发**，记「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**分不清当前是什么状态、或在状态变化后无法继续任务**"才定罪；否则按各分类的**释放理由**放过。够不到的状态一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 数据区域无解释空白

形式：

- A1 数据请求期间区域纯空白，看不出是加载中、空、错、权限不足还是页面没做完。
- A2 加载结束落到一片空白，没有任何态来解释为什么空。
- A3 多个候选态（loading / 空 / 错 / 权限）共用同一片空白，无法互相区分。
- A4 其它让用户面对空区域只能猜的情况。

判定：到达该数据区域的非完整成功态（只读够得到就看，够不到则记 `blocker` 缺口）→ 区域是否给出可读的态来说明"现在是什么情况"，且 loading / 空 / 错 / 权限能否被区分 → 一片无差别空白 = 嫌疑 → 截图确认用户分不清这是哪种状态、也无法判断页面是否做完 → 定罪。释放：区域已有可识别的态（骨架 / 空态 / 错态 / 权限态之一）把当前情况说清。

### B. 加载中无"系统仍在工作"感知

形式：

- B1 表格 / 列表 / 详情 / 提交后结果区加载时毫无反馈，看着像卡死或数据为空。
- B2 用一个孤立全局 spinner 替代所有加载，看不出在加载哪一块。
- B3 局部刷新 / 分页 / 筛选触发的加载没有任何处理中感知。
- B4 其它让用户误以为页面卡死或为空的加载。

判定：到达 loading 态（骨架屏 / 导航 loading 等多为只读可达，正常审；需副作用触发的加载够不到则记 `blocker` 缺口）→ 用户能否看出系统仍在工作、且看得出受影响的是哪一块 → 无感知或定位不到 = 嫌疑 → 截图确认会被误判为卡死或空数据 → 定罪。释放：有可感知的处理中线索（骨架 / 行内 loading / 区域占位），哪怕不是复杂 skeleton——本维度不要求复杂骨架，用户知道在工作即可。

### C. 空状态不解释、无下一步

形式：

- C1 空状态只显示 `No data` / `Empty` / `Nothing here`，不说明为什么空。
- C2 首次使用 / 还没创建资源 / 数据被清空，都落到同一个无差别空提示。
- C3 有自然下一步（创建 / 导入 / 配置 / 返回）却不给入口。
- C4 其它让用户停在"空了但不知道为什么、也不知道下一步"的情况。

判定：到达空态（只读够得到就看，够不到则记 `blocker` 缺口）→ 是否说明了"为什么空"，且在有自然下一步时给了入口 → 只有 `No data` 之类、无解释无入口 = 嫌疑 → 截图确认用户既不懂为何空、又找不到下一步 → 定罪。释放：空态说明了原因，且要么本无自然下一步、要么入口已给。注意：空态有没有配插画不在判定内——插画不是高置信要求，也不能替代解释和下一步。

### D. 无结果与真空状态不分、丢失条件

形式：

- D1 搜索 / 筛选无结果时显示成普通空状态，让用户以为产品里根本没数据。
- D2 无结果时清掉了搜索词 / 筛选条件，用户看不出是"当前条件没匹配"。
- D3 无结果态不给清除 / 调整条件 / 返回全部结果的路径。
- D4 其它让"没匹配"被误读成"没数据"的情况。

判定：在搜索 / 筛选里输入只触发 GET 的条件到达无结果态（只读可达正常审；需写数据才能构造的够不到则记 `blocker` 缺口）→ 是否和真空状态区分、是否保留搜索词 / 筛选条件、是否给清除或调整条件的路径 → 任一缺失 = 嫌疑 → 截图确认用户会误判成"产品里没有任何数据" → 定罪。释放：无结果态与空态可区分，条件保留，且有调整或返回路径。

### E. 错误态不告知能否恢复

形式：

- E1 只抛一句红色错误文案，区域内容消失，看不出能否重试 / 返回。
- E2 出错后直接把区域清空，没有任何恢复入口。
- E3 错误态没有重试 / 返回 / 检查权限 / 求助等任一合理下一步。
- E4 其它让用户出错后卡死、不知能否继续的情况。

判定：到达错误态（GET 本就会返回的错态多为只读可达，正常审；要靠断网 / 制造失败等副作用才能触发的够不到则记 `blocker` 缺口）→ 用户能否判断是否可恢复、是否看到合理下一步（重试 / 返回 / 检查权限 / 保留旧内容 / 场景内求助）→ 只剩孤立红字或区域消失、无恢复线索 = 嫌疑 → 截图确认用户出错后无法继续 → 定罪。释放：错误态让用户判断得出能否恢复并给了合理下一步——本维度不要求解释技术原因、也不强求"联系管理员"，那取决于产品场景。

### F. 权限 / 不可用伪装成错误或空数据

形式：

- F1 权限不足显示成普通空数据，用户去找根本不存在的"数据问题"。
- F2 功能不可用 / 前置条件未满足被呈现成通用错误。
- F3 不可用态看不出是"没权限 / 功能未开 / 条件未满足"，只剩空白或红字。
- F4 其它把"不该有数据"误说成"数据不存在 / 系统出错"的情况。

判定：到达权限不足 / 不可用 / 条件未满足态（后端 GET 就返回的权限态多为只读可达，正常审；需特定账号 / 副作用才能构造的够不到则记 `blocker` 缺口）→ 是否明确呈现为"没有权限 / 功能不可用 / 条件未满足"，而非空数据或通用错误 → 被伪装成空或错 = 嫌疑 → 截图确认用户被误导去排查不存在的数据问题 → 定罪。释放：不可用态如实表达为权限 / 可用性 / 前置条件，未冒充空或错。

### G. 状态变化后无法继续

形式：

- G1 分页最后一页删除后落到一个无解释的空页，看不出是成功、页码失效还是数据丢失。
- G2 刷新失败时清掉用户刚看的旧数据，只剩一个错误。
- G3 局部加载失败 / 部分数据返回时直接清空已加载内容，打断正在进行的阅读或操作。
- G4 其它在状态变化后把用户留在无解释空白或死界面的情况。

判定：这些续接态多需删除 / 刷新失败 / 部分数据等**副作用或特定数据**才能到达——只读够不到的按 `00` 环节二 2.4 记「声明了但只读被阻挡」缺口，输出 `blocker`，不猜测、不放过；只读够得到的（如 GET 回来本就部分缺失的数据区）正常审 → 状态变化后用户能否继续任务、旧内容是否被无谓清掉、是否落到无解释空页 → 是 = 嫌疑 → 截图确认用户被打断或迷失 → 定罪。释放：变化后仍有可继续路径，旧内容在合理时保留，未落到无解释死界面。

## 五、边界 / 非目标

- 这些数据状态的**文案是否准确**（是否说清对象 / 数量 / 影响 / 是否可撤销、用词是否清晰）归 `13`；本维度只主责状态**存在、可区分、有续接**。
- 任何状态切换造成的**布局跳动 / 内容位移**归 `04`。
- 操作前的即时可操作性 / 操作反馈归 `03`；**动作完成后**的结果确认、处理中反馈、撤销路径、动效分寸归 `12`。
- 表单内的特定输入 / 校验 / 内容保留 / 提交信心归 `07`。
- 不要求所有数据区域都有复杂 skeleton；用户知道系统在工作即可，不把 skeleton 当硬性 MUST。
- 不把具体状态管理方案（前端 store / query library / cache 策略 / 状态机）写成 MUST，它们只是实现手段。
- 不要求每个错误都解释技术原因；用户需要知道影响、恢复可能性和下一步。
- 不要求所有产品都提供"联系管理员"，这取决于产品场景和用户关系。
- 不把"接口必须怎么设计"放进本维度。
- 不要求空状态都配插画；插画不是最高置信要求，也不能替代解释和下一步。
- 不把内部数据建模、组件抽象、异常分类本身当成目标；只检查用户能看到、感到、被影响的状态连续性。
