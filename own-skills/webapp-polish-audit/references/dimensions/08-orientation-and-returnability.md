# 08. 用户的位置感与返回感 / Orientation and Returnability

这是一条**审查维度**，按操作手册使用：先由 `00-inspection-procedure.md` 把页面探寻成证据包，再用本文件的形式目录逐条比对、判定。本文件**不写探寻机制**（怎么截图 / 怎么点开状态都在 `00`），只写：① 解决什么问题；② 向引擎申请的额外探查；③ 形式目录 + 判定。

边界：表格 / 列表内的筛选范围、排序范围、分页范围、批量动作作用对象归 `06`；空数据 / 无结果 / 错误 / 权限不足状态本身如何解释归 `05`，这些状态的文案清晰度归 `13`；任何状态导致的布局移动归 `04`；窄视口重排挤坏归 `11`。本维度只主责**导航范围与返回路径**——当前位置标识、上下文范围变化、列表到详情的返回、深层 / 多步流程的退出，以及破坏性离开时对未完成工作的保护。

## 一、解决什么问题

用户能否始终知道**自己在哪、在什么范围里、怎么回去、离开会不会丢上下文或未完成的工作**。具体到六件事：能不能看出当前页面 / tab / section / 设置分类 / 详情对象；能不能把"当前位置"和"可点击入口 / 不可用入口"分开；从列表进详情再返回时关键上下文还在不在；modal / drawer / wizard / 设置子页 / 嵌套详情有没有可理解的退出与返回路径；切 tab / section / workspace / 账号 / 环境时上下文变化能不能被感知（而不是被读成数据异常）；编辑到一半离开时未保存的输入会不会被静默丢弃。

## 二、向探寻引擎申请的额外探查（→ `00` 环节二）

引擎在池化执行时，为本维度额外做（以下都不产生副作用，只读可做；需要副作用才能到达的状态记成「声明了但只读被阻挡」缺口，挂本维度名下，输出 `blocker`）：

- **盘点位置标识**：枚举 nav / tab / 侧边栏 / 面包屑 / section 标题 / 对象标题，记录当前项的状态表达（选中 / 当前）与可点击入口、不可用入口、未选入口的样式向量，供横向比对它们能不能被分清。
- **列表到详情走查**：从带筛选 / 搜索 / 排序 / 分页 / 滚动位置 / 选中上下文的列表点进一条详情，再走返回路径，记录返回后这些上下文是否还在。
- **进入深层面并找出口**：进入静态可达的 modal / drawer / wizard / 设置子页 / 嵌套详情 / empty detail state，盘点其中有没有清楚的退出、返回上层、回到主任务入口（不只是关闭图标或"下一步"）。
- **范围切换重截**：切 tab / section / 以及只读可达的 workspace / 账号 / 环境，盘点切换后有没有明显的上下文变化提示，截图比对前后。
- **未保存离开（多数需副作用）**：在有未保存输入 / 临时选择 / 多步进度时尝试离开 / 关闭 / 返回 / 切步——离开拦截提示往往要真去编辑并触发离开才出现，属副作用，**不触发，记 blocker 缺口**；只读能确认的（如已有的"未保存"标记、leave-guard 文案）正常审。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**不知道自己在哪 / 分不清当前位置 / 返回后丢上下文 / 被困住出不来 / 在错误范围里操作 / 静默丢失未完成工作**"才定罪；否则按各分类的**释放理由**放过。够不到的状态（典型是未保存离开拦截）一律 `blocker` 缺口，不猜测、不放过。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

### A. 当前位置看不出来

形式：

- A1 多 tab / 多侧栏 / 多层设置里，当前项没有明显的选中 / 当前态，用户不知道自己在哪一项。
- A2 详情页 / dashboard 子页 / 设置子页没有页面标题、对象标题或路径提示，进来后看不出落在哪。
- A3 当前 section 在内容区切换了，但导航 / 标题区毫无对应标识。
- A4 其它让用户要靠猜才能确定当前页面 / tab / section / 设置分类 / 详情对象的情况。

判定：用探查盘点的位置标识 → 当前项有没有可识别的选中 / 当前态、当前页面 / 对象有没有标题或路径线索 → 全无 = 嫌疑 → 截图确认用户确实要靠猜当前位置 → 定罪。释放：顶部导航、tab、分组标题、对象标题、路径提示、上下文标签等任一手段已让当前位置可读（不强制 breadcrumbs / sidebar）。

### B. 当前位置与可点击入口 / 不可用入口混淆

形式：

- B1 当前 nav / tab / section 做得和普通可点入口一样，看起来还能再点进去。
- B2 未选入口做得像 disabled，用户以为去不了。
- B3 当前态、可点态、禁用态、未选态四者视觉上分不开。
- B4 其它让用户分不清"我在这"和"我能去 / 不能去"的情况。

判定：对照盘点的当前项与各入口样式向量 → 当前态是否与普通可点项区分、未选态是否与 disabled 区分 → 任一对混淆 = 嫌疑 → 截图确认用户会迷路或误判某入口可用性 → 定罪。释放：四态之间有清楚可辨的区分线索即可，不规定导航组件长什么样。

### C. 列表到详情返回丢上下文

形式：

- C1 从筛选 / 搜索后的列表进详情，返回时筛选 / 搜索条件被重置。
- C2 返回后排序、分页跳回第一页 / 默认序，用户要重新翻找。
- C3 返回后滚动位置、选中对象丢失，每看一个详情就得从头找。
- C4 其它让用户返回列表后需要重建关键上下文的情况。

判定：用探查的列表到详情走查 → 返回后筛选 / 搜索 / 排序 / 分页 / 滚动位置 / 选中上下文是否还在 → 关键上下文丢失 = 嫌疑 → 截图确认用户需要重新构建刚才的位置 → 定罪。释放：关键上下文得到保留即可；不要求精确恢复滚动位置（那只是常见手段之一）。

### D. 深层 / 多步流程困住用户

形式：

- D1 modal / drawer 里只有关闭图标，没有清楚的返回主任务或退出说明。
- D2 wizard / 多步表单只有"下一步"，没有返回上层或退出整个流程的路径。
- D3 设置子页 / 嵌套详情 / empty detail state 进得去，却看不出怎么回上一层或回主任务。
- D4 其它让用户被困在深层界面、不知道怎么回去的情况。

判定：用探查进入的深层面 → 有没有可理解的退出 / 返回上层 / 回主任务入口（不只是关闭图标或"下一步"）→ 没有 = 嫌疑 → 截图确认用户会被困住、找不到回主任务的路 → 定罪。释放：存在任一清楚的返回 / 退出路径即可，形式不限。

### E. 导航变化伪装成数据变化

形式：

- E1 切 tab / section 时只替换内容区数据，导航 / 标题区无对应上下文提示。
- E2 切 workspace / 账号 / 环境时界面无明显范围变化提示，用户以为还在原范围。
- E3 范围切换被表现成加载失败 / 空数据 / 普通刷新，用户分不清是切了范围、没数据、没权限还是在加载。
- E4 其它让用户感知不到上下文已变、可能在错误范围里操作的情况。

判定：用探查的范围切换重截 → 切换前后有没有明显的上下文变化提示（当前项标识 / 范围标签 / 标题）、范围变化会不会被读成数据异常 → 感知不到变化 = 嫌疑 → 截图确认用户可能在错误 workspace / 账号 / 环境里操作 → 定罪。释放：上下文变化有明显可感知的提示，且不与空 / 错 / 加载态混淆。（空 / 错 / 权限态本身如何解释归 `05`，其文案归 `13`。）

### F. 破坏性离开不保护未完成工作

形式：

- F1 编辑到一半允许关闭 / 返回 / 切 tab / 切步骤，静默丢掉未保存输入。
- F2 多步进度中途离开，已填内容与进度无提示地清空。
- F3 临时选择 / 筛选结果 / 用户刚建立的判断上下文在离开时无提醒地丢失。
- F4 其它让破坏性离开不提醒、不保留就发生的情况。

判定：用探查的未保存离开走查 → 离开 / 关闭 / 返回 / 切步时有没有提醒，或内容有没有被保留 → **拦截提示多数需真编辑并触发离开才出现，属副作用 → 记 `blocker` 缺口，不猜测、不放过**；只读能确认到的（已有"未保存"标记、leave-guard 文案缺失）→ 无提醒且无保留 = 嫌疑 → 确认用户会静默丢失未完成工作 → 定罪。释放：离开前有明确提醒，或内容被保留。

## 五、边界 / 非目标

- 不要求所有页面都有 breadcrumbs；breadcrumbs 是可选手段，不是目标本身。
- 不要求所有产品都有 sidebar；顶部导航、tab、分组标题、对象标题、路径提示、返回入口、上下文标签都可能满足位置感。
- 不规定导航组件长什么样；当前态、可点击入口、返回路径、上下文变化能被用户理解才是目标。
- 不要求所有返回都精确恢复滚动位置；MUST 是不丢关键上下文，滚动恢复只是常见手段之一。
- 不要求营销单页也有复杂当前位置标识；没有真实层级移动或任务上下文时，不强行添加产品型导航结构。
- 不把路由实现、URL 结构、active state token、router 配置、缓存策略、浏览器 history 管理、React state 管理写成 MUST；只检查用户能看到、感到、被影响的位置感与返回感。
- 不主责表格 / 列表内的筛选范围、排序范围、分页范围、批量动作作用对象（→ `06`）。
- 不主责空数据 / 无结果 / 错误 / 权限不足状态本身如何解释（→ `05`），这些状态的文案清晰度（→ `13`）。
- 不主责任何状态导致的布局移动（→ `04`）、窄视口重排挤坏（→ `11`）。
