# 07. 表单填写的低摩擦体验 / Form Completion Confidence

这是一条**审查维度**：证据由 `../pipeline/16-page-exploration-and-capture.md` 统一采集，本文件只写判定标准——① 解决什么问题；② 向采集环节申请的额外探查；③ 形式目录 + 判定。

边界：数据状态与字段 / 按钮 / 错误文案是否准确表达对象 / 数量 / 影响归 `13`；操作前的可操作性 / disabled / pressed / 危险二次确认归 `03`；键盘 / 触摸 / 缩放 / 对比度 / 仅悬停可达归 `10`；窄视口重排挤坏归 `11`。普通数据区域的 loading / 空 / 无结果 / 错误 / 权限态本身、动作触发后的全局结果反馈、同类元素的视觉一致性，均不在主流程五维度覆盖范围内。本维度只主责**表单这个任务表面**：登录 / 注册 / 设置 / 创建·编辑 / 筛选 / 结账 / 发布 / 邀请 / 权限 / 批量操作表单的输入、校验、保留与提交信心。

## 一、解决什么问题

用户能不能**有信心地填完并提交一个表单**：理解每个关键输入在要什么、在高影响字段提交前理解后果、犯错后知道哪里错怎么改、看得出提交是在处理 / 成功 / 失败、已经填好的内容不被无故清空、必填 / 可选 / 不可编辑 / 等待中这些状态分得清——而不是靠猜字段含义、看不懂错误、重复点提交、失败后重填，或分不清字段是坏了、没加载完、禁用还是不用填。

## 二、向探寻引擎申请的额外探查（→ `16` 第 2 步）

引擎在池化执行时，为本维度额外做（以下凡只读可达的全部做；凡需提交 / 保存 / 写数据等副作用才能到达的状态，**记成 `pending_authorization` 待授权路径**（见 SKILL.md §0），写清发现了什么、打算怎么验、不验的代价、怎么授权——**这不是遗漏，是等用户拍板**）：

- **盘点字段契约**：枚举表单内每个输入的标签 / 说明 / placeholder / 必填标识 / 单位·格式·范围提示，记录字段是否只靠 placeholder 承载含义、有歧义 / 风险 / 后果的字段有没有额外说明。
- **盘点字段状态表达**：盘点必填 / 可选 / 不可编辑 / 只读 / 禁用 / 等待中 / 权限不足各状态在视觉上长什么样，记录它们是否互相可分辨。
- **触发只读可达的前端校验态**：对只读可达的纯前端校验（漏填必填、错误格式、超范围、边界值、超长文本），用边界值 / 长文本 / 错误格式触发，记录错误表达（DOM + 文案 + 样式向量），看是否指向具体字段并说明为什么错 / 怎么改。
- **声明需要的提交 / 保存态**：对每个提交 / 保存 / 失败重试动作声明要到达的态——提交中 / 成功 / 失败·超时 / 校验失败后已填内容是否保留 / 高影响提交前的确认步骤。
- **需授权的逐条报上来**：真实提交 / 保存的成败态、网络失败后的保留态、写库后的结果，都需要写数据才能触发。**不执行，但必须逐条记 `pending_authorization`**：这条路存在、我知道怎么验、按边界停下了，请你决定。绝不因为没查就当 `already_satisfied`，也绝不降级成 `blocker` 一笔带过。

## 三、判定的两道闸（贯穿所有分类）

1. **形式目录挂嫌疑（高查全）**：宁可错抓，命中任一形式 = 嫌疑。
2. **判定定罪（高查准）**：只有"会让用户**填不下去、提交不放心、犯错后修不回、或提交了才发现后果**"才定罪；否则按各分类的**释放理由**放过。需写数据才能到达的提交 / 保存态一律记 `pending_authorization` 报给用户，不猜测、不放过、也不当已满足。

本维度不做样式聚类，没有显眼 / 细微轴；每个分类的嫌疑信号与定罪测试各自写在判定里。

## 四、形式目录 + 判定

> **本维度的判定读 JSON，不看图。** 证据是采集环节落盘的 DOM 结构、可见文本、样式向量、几何与状态属性。判断者不开浏览器——判不了就输出 `blocker` 并写清缺什么，由主代理决定重采还是动用 SKILL.md §0 的「受限截图」例外。**判断者自己去页面上看一眼是越权。**

### A. 字段含义不可懂

形式：

- A1 关键字段只靠 placeholder 承载含义，输入后说明消失，用户无法确认字段在要什么。
- A2 字段没有任何标签 / 说明，含义要靠用户猜。
- A3 字段名含糊（如 `值`、`类型`、`配置`），看不出该填什么。
- A4 其它让用户必须猜才能填的字段含义问题。

判定：用探查盘点的字段契约 → 关键字段是否只靠 placeholder / 是否无可见标签 → 是 = 嫌疑 → 读证据 JSON 确认用户不依赖 placeholder 就读不懂字段含义 → 定罪。释放：含义本就通用、上下文已足够明确（如登录页的"邮箱""密码"），或字段已有持久可见的标签 / 说明。label 用顶部 / 左侧 / 分组 / inline 哪种布局不限，只看用户能不能理解含义。

**证据与判定模式：`脚本+文本判`**。脚本对每个输入采 `{ label_source, label_text, aria_label, placeholder, adjacent_help_text, field_name }`，其中 `label_source` 区分 `for` 关联 / `aria-labelledby` / 包裹式 / 无。LLM 读 JSON 判"遮住 placeholder 还读不读得懂这个字段要什么"——语义判断，不需要截图。走 SKILL §4 的双判断者并行决胜。

### B. 有歧义 / 风险 / 后果的字段缺上下文

形式：

- B1 格式 / 单位 / 范围有要求的字段（金额、时长、百分比、日期）不给提示，用户填错才知道。
- B2 会公开 / 发送 / 覆盖 / 影响他人的字段（可见范围、通知对象、权限）不说明后果。
- B3 必填性有歧义的字段不标必填 / 可选，用户不确定是否要填。
- B4 其它让用户提交后才发现单位理解错、范围不合法、或填写内容会被公开 / 发送 / 覆盖的缺上下文问题。

判定：用探查盘点的字段契约 → 有歧义 / 格式 / 风险 / 后果的字段是否提供足够上下文 → 缺失 = 嫌疑 → 读证据 JSON 确认用户会在提交后才发现单位 / 范围 / 影响理解错 → 定罪。释放：含义通用且不易误填的字段无需额外说明；不要求每个字段都有 help text，只有有歧义 / 格式 / 风险 / 后果或易误填时才需要。

**证据与判定模式：`脚本+文本判`**。脚本按字段采 `{ input_type, unit_or_format_hint, required_marker, help_text, scope_impact_text, pattern, min, max }`。LLM 读 JSON 判"这个字段的歧义 / 格式 / 后果有没有被说清"——语义判断，不需要截图。走双判断者并行决胜。

### C. 错误不可定位 / 不可修正

形式：

- C1 错误只把边框变红，不说哪里错、为什么错。
- C2 错误只显示 `Invalid` / `Required` 等通用词，用户不知道具体该改什么。
- C3 错误用页面级通用 toast，不指向具体字段，多字段出错时无法定位。
- C4 其它让用户知道"错了"却不知道"哪个字段错、怎么改"的错误表达。

判定：用探查触发的只读可达校验态 → 错误是否指向具体字段、是否说明为什么错 / 怎么改（需提交才能触发的后端校验态记 `pending_authorization`）→ 只变红 / 只通用词 / 不指向字段 = 嫌疑 → 读证据 JSON 确认用户无法定位并修正 → 定罪。释放：错误已指向具体字段并给出可修正的原因 / 修法。

**证据与判定模式：`脚本+文本判`**。触发只读可达校验后，脚本采 `{ aria_invalid, aria_describedby_target, error_text, error_dom_distance_to_field, is_page_level_toast, field_border_color_delta }`。"只变红"由 `error_text` 为空 + `border_color_delta` 非零确定性识别；错误文案够不够可修正由 LLM 读文本判。不需要截图。走双判断者并行决胜。

### D. 提交状态不清楚（提交中 / 成功 / 失败）

形式：

- D1 提交后按钮无 loading / busy 态，用户重复点提交。
- D2 提交成功后无可感知反馈（无跳转 / 无 inline / 无状态变化），用户不知道是否保存成功。
- D3 提交失败 / 超时静默，界面停在原样或成功样式，用户以为成功了。
- D4 其它让用户靠猜"动作是否被接收、是否完成、失败后还能做什么"的提交态问题。

判定：到达提交 / 保存的处理中 / 成功 / 失败态（只读够得到的看，需写数据才能到达的记 `pending_authorization`）→ 三态是否被清楚区分、提交时是否防住重复点 → 缺失 = 嫌疑 → 读证据 JSON 确认用户只能靠猜或会重复提交 → 定罪。释放：状态已由按钮态 / 跳转 / inline / 保存时间 / 列表变化等清楚表达；成功反馈不要求一定 toast。

**证据与判定模式：`脚本+文本判`**。脚本采提交控件的 `{ disabled, aria_busy, aria_live_region_text, label_text_before_after, form_dom_diff_summary }`。防重复点由 `disabled` / `aria_busy` 确定性判定；三态可区分性由 LLM 读文案与 DOM 比对判。不需要截图。**需真实提交才能到达的成 / 败态一律记 `pending_authorization`，不模拟、不猜测**——附上「授权后可在 staging 验证提交中/成功/失败三态是否可区分」。 走双判断者并行决胜。

### E. 非必要清空已输入内容

形式：

- E1 校验失败后整个表单被重置，用户被迫重填。
- E2 网络失败 / 提交失败后已填内容丢失。
- E3 局部刷新某区域或切换非破坏性选项（tab / 折叠 / 排序）时清空别处已填内容。
- E4 其它在非必要时迫使用户重填的清空问题。

判定：触发校验失败 / 局部刷新 / 非破坏性选项切换（只读够得到的看，网络 / 提交失败的保留态需写数据→记 `pending_authorization`）→ 已输入内容是否被非必要清空 → 是 = 嫌疑 → 读证据 JSON 确认用户被迫重填 → 定罪。释放：清空是用户主动 / 破坏性操作的预期结果。保留已填是目标，autosave 只是其中一种方式，不要求所有表单都做。

**证据与判定模式：`脚本+文本判`**。脚本在触发前后各采一次全字段 `value` 快照，输出 `{ trigger_action, cleared_fields: [...], preserved_fields: [...] }`。清空事实由 value 比对 **确定性**得出；"是否必要"（触发动作是不是破坏性 / 用户主动）需 LLM 判。不需要截图。走双判断者并行决胜。

### F. 高影响提交前无后果确认

形式：

- F1 改权限 / 发布 / 转账 / 删除关联 / 批量更新 / 发送通知等高影响表单直接提交，无 review / 确认 / 后果说明。
- F2 高影响提交不展示对象范围（改了谁、发给谁、删了哪些关联）。
- F3 批量操作不说明命中数量或影响面就提交。
- F4 其它让用户事后才发现影响了权限 / 发布 / 通知 / 批量对象 / 关联数据的确认缺口。

判定：盘点高影响表单 → 提交前是否能看见关键后果 / 对象范围 / 确认步骤（提交后的真实结果需写数据→记 `pending_authorization`）→ 直接提交且无后果可见 = 嫌疑 → 读证据 JSON 确认用户提交前看不到关键后果 → 定罪。释放：已有 review / 确认 / 后果说明 / 明确对象范围。这与危险操作保护一致，但表单里尤其常见；操作前的危险二次确认机制本身归 `03`，本类只主责**表单提交前后果是否可见**。

**证据与判定模式：`脚本+文本判`**。脚本采高影响表单的 `{ submit_label, pre_submit_visible_text, has_review_step, has_confirm_dialog, scope_object_text, batch_count_text }`。确认步骤存在性确定性可判；后果说明够不够由 LLM 读文本判。不需要截图。走双判断者并行决胜。

### G. 字段状态不可区分（必填 / 可选 / 不可编辑 / 等待中）

形式：

- G1 disabled / read-only 表现相同，用户分不清是禁用还是只看。
- G2 等待中（字段还在加载选项 / 值）和禁用混在一起，用户以为字段坏了。
- G3 权限不足的字段表现得像普通禁用，用户不知道是自己没权限。
- G4 其它让用户误判字段是坏了 / 没加载完 / 必须填 / 自己没权限的状态混淆。

判定：用探查盘点的字段状态表达 → 必填 / 可选 / 不可编辑 / 只读 / 等待中 / 权限不足是否互相可分辨 → 多状态表现混同 = 嫌疑 → 读证据 JSON 确认用户会误判字段状态含义 → 定罪。释放：各状态有清晰可分辨的线索（标识 / 文案 / 视觉差异）。注意：状态只靠颜色 / 低对比导致不可辨归 `10`，本类只主责"几类状态被做成同一种表现、含义混同"。

**证据与判定模式：`脚本+文本判`**。脚本按状态分组采 `{ state, attrs: {disabled, readonly, aria_disabled, required, aria_busy}, style_vector, accompanying_text }`，并输出各状态两两之间的 `style_vector` 是否相同。**表现混同由样式向量比对确定性得出**；"用户会不会误判含义"由 LLM 读属性与文案判。不需要截图——纯对比度 / 只靠颜色的不可辨归 `10`（脚本直判）。走双判断者并行决胜。

## 五、边界 / 非目标

- 不规定必须用某种 label 布局：顶部 / 左侧 / 分组 / inline label 都只是实现选择，用户理解字段含义才是目标。
- 不规定所有字段都要有 help text；只有有歧义 / 格式 / 风险 / 后果或易误填时才需要额外说明。
- 不要求所有表单都做 autosave；保留已填内容是目标，autosave 只是其中一种方式。
- 不要求所有成功都 toast；成功反馈可以是页面状态 / 跳转 / inline / 保存时间 / 结果页等任何明确形式。
- 不把具体表单库、schema validation、ARIA 实现、字段配置结构、校验代码组织、后端接口设计写成必须；它们是实现手段，不是本维度的用户可见结果。
- 不把"更漂亮 / 更现代"列成目标；必须落到用户能否**理解输入、修正错误、确认提交、保留已填、看懂字段状态、提交前理解后果**。
- 不主责数据状态与字段 / 错误文案是否准确表达对象 / 数量 / 影响（→ `13`）、操作前可操作性 / disabled / pressed / 危险二次确认（→ `03`）、键盘 / 触摸 / 缩放 / 对比度 / 仅悬停可达（→ `10`）、窄视口重排（→ `11`）。
- 普通数据区域的空 / 错 / 加载 / 权限态本身、动作触发后的全局结果反馈、视觉一致性，均不在主流程五维度覆盖范围内。
