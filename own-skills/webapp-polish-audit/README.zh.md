# Webapp 打磨审查

只读审查浏览器渲染的用户可见 Web UI, 判断者每次只面对一张截图 + 一小份筛选后的维度文档, 主 Agent 只调度绝不亲自看图, 一图一 TODO 一判断者。

## 什么时候用它

**单页品味**: 我丢一个页面 URL, 想让 skill 抓抓像 demo 不像正式产品的地方——留白、字号、状态覆盖、边界情况、复制感。

**多页面从种子**: 我丢一个入口 URL, 想让 skill 先跑子页面树, 再对每类代表页面各自跑一遍打磨审, 最后合成跨页面报告。

**响应式对比**: 桌面看着还行, 手机/平板塌了, 想让 skill 走视口对比组 (11 视口对) 揪对不齐的地方。

**状态审**: 表单/弹窗/流程里空、加载、错、成功、禁用等状态需要一遍过, 想让 skill 用状态前后对 (04 状态对) 逐态截、逐态判。

**真产品感**: 我明确担心页面读起来像 demo, 想让 skill 走「not demo-like / real product / completion quality / 完成度 / 好品味」这类打磨触发词做整体品味审。

**发布前打磨**: 新页面即将上线或刚上线, 想在发公告前跑一遍完成度体检, 抓可见 bug 与打磨缺口。

**不接**: 纯后端 / CLI / 数据库 / 仅 API 任务; 无 Web UI 的产品策略 / 用户研究 / 信息架构 / 仅文案; 大范围重新设计 (只审拟议/当前设计); 代码级 polish 或需要产出实现 diff 的编码任务。

## 它会产出什么

**只审不改, 只报证据, 从不动项目文件——这是最严的锁定**:

- **四阶段管线**: 阶段 1 子页面树发现 (仅多页); 阶段 2 每页选维度 + 证据; 阶段 3 探索 + 截图落盘; 阶段 4 逐张截图判断 + 页面级 `class_coverage` 聚合; 单页任务跳过阶段 1, 其余照走。
- **运行产物只落 /tmp**: 所有截图 PNG / 盘点 JSON / manifest / raw-urls.txt / 进度文件写到 `/tmp/webapp-polish-audit/{YYYYMMDD-HHMMSS}-{run-name}/`, 项目目录里不留一个字节。
- **判断者绝不进主会话**: 主 Agent 只调度, 每张截图派一个只读判断 SubAgent, 先无规则新鲜眼一遍再按筛选后维度对照, 目录罩不住的进 `uncatalogued` 不丢弃。
- **多图比对走对比组**: 主题对 (02) / 视口对 (11) / 状态前后对 (04), 单图判不了的绝不硬判。
- **缺口诚实**: 没截到、没翻到、被副作用挡住的都进 `gaps`, 绝不静默当全清; 最终报告是发现 + gaps 双份。
- **绝不会做**: 读项目源码 / 改文件 / 提交表单 / 触发外部副作用 / 产出实现 diff / 做像素级重新设计。

## 前置条件 / 边界

**前置**: 目标是浏览器可访问的渲染 Web UI (公网或本地开发服务器); 运行时需暴露 Browser 工具与 SubAgent / Task 机制 (缺 SubAgent 时按 `local sequential fallback` 走, 但仍先落盘再判断); `/tmp/webapp-polish-audit/` 可写。

**不接的场景**: 代码级审查 (走 code-review); 纯 API / 数据 / 后端 / CLI / 基础设施任务; 大范围重新设计 (只审已有/拟议设计); 需要产出实现 diff 的任务。

**微妙边界**:

- 用户说 "optimize / fix / improve / polish / make it real" → 一律解释为审查任务, 只输出发现和推荐修复, 绝不实施。
- 浏览器交互只允许检查, 不许提交表单 / 创建/更新/删除数据 / 变更权限 / 上传文件 / 触发外部副作用。
- 单页任务也必须走完整管线 (跳阶段 1), 判断绝不留在主会话。
