## 组件原则

- 组件必须保持可预测、克制，并贴近真实产品语境；视觉表达应服务任务完成，而不是装饰性炫技。

- 数据模块、功能入口、表单分组、设置项、流程步骤、知识库 / 文档条目必须有清晰承载。页面背景上使用卡片或容器；已成型的大面板内使用分组标题、列表行、浅表面块或留白承载。

- 默认避免卡片套卡片。仅在对话框、独立重复项、复杂表单分组或设计明确需要嵌套结构时，才允许增加一层嵌套承载。

- 组件默认不依赖外描边或重阴影建立层级。层级优先通过表面色、间距、分组、状态填充、内容结构和 focus outline 表达；外描边、轻阴影仅在分隔、可用性、浮层、交互状态或明确层级强调时使用。

- 只补齐与组件功能相关的状态：交互控件通常覆盖 default、hover、active、focus-visible、disabled，以及适用的 selected / current、expanded / open、checked、read-only；异步或数据组件按需覆盖 loading、empty、error、warning、success、info。静态展示组件不得机械添加无意义状态。

## 承载模型

常规页面最多使用三层视觉承载：

`page background -> section / panel -> content item`

该规则限制的是视觉承载层级，不限制必要的 DOM 结构、语义分组或组件内部实现。

### 规则

- 独立卡片必须具备独立表面、圆角、内边距、清晰标题、正文 / 元信息和状态区域；如存在可操作任务，应提供明确操作区域。

- 在已有面板内部，优先使用分组标题、列表行、浅色表面块、分隔留白或状态填充承载内容，而不是继续嵌套一张完整卡片。

- 目标岗位、候选人资料、知识库、文档、资源、工作流步骤、设置项、表单分组等模块必须被视觉分组；分组不等于新增卡片。

- 仅在对话框、独立重复项、复杂表单分组、嵌套数据结构或明确设计需求下，允许增加一层嵌套承载。

## 信息型内容卡片

该模型适用于信息型内容卡片，用于承载资源、Skill、文档、模板、工具、工作流步骤、知识库条目、候选资料摘要等结构相近的内容。资源卡和 Skill 卡是该模型的典型实例。

该模型基于 Figma 节点 `1350:6267` 的历史视觉参考；执行时应优先遵循当前 Token、承载模型和响应式规则。

### 容器与组件专属几何配方

- 表面：`Card Surface`。

- 圆角：`--radius-lg`（`16px`）。

- 内边距：`--space-card-inset`（默认 `16px`）。

- 默认外描边：无

- 默认阴影：无

- 内容区域参考：约 `388 x 204px`

- 原始视觉参考卡片：约 `420 x 226px`

- Hover：可选 `--shadow-hover-strong`。

### 内部结构

- 结构：纵向堆叠。

- 头部与正文间距：`--space-3`（`12px`）。

- 正文与底部元信息间距：`--space-6`（`24px`）。

- 头部：左侧 `48 x 48px` 功能符号容器（组件专属几何），右侧标题和标签组，间距 `--space-3`。

- 标题：`Heading S`，`Text Primary`，优先单行显示。

- 标签：高度 `20px`（组件专属几何），圆角 `--radius-sm`，水平内边距 `--space-2`，文字使用 `Label`。

- 主标签：`Brand Red` 文字与 `Brand Red Soft` 表面。

- 次标签：`Text Tertiary` 文字与 `Page Background` 表面。

- 描述：`Body L` 与 `Text Primary`，建议约两行。

- 元信息：`Supporting Text` 与 `Text Tertiary`，横向组间距 `--space-3`。

- 元信息图标：`--icon-size-sm`，图标与文字间距 `--space-1`。

- 下载 / 操作热区：`32 x 32px`（组件专属几何），内含 `--icon-size-lg` 图标，圆角 `--radius-sm`，默认不加外框。

- 内部 utility line 只允许作为独立的低对比分隔线；不要发展成卡片描边系统。

## 产品展示面板

产品展示面板用于承载产品能力说明、真实界面截图、工作流演示或 UI-like product visuals。它更接近大型产品面板，而不是传统信息卡片，不应套用资源卡 / Skill 卡的小卡片结构。

### 容器

- 使用大面积浅中性容器作为承载层，优先使用白色、浅灰或低对比中性表面。

- 面板内部通常包含信息区和截图区：信息区承载标题、说明、标签、状态或 CTA；截图区承载真实产品图片、界面捕获或 UI-like product visuals。

- 面板层级应通过表面色、留白、内容结构和轻量状态表达建立，默认不依赖外描边或重阴影。

- 产品展示面板可以比普通内容卡片更大，但仍应遵守响应式布局和 Content Rail / Viewport Band 规则。

### 产品截图

- 产品截图应优先使用真实产品图片、真实界面捕获或 UI-like product visuals，不使用抽象插画替代产品证据。

- 截图容器圆角：`--radius-2xl`（`32px`）。

- 图片必须使用稳定宽高比和 `object-fit` 规则，避免拉伸、压扁或变形。

- 除非源设计已经使用装饰性边框，否则不要额外给截图添加装饰边框。

- 截图内已有导航、Logo、状态栏或工具栏时，外层不再额外叠加装饰性浏览器框、阴影或水印。

## 按钮

按钮用于表达明确操作。品牌红按钮只用于页面主行动、关键提交、当前选中或高优先级入口，避免在同一区域出现多个红色按钮。

### 主按钮

- 背景：`Brand Red`。

- Hover：`Brand Red Hover`。

- 文字：`Text Inverse`，使用 `UI Label L`。

- 高度：常用 `48px`；导航按钮 `40px`；小卡片操作 `32px`

- 圆角：常规按钮使用 `--radius-md`；大 CTA / Hero 按钮可用 `--radius-lg`。

- Hover 动效：可轻微上移 `1–2px`

- Pressed 动效：可轻微缩放到约 `0.96`，或使用按下态表面

- Loading：保持原尺寸，不因文案或 spinner 导致宽高跳动

- Disabled：降低强调但保持文字可读，不直接使用过低透明度

### 次按钮

- 用于次级操作、取消、返回、辅助入口或同一区域内的非主行动。

- 表面优先使用白色、浅灰或透明背景。

- 默认不依赖重描边建立层级；必要时可使用低对比描边或浅色填充表达边界。

- Hover 可使用浅灰填充、轻微位移或状态表面变化，不使用品牌红大面积填充。

### 文本按钮 / Ghost 按钮

- 用于低优先级操作、辅助链接、列表行操作或卡片内部轻量入口。

- 默认保持轻量，不制造新的卡片层级。

- Hover 可使用文字加深、浅灰背景或下划线，但不要引入重阴影。

### 图标按钮

- 必须提供 accessible name。

- 常用热区：`32 x 32px`；移动端触控热区至少 `44 x 44px`。

- 内部图标使用 `--icon-size-md` 或 `--icon-size-lg`，需与按钮尺寸保持视觉居中。

- 默认不加外框；Hover / active 可通过浅灰填充、状态色或 focus outline 表达。

- 图标按钮不得仅依赖颜色表达状态。

### 危险 / 风险操作

- 删除、清空、不可逆提交等风险操作必须与普通主按钮区分。

- 是否使用品牌红作为风险色，以 `tokens.md` 的状态 Token 为准。

- 同一区域同时存在主行动和风险操作时，避免两个按钮都使用高饱和红色填充。

### 状态与可访问性

- 必须覆盖 default、hover、active、focus-visible 和 disabled；仅在按钮会触发异步操作、提交或等待结果时覆盖 loading。

- `focus-visible` 必须清晰可见，不能被 hover 或 active 覆盖。

- 移动端主操作触控热区至少 `44px`。

- Loading 状态必须保持原尺寸。

- Disabled 状态必须保持可读，并避免看起来仍可点击。

## Tabs 与分段控件

Tabs 与分段控件用于在同一层级内切换内容、视图或状态。整体应保持轻量、克制和可预测，不使用强装饰、重阴影或高饱和背景制造层级。

### 容器

- 外壳背景：`Page Background` 或其他文档化低对比中性表面。

- 外壳内边距：`--space-1`。

- 外壳圆角：`--radius-lg`。

- 默认不使用外描边；必要时可使用低对比分隔线或浅表面差异表达边界。

### 选项

- 选中项：`Selected Surface`，圆角 `--radius-md`。

- 文字：`UI Label L`。

- 选中态使用 medium 字重。

- 未选中项保持低对比，但必须可读。

- Hover 可使用轻微表面变化或文字加深，不使用品牌红大面积填充。

- Active / pressed 可使用按下态表面或轻微缩放，不做夸张动效。

- Disabled 项必须保持可识别且不可误认为可点击。

### 尺寸参考

- 产品 Tab 尺寸参考：`240 x 56px`。

- Hero Tab 尺寸参考：`108 x 40px`。

- 上述尺寸仅作为特定场景的视觉参考；通用 Tabs 应根据内容长度、容器宽度和响应式规则调整，不强制固定宽度。

### 响应式与溢出

- 选项数量较少时，优先保持同排显示。

- 选项较多或移动端空间不足时，可使用横向滚动、换行、下拉选择或折叠策略，但不得压缩到文字不可读。

- 移动端触控热区应满足可点击要求，避免仅保留文字本身作为点击区域。

### 可访问性

- 键盘焦点必须可见，`focus-visible` 不得被选中态或 hover 态覆盖。

- Tabs 应能通过键盘导航切换。

- 当前项需要有明确 selected / current 状态。

- 不得仅依赖颜色区分当前项。

## 标签与 Badge

标签与 Badge 用于表达分类、状态、属性、数量、版本或轻量强调。默认保持低对比、短文案、易扫描，不作为页面装饰元素批量堆叠。

### 尺寸

- 状态标签：高度 `32px`（组件专属几何），水平内边距 `--space-4`，圆角 `--radius-sm`。适用于页面级状态、筛选结果、重要属性或较强提示。

- 小属性标签：高度 `24px`（组件专属几何），水平内边距 `--space-2`，圆角 `--radius-sm`。适用于卡片、列表行、元信息和轻量分类。

- 资源卡 / Skill 卡等紧凑信息卡内，可继续使用高度 `20px` 的小标签模型，以卡片规则为准。

### 颜色

- 最新 / 热门标签：`Featured Orange` 文字与 `Featured Orange Soft` 表面。

- Skill / 工具标签：灰色文字，白色或浅灰表面。

- 主标签或关键强调标签可使用 `Brand Red` 文字与 `Brand Red Soft` 表面，但仅用于主分类、选中、焦点或关键强调。

- 状态标签应优先遵循 `tokens.md` 中定义的状态 Token。

### 使用规则

- 标签默认不加描边，优先通过文字颜色、浅表面、字重和位置表达层级。

- 当标签位于同色或低对比背景上，或承担可点击筛选、选中、focus-visible 状态时，可使用低对比描边或清晰 outline。

- 标签文案应短，优先使用 1–3 个词，避免长句。

- 同一组标签数量应克制，避免堆叠过多影响扫描。

- 标签只有在明确可点击、可筛选或可切换时，才使用按钮式 hover / active 状态。

### 禁用规则

- 不使用大面积品牌红或橙色填充制作普通标签。

- 不把标签当作按钮、Tabs 或表单控件的替代品。

- 不仅依赖颜色表达状态，应配合文字、图标、位置或 selected / current 语义。

## 表单、搜索与筛选

表单、搜索和筛选用于收集输入、缩小范围和调整内容视图。它们应保持清晰、可预测和可恢复，避免用重描边、重阴影或大面积警示色制造层级。

### 基础样式

- 输入框、搜索框、选择器、筛选条使用白色或浅中性表面。

- 圆角：常规输入、搜索框、筛选条和高强调控件统一使用 `--radius-md`（`12px`）；只有明确属于紧凑组件族的既有控件可继承 `--radius-sm`，并在报告中记录继承约束。

- 高度：紧凑控件 `40px`；常规控件 `44px`；主表单或移动端高频操作控件 `48px`。

- 默认不依赖外描边建立层级。

- 水平内边距：`--space-3` 或 `--space-4`。

- 图标尺寸：`--icon-size-sm` 或 `--icon-size-md`。

- 图标与文字间距：`--space-2`。

- Label、分组标题和必填 / 可选提示必须清晰。

### 状态表达

- 默认状态通过表面色、内边距、图标、占位文案和内容结构表达。

- Hover 可使用轻微表面变化或低对比状态填充。

- `focus-visible` 必须清晰可见，可使用 outline、焦点环或品牌红信号，但不得破坏可读性。

- Disabled 状态应降低可操作感，但文字、值和说明必须可读。

- Loading 状态应保持控件原尺寸，不因 spinner 或文案切换造成布局跳动。

- Empty 状态应说明当前无结果，并提供清除筛选、修改关键词或返回默认视图的路径。

- Success 状态可使用局部状态提示、图标或浅色填充，不应过度抢占主行动。

### 错误与恢复

- 辅助说明：`Supporting Text` 与 `Text Tertiary`。

- 错误文案应说明原因和恢复方式。

- 错误态使用项目已定义的 Error Token、错误图标、focus outline 或错误浅表面表达；不得直接复用 Brand Red。缺少 Error Token 时先定义语义 Token。

- 不把整块表单刷红，不使用大面积红色背景制造警示。

- 错误提示应靠近对应字段，避免用户无法定位问题。

- 多字段错误应提供顶部摘要或逐项定位路径。

### 搜索与筛选

- 搜索框应包含明确占位文案，说明可搜索对象或输入格式。

- 筛选器应支持清除、重置或返回全部结果。

- 已选筛选条件应有清晰的 selected / current 状态。

- 多筛选条件应避免堆叠过密，必要时使用标签、折叠组、菜单或抽屉承载。

- 桌面端筛选器可以横向排列。

- 移动端筛选器可换行、折叠、转为下拉，或放入抽屉；不得简单压缩到不可读或不可点击。

## 列表、表格与分页

列表、表格与分页用于承载高密度信息、横向比较和结果浏览。默认保持清晰、轻量、可扫描，不使用重卡片、粗描边或大面积阴影制造层级。

### 列表

- 列表行使用浅表面、间距、标题、元信息、状态和操作区建立层级；不要把每一行都做成重卡片。

- 列表项应有清晰主信息和次级信息，避免所有文字同等权重。

- 可点击行热区高度至少 `36px`；移动端触控热区应满足至少 `44px`。

- 行 Hover 使用浅表面变化、文字加深或轻量状态填充。

- 选中行使用 `Selected Surface`、Brand Red 小位置标记、状态图标或 selected / current 语义，不使用未定义的浅红填充或粗描边。

- 行内操作应靠近对应内容，避免所有操作堆到页面角落。

- 批量操作栏只在存在选中项时出现，并贴近列表上下文。

### 表格

- 表格用于横向比较、结构化数据浏览和批量管理，不作为默认装饰容器。

- 表头：使用 `Label` 并采用强调字重。

- 表格正文：使用 `Body M`。

- 列宽应服务内容识别，避免平均分配导致关键信息被截断。

- 数字、状态、操作列应保持对齐稳定。

- 表格行 Hover 使用浅表面变化。

- 选中行使用 `Selected Surface` 或 Brand Red 小位置标记，不使用未定义的浅红填充或粗描边。

- 表格应支持 empty、loading、error 和 success / saved 等必要状态。

- 移动端表格可横向滚动、减少列、转为列表卡片或进入详情页查看；不得强行压缩到不可读。

### 分页

- 分页高度：`32px` 或 `40px`。

- 分页控件必须覆盖 default、hover、active、focus-visible、selected / current、disabled 状态。

- 当前页必须清晰可见，不能只依赖颜色表达。

- Disabled 的上一页 / 下一页必须保持可识别，但不可误认为可点击。

- 移动端分页应减少页码数量，保留上一页 / 下一页和当前状态。

- 当结果量较少时，可隐藏分页；当结果加载中时，应保持分页区域稳定，避免布局跳动。

## 菜单、Popover、Tooltip、对话框、抽屉、Toast

临时层用于承载短时操作、补充信息、确认流程、筛选导航或状态反馈。所有临时层必须有明确触发源、关闭路径、键盘焦点路径和层级关系。

### 通用规则

- 临时层必须与触发源建立明确关系，关闭后焦点应返回触发源或合理的后续位置。

- 可关闭层应支持明确关闭按钮、Esc、点击外部或完成操作关闭；关键确认流程不得只依赖点击外部关闭。

- 模态打开时应防止背景滚动或误触。

- 遮罩使用低对比黑色透明度，不制造沉重压迫感。

- 层级应服务任务，不用重阴影、强描边或大面积品牌色制造浮层感。

### Tooltip

- Tooltip 仅用于短说明、补充解释或状态提示，不承载主要操作。

- 表面使用白色、深色或浅中性表面，取决于背景对比。

- 圆角：`--radius-sm` 或 `--radius-md`。

- 内边距：`--space-2` 或 `--space-3`。

- 文字：`Supporting Text`。

- 内容应短，不放复杂表单、按钮组或长段落。

### Menu / Popover

- Menu / Popover 用于轻量操作、选择项、快捷设置或上下文信息。

- 表面：白色或浅中性表面。

- 圆角：`--radius-sm` 或 `--radius-md`。

- 内边距：`--space-2` 或 `--space-3`。

- 文字：`UI Label M`。

- 菜单项必须有清晰 hover、active、focus-visible、selected / current、disabled 状态。

- 菜单项热区应大于文字本身，移动端触控热区至少 `44px`。

- Popover 内部可以使用分组、列表行、浅表面块或少量说明，不堆叠卡片。

### 对话框

- 对话框用于确认、少量表单、关键流程或不可逆操作前的决策。

- 标题、说明、主操作和次操作必须清楚。

- 主操作应表达确认目标，避免只写“确定”。

- 次操作应提供取消、返回或稍后处理路径。

- 对话框内部不堆叠卡片；复杂内容应使用分组、列表行或浅表面块承载。

- 危险操作需要明确后果和恢复方式，不只依赖红色表达风险。

### 抽屉

- 抽屉用于筛选、导航、长设置、详情补充或移动端次级流程。

- 内部使用分组标题、列表行、表单控件和浅表面块，不堆叠卡片。

- 抽屉必须提供明确关闭路径。

- 桌面端抽屉应保持合理宽度，不覆盖无关内容过多。

- 移动端抽屉可使用底部抽屉、全屏或近全屏布局，但不得压缩内容到不可读。

### Toast

- Toast 用于短消息、轻量反馈或异步状态提示。

- Toast 位于前层但保持轻量，不应遮挡关键主操作或模态确认。

- 圆角：`--radius-sm` 或 `--radius-md`。

- 内边距：`--space-3` 或 `--space-4`。

- 图标：`--icon-size-sm` 或 `--icon-size-md`。

- 文案应短，优先说明结果和必要的下一步。

- 错误 Toast 应说明原因或恢复方式；不使用大面积红色背景。

- 多个 Toast 不应无限堆叠，应限制数量或合并提示。

### 层级顺序

`page content < sticky / header < popover / menu / tooltip < drawer / dialog < toast < critical overlay`

普通 Toast 可以高于页面内容和非关键浮层，但不应遮挡关键 Dialog 操作；critical overlay 仅用于阻断式故障、权限、安全或不可继续流程。

## Empty、Loading、Error、Success

反馈状态用于解释系统当前结果、处理进度和操作反馈。状态表达应清楚、轻量、可恢复，不使用大面积高饱和颜色制造情绪化视觉。

### Empty

- Empty 状态包含简洁图标、清楚标题、一句说明和一个可执行的下一步操作。

- 标题应说明当前为空的原因或对象，例如“暂无文档”“未找到匹配结果”。

- 下一步操作应具体，例如“上传文档”“清除筛选”“创建第一个 Skill”。

- Empty 不应只展示装饰插画或空泛鼓励语。

### Loading

- Loading 状态必须保持尺寸稳定，避免内容加载前后布局跳动。

- 可使用 spinner、skeleton、进度文本或局部占位。

- 避免强闪烁动画、快速闪动 skeleton 或大面积高对比 loading 效果。

- 长时间加载应提供说明、取消、重试或后台处理路径。

### Error

- Error 状态应说明发生了什么、为什么重要，以及用户可以怎么做。

- 错误文案应尽量具体，避免只写“出错了”。

- 应提供恢复路径，例如重试、返回、修改输入、清除筛选、重新授权或联系支持。

- Error 使用项目已定义的 Error Token、错误图标、错误浅表面或焦点提示，不把整块区域刷红，也不直接复用 Brand Red。

- 高风险或阻断型错误可以提高强调层级，但仍应保持可读和可恢复。

### Success

- Success 状态用于局部确认、保存成功、提交完成或流程完成反馈。

- 成功反馈应尽量贴近触发操作或结果区域，不无意义地打断用户流程。

- 可使用轻量图标、短消息、Toast、局部状态标签或结果更新。

- 不引入大面积绿色视觉主题，不让 Success 变成新的页面主视觉。

- 成功后如有下一步，应提供明确入口，例如“查看结果”“继续配置”“返回列表”。

## 组件中的媒体

组件中的媒体用于提供产品证据、状态说明、内容预览或视觉辅助。媒体应服务用户理解产品行为，不使用抽象装饰替代真实产品信息。

### 基础规则

- 产品组件应优先使用真实产品截图、界面捕获、产品化 mockup 或 UI-like product visuals。

- 当用户需要检查产品行为、界面状态、配置路径或工作流结果时，避免使用抽象插画。

- 截图和媒体容器必须使用稳定宽高比和响应式尺寸。

- 图片不得裁掉理解产品状态所需的关键信息，例如导航、状态栏、选中项、错误提示、关键按钮或数据结果。

- 媒体应与所在卡片、面板或列表项的内容层级一致，不额外制造装饰性边框、重阴影或浏览器壳。

### 尺寸与适配

- 图片应设置明确的宽高比、最大宽度和 `object-fit` 规则，避免拉伸、压扁或变形。

- 产品截图优先使用 `object-fit: contain`，确保关键界面信息完整可见。

- 封面图、头像、图标预览等非关键状态媒体可使用 `object-fit: cover`，但不得裁掉主体。

- 移动端应根据容器重排媒体与文本，不应简单压缩到不可读。

### 视觉处理

- 媒体容器圆角应跟随所属组件规则；产品截图面板可使用更大的圆角，普通卡片媒体使用中等圆角。

- 除非源设计本身包含边框或浏览器壳，不要额外添加装饰性边框。

- 不使用高饱和渐变、bokeh、玻璃拟态、重投影或无意义遮罩包装产品截图。

- 如果截图内已包含 Logo、导航或状态栏，外层不再重复叠加装饰性 Logo、水印或假浏览器框。

### 状态

- 媒体加载中应保持容器尺寸稳定，可使用浅色 skeleton、占位块或低对比 loading。

- 媒体加载失败时，应提供轻量错误提示、替代文案或重试路径。

- 媒体需要有合适的替代文本；纯装饰媒体应避免干扰辅助技术。
