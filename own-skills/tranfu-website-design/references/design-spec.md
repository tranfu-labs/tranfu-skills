## 模块化引用

本文件是 TranFu 官网与产品 UI 的设计总览，保留视觉方向、原则判断、历史反推依据和跨模块执行约束。执行具体设计或实现任务时，应优先加载同目录下的对应模块：

- `tokens.md`：定义颜色、字体、间距、圆角、阴影、图标与状态 Token。
- `cards.md`：定义卡片、组件承载容器、按钮、Tabs、标签、表单、列表、弹层与反馈状态。
- `responsive-layout.md`：定义浏览器响应式宽度、Section 外层、Content Rail、Viewport Band、导航宽度、断点与溢出检查规则。

当模块化引用与本总览存在差异时，以模块中更具体的规则为准；如出现原则性冲突，则按「十大原则性设计」进行判断。

## 1. 设计定位

TranFu 视觉属于 **AI Agent 系统型品牌与产品体系**。官网侧重连续叙事、品牌理解和真实产品证据；产品 UI 侧重任务效率、状态可见性和高密度信息中的稳定操作。两者共享以下整体气质：

- 理性、克制、工程化、可信赖。
- 以真实产品截图和工具界面作为主视觉，而不是抽象插画。
- 通过浅灰网格、命令行、状态标签、扫描/解密/流式输出等细节强化 Agent / OS / Workflow 感。
- 品牌红只做关键动作和聚焦，不大面积铺满。

关键词：`AI Agent`、`OS`、`控制台`、`工作流`、`真实产品`、`轻量科技感`。

## 十大原则性设计

1. **真实产品优先**
优先展示真实界面、真实工作流、代码、配置、数据状态或高保真产品化 mockup。视觉表达必须服务产品可信度，不用抽象装饰替代产品证据。
2. **中性表面承载内容**
页面以白色、#F7F7F7、#F0F0F0、#F5F5F5 等中性表面建立秩序，让内容、状态和行动成为视觉焦点。
3. **品牌红是信号色，不是背景色**
#E63A46 仅用于 Logo、主行动、选中、焦点、风险提示与关键强调；避免作为大面积背景、渐变装饰或纯氛围色使用。
4. **导航服务浏览器视口**
顶部导航应保持浏览器级稳定性，跟随 viewport 建立位置和宽度逻辑，不被内容舞台或正文容器错误约束。具体 inset、宽度和断点规则以响应式布局模块为准。
5. **信息密集但可读**
保留工程化产品应有的信息密度，但必须通过稳定间距、清晰分组、短标签、状态层级和扫描路径保证可读性；不通过放大空卡片制造“高级感”。
6. **层级优先来自结构，而非描边**
卡片、面板、导航、会话气泡和状态块默认不依赖 1px 描边建立层级。优先使用表面色、留白、状态填充、阴影克制和内容结构；描边仅在分隔、可用性或状态表达必要时使用。
7. **排版服务信息层级**
MiSans 是主体 UI 字体，字距保持 0。标题、正文、元信息、标签和徽标各自遵守对应尺寸、字重与行高，不用巨型营销字破坏工具型界面的秩序。
8. **组件行为可预测**
按钮、标签、卡片、Tabs、输入框和状态反馈遵循统一的圆角、尺寸、间距、状态和交互语法，让用户一眼判断可点击对象、当前位置和操作结果。
9. **响应式忠于浏览器，而非压缩桌面**
先还原 1440 桌面基准，再验证 1920、1280、756、375 等关键宽度。移动端应重排信息结构，而不是简单压缩桌面布局。
10. **克制胜过炫技**
避免渐变球、bokeh、玻璃拟态、重阴影和无意义动效。动效只用于状态变化、聚焦、层级转换和可操作反馈，避免装饰性抢戏。

## 2. 三层视觉层级

TranFu 页面应根据「底层 / 中层 / 前层」建立清晰视觉层级，让用户先建立页面主题和核心模块认知，再理解说明内容，并能快速识别入口、步骤编号和关键操作。

- **底层：页面背景与大区域结构**
使用浅灰或低对比背景作为整体承载层，不抢视觉焦点；区域之间保持充足留白，让页面结构清晰、有呼吸感。
- **中层：主要内容模块**
页面标题和 Section 标题用于建立主识别；核心功能卡片、流程步骤、模块标题与说明文字构成中层信息结构。内容应对齐清晰、间距统一、分组稳定，帮助用户快速扫描和理解。
- **前层：重点操作与视觉强调**
按钮、入口文案、关键图标、数字步骤、状态提示和品牌强调色属于前层。前层应通过更高对比、更明确字重、状态填充或品牌红，引导用户识别可点击操作、当前重点和任务进展。


### 中层容器规则

资料模块、功能入口、表单分组、设置项、流程步骤、知识库 / 文档条目必须有清晰承载。页面背景上使用卡片或容器；已成型的大面板内使用分组标题、列表行、浅表面块或留白，避免文字、图标、状态和操作按钮直接漂浮在页面背景上。

父级已是卡片或面板时，应优先使用分组、列表行或浅表面块承载内容，避免继续套卡片。

常规页面最多使用「页面背景 -> 区块 / 面板 -> 内容项」三层视觉承载。除弹窗、独立重复项或明确设计需求外，不再增加第三层卡片。

独立卡片 / 容器至少包含独立背景、圆角、内边距、模块间距、清晰标题、说明、状态和操作区域。默认无外描边；轻阴影仅用于 hover、active、浮层、可交互前层或明确需要强调层级的场景。

## 3. 画布与页面结构

### 当前布局原则

TranFu 页面以浏览器自适应为默认实现方式。页面主体、Section 外层和全宽背景 Band 应跟随 viewport 变化，不应将页面主体固定为 `1920px`。

`1920px` 仅作为历史设计稿和大屏视觉校验参考，不作为新页面、重构页面或响应式修复的固定画布宽度。

### 页面背景

页面默认背景使用 `#F0F0F0`，并通过白色、浅灰表面、内容 Rail 和模块容器建立清晰层级。背景本身应退后，不承担主要视觉焦点。

### 内容宽度

页面外层和 Section 外层保持 fluid。核心阅读内容、正文、卡片栅格和表单内容可以使用内层 Content Rail 控制阅读宽度；具体最大宽度、断点、Section 外层、Viewport Band 和溢出检查规则以 `responsive-layout.md` 为准。

Hero 视觉、产品截图舞台、全宽背景 Band、浏览器级导航和其他需要贴合 viewport 的表面，不应被默认限制到正文 Content Rail 内。

### 导航例外

顶部导航属于浏览器级 Header，应跟随 viewport 建立位置和宽度逻辑，不参与旧版固定舞台或正文内容容器的宽度约束。具体导航高度、左右 inset 和响应式规则以 `responsive-layout.md` 为准。

### 历史固定舞台参考

旧版 `1920px` 固定设计稿中的绝对位置和高度仅用于理解历史页面、排查旧实现或做窄范围兼容编辑。新页面、重构页面和响应式修复不得按旧稿 `top` / `height` 坐标复刻布局，应优先遵循 `responsive-layout.md`。

## 4. 色彩系统

色彩系统的权威 token、用途和禁用规则维护在 `tokens.md`。本总览只保留方向：页面以白色、`#F7F7F7`、`#F0F0F0`、`#F5F5F5` 建立明亮中性舞台；品牌红只用于 Logo、主行动、选中 / 激活、焦点、关键风险提示和关键强调；卡片、面板、导航、标签、输入、会话气泡和状态块默认不使用 `1px` 外描边作为层级系统。

## 5. Logo 规范



### 基础组合

- 当前主 Logo 为横向组合：左侧品牌图形标志 + 右侧 `TranFu` 英文字标。
- 标准尺寸：`135 x 32px`，其中图形标志约 `32 x 32px`，字标区域约 `88 x 21px`。
- 图形与字标应作为一个整体 lockup 使用，默认不拆分、不重新排版。
- 导航栏中 Logo 右侧可搭配 `AI Agent OS` 深色徽标，二者间距 `16px`。
- `AI Agent OS` 徽标不属于 Logo 本体，属于品牌说明 badge；在空间不足、页尾或第三方露出时可省略。



### 颜色版本


| 版本    | 颜色                    | 使用场景                |
| ----- | --------------------- | ------------------- |
| 标准红色版 | `#E63A46`             | 浅灰、白色、低对比浅色背景上的默认版本 |
| 深色单色版 | `#111111` / `#2F2D2D` | 红色会与环境冲突，或需要降低品牌强调时 |
| 白色反白版 | `#FFFFFF`             | 深色背景、深色图片遮罩、视频封面等场景 |


- 标准红色版优先级最高，适用于顶部导航、品牌露出、CTA 邻近区域。
- 不在红色、橙色、复杂图片或高饱和渐变背景上直接使用红色 Logo；此类场景应加浅色底板或使用白色反白版。
- 不使用透明度降低 Logo 识别度；禁用态可降低容器/按钮状态，不直接降低 Logo alpha。



### 留白与最小尺寸

- 安全留白以图形标志高度 `X` 为基准：Logo 四周至少保留 `0.5X` 空白。
- 与导航链接、按钮、语言切换器等交互元素的最小间距不低于 `32px`；与 `AI Agent OS` badge 间距固定 `16px`。
- 桌面端主导航 Logo 高度保持 `32px`。
- 移动端或紧凑场景中，横向 Logo 高度不低于 `24px`；低于该尺寸时优先只使用图形标志。
- 单独图形标志最小显示尺寸不低于 `20 x 20px`，低于该尺寸不作为可识别品牌标识使用。



### 使用方式

- 顶部导航使用「Logo lockup + `AI Agent OS` badge」作为标准组合。
- 页尾、社交头像、favicon、App icon 等空间受限场景可只使用图形标志。
- Hero 或标题区如需品牌露出，优先使用文字内容中的 `TranFu`，避免重复堆叠完整 Logo。
- 产品截图内若已出现 Logo，页面外层不再额外添加装饰性 Logo 水印。
- Logo 附近避免同时出现多个红色装饰元素，确保品牌红的视觉焦点明确。



### 禁用规则

- 不拉伸、压扁、旋转、倾斜或重新绘制 Logo。
- 不改变图形与字标的比例、间距或相对位置。
- 不给 Logo 添加投影、描边、渐变、发光、纹理或 3D 效果。
- 不把 Logo 放进胶囊按钮、标签或复杂卡片装饰中作为普通图标使用。
- 不用 Logo 替代普通功能图标；功能按钮优先使用明确的 UI 图标。

## 6. 内容与文案语气



### 语气

- 简洁、具体、工程化。
- 多使用“真实工作”“复制即用”“实测”“驾驶舱”“看板”“路径”“闭环”等可执行词。
- 避免空泛营销词堆叠。



### 命名模式

- 产品：`TranFuBar`、`TranFu//Agents`、`GitHub Learning Lab`
- 功能标题：`代码库理解驾驶舱`、`30 日抢救陪练`、`团队级 Agent 实时看板`
- CTA：`开始使用`、`加入我们`、`加入社群`、`安装 Skill`



## 7. 新增页面 / 组件设计检查清单

### 引用与优先级

- 是否按任务加载对应模块：`tokens.md`、`cards.md`、`responsive-layout.md`？
- 如总览与模块规则存在差异，是否优先采用更具体的模块规则？

### 视觉层级

- 是否使用 TranFu 的中性表面系统，并让品牌红只服务品牌、主行动、选中、焦点和关键强调？
- 是否建立「底层 / 中层 / 前层」层级：背景退后，核心模块稳定，按钮、入口、步骤编号和状态提示明确前置？
- 资料模块、功能入口、表单分组、设置项、流程步骤、知识库 / 文档条目是否有清晰承载？
- 是否避免文字、图标、状态或按钮直接漂浮在页面背景上？
- 是否避免过度卡片套卡片？

### 布局与响应式

- 页面、Section 外层和全宽 Band 是否跟随浏览器变化，而不是固定为旧版 `1920px` 舞台？
- 是否正确区分 Content Rail、Viewport Band、浏览器级导航和旧版固定舞台？
- 是否完成关键宽度检查，避免横向溢出、内容挤压或桌面布局被简单压缩到移动端？

### 组件与状态

- 字体、颜色、间距、圆角、图标、阴影和状态是否沿用 `tokens.md`？
- 卡片、按钮、Tabs、标签、输入、列表、弹层和反馈状态是否遵循 `cards.md`？
- 卡片和面板是否默认轻量，不依赖满屏 `1px` 描边建立层级？
- 是否覆盖 default、hover、active、focus-visible、selected、disabled、loading、empty、error、success 等必要状态？
- 表单、搜索、筛选、菜单、弹窗、toast、空态、加载态和错误态是否遵循同一套中性表面与红色焦点语言？

### 品牌、内容与动效

- Logo 是否使用标准 lockup、正确颜色版本、足够留白和最小尺寸？
- 是否优先使用真实产品图、界面截图或 UI-like product visuals，而不是纯抽象装饰？
- 动效是否轻、慢、克制，并支持 `prefers-reduced-motion`？
- 文案是否明确说明“解决什么真实问题”，避免空泛营销词堆叠？

### 可访问性

- 键盘焦点、图标按钮名称、触控热区、文字对比、表单提示和弹层关闭路径是否可用？
