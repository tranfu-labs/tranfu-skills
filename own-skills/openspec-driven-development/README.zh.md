# openspec-driven-development

把日常开发跑成「方案(停下等你确认) → spec → 码 → 归档 → commit → 问要不要 PR」的闭环, 让方案、代码、事实源始终对得上。

## 什么时候用它

**开发新功能**:

我在 openspec 约定的仓库里加个功能 / 修个 bug / 重构一段代码, 想让它先出方案我点头再动手, 落盘、写码、归档一条龙, 最后问我一句要不要提 PR。

**咨询开场**:

我在问「你会怎么改这段」「这个功能做在哪里合适」「为什么这里不对」, 想让它顺势把讨论收敛成方案, 而不是陪我漫无目的地聊。结论若「其实不用改代码」, 它给解释就收工, 绝不硬造 change。

**实施已定方案**:

我已经在别的对话里写好了 `openspec/changes/<X>` 的方案, 直接说「实施这个 change」, 让它跳到写码环节, 别再从采访重头走。

**符合度复核**:

「实施完了帮我检查代码是否符合方案 / 有无遗漏」——让它以 change 为基准逐条核对, 有偏差就回到写码环节补齐。

**快车道**:

一两行的微改动、改个文案、调个默认值, 我不想被完整闭环压得喘不过气——跳过 change 落盘, 但反思、commit 照走, 结束时照样问我要不要 PR。

**放行自动**:

「方案写好后自动开始写代码」「后面全自动」——解除方案确认这个默认停点, 一路自动跑到 commit; 但提 PR 不在"自动"范围里, commit 后仍会停下问我。想连 PR 也托管, 得明确说「最后提 PR」。

**不接**:

冷启动搭 `openspec/` 骨架 → **project-init-docs**; 打 tag / 写 changelog / 定版本号 → **release**; 「项目整体是否符合规范 / 标准」这类脱离具体改动的合规审计 → 不触发; 纯查询 / 查资料 / 跑一条不改代码的命令 → 不触发。

## 它会产出什么 / 你会看到什么

**默认两处停: 方案落盘后停下等你说「开始写代码」; commit 后停下问要不要提 PR**——别指望它一上来就闷头写, 也别担心它擅自把东西推出去。

- **推进计划**: 一进来就把 `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) ⏹停⏹ 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) ⏹问PR⏹` 亮出来, 让你指定停点、放行自动或加事; 状态一律「中文名(英文slug)」双名显示, 你用中文或英文指认都行; 「后面全自动」只放行方案停点, PR 和建分支永远要明确点名
- **采访诊断**: 采访阶段天然停下等你回答, 最后一问固定「还有要补充的吗？」——它不会替你答、不会跳过
- **聊天框方案**: 先在对话里出方案 (改页面附字符图; 含可测逻辑或单文件 diff > 200 行附单测和 AI 验证用例), 自己复查通过才落盘
- **默认不建分支**: 就在当前分支落盘、写码、commit——包括 main; 只有你明确说「切个分支」或明确要提 PR 时才切 feature 分支
- **落盘 change**: `openspec/changes/<change-id>/` 下写 proposal / design / tasks / spec-delta; 改页面时另加 `wireframes.md` (绝不塞进 `design.md`)
- **按 spec 写码**: 落地方案里定义的测试用例, 跑单测或 AI 验证流程 (playwright 截图 / 跑命令看输出 / 手动一遍)
- **归档三步**: change 目录移到 `openspec/changes/archive/<日期>-<id>/`; spec-delta 合进 `openspec/specs/<domain>/`; 字符图回流到 `docs/wireframes/pages/<page>.md` 与 `flow.md`
- **收尾询问 PR**: 更新 `AGENTS.md` + commit 到当前分支; 有 remote 就停下问「要不要提 PR？」——remote 有 `dev` 分支默认提到 dev, 否则 main; 你同意后它才切分支承载提交、push、`gh pr create` (body 直接粘 `proposal.md`; 上下文出现过 issue 编号首行加 `Closes #<编号>`), 把 PR URL 报给你
- **绝不会做**: 未经询问或明确指示就 push / 开 PR; 擅自新建分支; 自己合并 PR; 替你答采访问题; 在没有 `openspec/` 的仓库里擅自搭骨架; 从归档里翻出旧 proposal 冒充新方案

## 前置条件 / 边界

**前置**:

仓库根有 `openspec/` 目录 (`openspec/specs/` + `openspec/changes/`); 能跑 `git`, 有 remote 且想开 PR 时需要 `gh` (或 `glab mr create` 之类的等价物)。

**相邻 skill 分工**:

| 动作 | 交给 |
|---|---|
| 冷启动搭 `AGENTS.md` + `openspec/` 骨架 | **project-init-docs** |
| 打 tag / 写 changelog / 定版本号策略 | **release** |
| 判断素材值不值得沉淀成 skill / spec | 内容判定链, 与本 skill 无关 |

**不接的场景**:

- 纯查询 / 查资料 / 跑一条不改代码的命令
- 与具体改动无关的合规审计
- 往已归档 `archive/<日期>-<id>/` 里追加改动——要重新起一个新 change

**微妙边界**:

- 问「代码是否符合方案 / 某个 change」→ 触发符合度复核; 问「项目整体是否符合规范 / 标准」→ 不触发 (脱离改动的合规审计)
- 「实施 openspec/changes/<X>」显式句式 → 直接进写码, 前提是方案完整; 残缺 → 退回采访, 绝不硬撑着实现
- 「后面全自动」只解除方案确认停点——采访照停 (对话往返没法替答), PR 照问, 分支照旧不建; 拿不准"自动"覆盖到哪, 它按窄的算
- 微改动快车道 → 跳过 change 落盘, 但仍走反思 + commit + PR 询问; 动手中发现改动其实不小 (碰可测逻辑 / 多文件 / 远超几行) → 立即退回采访阶段走完整流程
