# OpenSpec 驱动的开发执行闭环

把一个开发任务从「需求」一路带到「代码落地 + 事实源归档」，**先有经确认的方案、再写代码、写完让事实源反映现状**。

这个 skill 的价值不在某一步，而在闭环本身：跳过方案直接写码会失控，写完不归档会让事实源慢慢失真。所以它的每一步都为同一个可观察结果服务——**方案、代码、事实源三者始终一致**。

**项目有两套事实源**（视项目而定）：

- `openspec/specs/<domain>/spec.md` 是**行为事实源**，靠 change 的 `spec-delta/` 流转更新。
- `docs/wireframes/`（若项目有）是**版式事实源**，靠 change 的 `wireframes.md`（本项目扩展，按需新建）流转更新。

归档时两套事实源一起更新——`spec-delta` 合并回 `specs/`，`wireframes.md` 回流到 `docs/wireframes/`。**项目级归档约定**写在 `openspec/changes/AGENTS.md`，**这个 skill 是它的执行口径**。

## 边界（和相邻 skill 的分工）

这个 skill 只管「**已在 openspec 约定下**的日常开发执行」——也包括咨询口吻的开发问题（「你会如何做 / 改 X」「X 做在哪里」）和 bug 讨论（「为什么 X 不对」），这些都从步骤 1 进。碰到下面的活，交出去、别自己干：

- 项目还没有 `openspec/`、要从零搭 AGENTS.md + openspec 骨架 → 停下，**主动提议**用 **project-init-docs**（见步骤 0）。
- 用户要打 tag / 写 changelog / 定版本号 → 交给 **release**。
- 非开发任务（问答、查资料、跑一条命令）→ 不触发，正常回答即可。
- **「符合」消歧**：问「代码是否符合**方案 / spec / 某个 change**」→ 这是步骤 6，触发；问「项目是否符合**规范 / 标准 / 约定**」这类泛合规审计 → 与具体改动无关，不触发。

## 工作流总览

```
步骤0 识别开发任务 + openspec 检测 + 进场路由
   ├─非开发───────────────────────────> 终止
   ├─无 openspec──────────────────────> 停下，主动提议初始化（project-init-docs）
   └─有 openspec，按开场分流：
       "实施 openspec/changes/X" ───────> 直接进 步骤5
       "实施完了，检查是否符合方案/遗漏" ─> 直接进 步骤6
       微改动 ─────────────────────────> 步骤1 快车道
       其余开发任务 / 开发问题 / bug 讨论 ─> 步骤1
   │
步骤1 采访 / 诊断澄清目标
   ├─结论：不需要改代码──> 给出诊断/解释/结论，结束
   └─结论：需要改代码────> 步骤2
   │
步骤2 聊天框出方案（字符图 / 测试用例）→ agent 自审 → 自动到步骤3
   │
步骤3 落盘 openspec/changes/<change>/（默认在当前分支，不新建分支）
   │
步骤4 反思方案 → ✓ 方案已定(plan-written) ──⏸ 默认停下等「开始写代码」（除非用户指定自动）
   │
步骤5 按 spec 写代码 → ✓ 代码写完(code-written)
   │
步骤6 反思代码符合度 ──偏差──> 回步骤5 改 ──通过──> ✓ 验证通过(code-verified)
   │
步骤7 归档（按 openspec/changes/AGENTS.md 的「归档」节）：
       ① changes/<id>/ → changes/archive/<YYYY-MM-DD>-<id>/
       ② spec-delta 合进 openspec/specs/<domain>/
       ③ 若有 wireframes.md → 回流到 docs/wireframes/pages/<page>.md 与 flow.md
   │
步骤8 更新 AGENTS.md → git commit（当前分支）→ ✓ 已提交(committed)
       └─有 remote ──⏸ 默认停下询问是否提 PR（dev 优先，否则 main）
            └─用户同意 / 开场已明确说要提 PR ──> push + 开 PR → ✓ 已开PR(pr-opened)
```

## 状态枚举与外部锚点

整条流水线只有 **7 个状态枚举**，给外部（用户在 prompt 里、上游 skill、流程编排）一个稳定锚点，可指定「跑到某个状态就停」「某个默认停点放行自动」或「在某个状态额外做点什么」。中间的机械动作（落盘、归档、commit）合并到上下游枚举里，用自然语言报告进度即可，不再单独命名。

**状态的标准表述是「中文名(english-slug)」双名形式**，如 `方案已定(plan-written)`——对话里展示推进计划、汇报进度 MUST 用这种形式；英文 slug 是稳定锚点（不随措辞和翻译漂移），中文名管一眼读懂。**用户指认状态时，说中文名或英文 slug 任一都有效**。

**三个天然 / 默认停点**：

1. `采访中(interviewing)` 是 **HARD STOP**——必须等用户回答，agent 不能自审通过、不能替答、不能跳过。
2. `方案已定(plan-written)` **默认停**——方案落盘 + 反思通过后停下，等用户明确说「开始写代码」（或等价指令）才进入步骤 5。**除非用户指定自动**（如「方案写好后自动开始写代码」「后面全自动」），才自动衔接。
3. `已提交(committed)` 后的 **PR 询问**——commit 完成、有 remote 时，默认停下**询问**要不要提 PR（并报默认目标分支）。**除非用户明确说过要提 PR**（开场或中途点名「最后提个 PR」），才不问直接提。

**「自动」的范围有硬边界**：用户说「后面全自动」「一路跑到底」之类，解除的只是 `方案已定(plan-written)` 这个默认停点——流水线自动跑到 `已提交(committed)`。**提 PR 和新建分支永远不在"自动"的默认范围里**：只有用户话里明确出现「提 PR / 开 PR」才把 `已开PR(pr-opened)` 纳入自动；只有明确出现「切分支 / 开个分支」（或明确要提 PR 因而必须有分支）才新建分支。拿不准用户的"自动"覆盖到哪 → 按窄的算，到停点照停照问。

**`已开PR(pr-opened)` 达成后 NEVER 自己合并 PR**——合并由 reviewer 拍板。

### 状态枚举表

| 状态 | 含义 | 覆盖步骤 | 默认行为 |
|---|---|---|---|
| `采访中(interviewing)` | LLM 正在采访 / 诊断，等用户回答 | 步骤 1 进行中 | **天然等用户**（HARD STOP） |
| `目标确认(interview-confirmed)` | 采访结束，目标明确，进入分叉（不改代码 → end；要改代码 → 继续）| 步骤 1 末 | 自动 |
| `方案已定(plan-written)` | 方案出完 + 落盘 `openspec/changes/<change>/` + 反思通过 | 步骤 2 + 3 + 4 整体 | **默认停**，等「开始写代码」；用户指定自动才放行 |
| `代码写完(code-written)` | 按 spec 的代码（含落地测试）写完 | 步骤 5 末 | 自动 |
| `验证通过(code-verified)` | 代码符合度反思通过、单测绿 / AI 验证流程跑通 | 步骤 6 末 | 自动（可被指定为停） |
| `已提交(committed)` | 归档（移目录 + 合 spec-delta + 必要时回流 wireframes）+ 更新 AGENTS.md + commit 到当前分支 | 步骤 7 + 8 前半 | 自动；**到达后有 remote 则默认停下询问是否提 PR** |
| `已开PR(pr-opened)` | push + 开 PR（目标：remote 有 `dev` → `dev`，否则 `main`；用户指定则从其指定）| 步骤 8 后半 | **默认不做**——询问得同意、或用户明确说过要提 PR 才做 |

> 为什么 `方案已定(plan-written)` 默认停：方案是用户在写码前唯一的纠偏点，默认停下让用户过目，比"agent 自审通过就往下冲"更符合"先有经确认的方案再写码"的立意。嫌烦的用户一句「后面全自动」就能放行，成本极低。
> 为什么提 PR / 新建分支默认不做、只询问或等明确指示：这两个动作改变仓库拓扑、且 PR 是外向动作（推到 remote、进别人的 review 队列），做错了撤销成本高。默认询问、明确点名才自动，比"默认做"安全得多。
> 为什么把落盘 / 归档 / commit 都合并进相邻枚举：这些是机械动作，不会有人想"停在落盘之后"。失败时自然会报错，用户不需要枚举名也知道哪炸了。

### 推进计划的显示模板

**步骤 0 一定下进场点，MUST 先在对话里明示推进计划，再开干**——这是给用户最后一次「指定停在哪儿 / 放行哪儿 / 加事」机会。常见形态：

**A. 默认（两处默认停）**：

> 推进计划（步骤 1 采访会自然停下等你回答；方案落盘后停下等你说「开始写代码」；commit 后停下问要不要提 PR——若提，remote 有 dev 就默认提到 dev，否则 main）：
> `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) ⏹停⏹ 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) ⏹问PR⏹`

**B. 用户放行自动**（例：prompt 里写「方案写好后自动开始写代码」或「后面全自动」）：

> 推进计划（按你要求方案落盘后自动开始写码，一路跑到 commit；提 PR 不在自动范围，commit 后仍会停下问你）：
> `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) → 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) ⏹问PR⏹`

**C. 用户明确说了要提 PR**（例：「做完最后提个 PR 到 dev」）：

> 推进计划（方案落盘后仍会停下等你说「开始写代码」；你已明确要提 PR，commit 后不再问、直接切分支推 PR 到 dev）：
> `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) ⏹停⏹ 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) → 已开PR(pr-opened→dev)`

**D. 用户指定额外动作**（例：「在方案已定时同步把方案贴到 Slack」）：

> 推进计划（在 `方案已定(plan-written)` 触发额外动作：把方案贴 Slack）：
> `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) ⨁[贴 Slack] ⏹停⏹ 代码写完(code-written) → …`

形态可组合（B+C = 全自动且最后提 PR，中间一次都不停）。符号 `⏹` / `⨁` 不是硬约定——核心是**把状态双名写清楚**，让用户能精确指认是哪一步。

> 为什么要枚举名：让外部能精确指认"在哪儿停 / 在哪儿加事"。"在第 3 步停"是脆弱的（步骤编号会随版本漂移），"在 `方案已定(plan-written)` 停"是稳定的。
> 为什么默认停但允许一句话放行：逐步问"要不要继续"会演成"用户疯狂回车"，全自动到底又剥夺了用户在关键点的控制权。折中是**只在两个高价值决策点默认停**（方案确认、PR 外发），其余自动；用户用一句自然语言就能改写默认（「后面全自动」「最后提 PR」「停在验证通过」）。

## 步骤 0 · 识别 + openspec 检测 + 进场路由

先判断这是不是一个**开发相关任务**（要写 / 改代码，或讨论怎么开发、排查 bug）。不是（纯查询、研究、装 skill、审稿、与具体改动无关的合规审计、单独的 git 操作）→ 终止本流程，正常回应。

是开发相关，再查仓库根有没有 `openspec/` 目录：

- **没有** → **停下，不要自己搭骨架**，但要**主动提议**：当前项目还没初始化 openspec，建议先用 `project-init-docs`（或 `openspec init`）搭好 `openspec/specs` + `openspec/changes` + AGENTS.md。征得同意可引导 / 协助初始化，完成后再回来走这条流水线。
- **有** → 按开场**路由进场点**（穷尽，带兜底）：

| 开场判据 | 进场点 |
|---|---|
| `实施 openspec/changes/<X>`（spec 已写好，只差实现）| 直接进 **步骤 5** |
| `实施完了 / 已完成，检查是否符合方案、有无遗漏` | 直接进 **步骤 6** |
| 一两个文件、几行的微改动 | **步骤 1 快车道** |
| 其余开发任务 / 开发问题 / bug 讨论（含「为什么 X 不对」「你会如何做 X」）| **步骤 1** |
| 否则（判不准）| 先问用户一句确认意图，再决定进场点或终止 |

> 为什么停下而不自动搭骨架：搭骨架会写入大量约定文件、改变仓库结构，属于一次性脚手架决策，应由用户显式发起、且有专门的 skill 负责。这里只「提议 + 引导」，不擅自改仓库结构。
> 为什么要进场路由：真实开场常常不是从需求起步，而是「实施已写好的 change」「检查刚实施的符合度」「先讨论 / 诊断一个 bug」。硬把它们拽回步骤 1 会重复劳动或答非所问。

**步骤 0 末必做**：进场点定下后，**在对话里明示推进计划**（按上节「推进计划的显示模板」给出对应形态）——把整条状态链亮出来，标清哪些自动跑、哪些默认停、PR 是问还是直接提。同时**解析用户 prompt 里的自动化指令**：「后面全自动 / 方案写好自动写码」→ 放行 `方案已定(plan-written)`；「最后提 PR（到 X）」→ `已开PR(pr-opened)` 纳入自动（目标 X 或默认 dev/main）；「切个分支做」→ 落盘前建分支。**NEVER 跳过明示推进计划直接进入步骤 1**。

## 步骤 1 · 采访澄清真实目标

> **⏸ HARD STOP：本步骤必须与用户对话。「自动推进」不覆盖这一步，agent 不能自审通过、不能替用户回答、不能一上来就出方案。**

目的是找到**项目 / 这次改动的真实目标**，而不是照着字面需求闷头做。采访时：

- 偏向**小而隔离的 spec**——把大需求切成边界清晰、能独立验证的小块，别一上来就铺一张大网。
- 关键决策**逼用户显式确认**，不要替他默认，避免漏掉他在意的取舍。
- **最后一个问题固定是**：「还有要补充的吗？」——直到用户明确说没有了，才进入步骤 2。

**问题型开场**（bug 讨论 / 开发问题，如「为什么 X 不对」「你会如何做 X」）进来时，步骤 1 是**诊断 + 澄清你到底想要什么结果**，不必非走完整的项目目标采访——先把问题定位清楚、把期望结果问明白。

**采访 / 诊断完，分叉**（这是给「问题型开场」留的出口，因为它不一定要改代码）：

- 结论 **需要改代码** → 进入步骤 2 出方案。
- 结论 **不需要改代码**（就是想搞懂 / 是预期行为 / 调个配置或用法就能解决）→ 给出诊断、解释或结论，**干净结束**——不强行造 openspec 变更、不写代码。

**快车道（小任务旁路）**：如果一眼看出这是一两个文件、几行就能完成的微改动，不要走完整采访。直接问用户「这个我可以直接改，要不要我直接动手？」——确认后跳过步骤 2-3 的重型方案，直接做改动。改完的去向写死，别留白：

- **默认在当前分支上直接改**，不新建分支（分支策略同步骤 3a：用户明确要求才建）。
- 仍走**步骤 6**（代码反思，对照用户口头需求核一遍）。
- **跳过步骤 7**：微改动没有 `openspec/changes` 文档可归档，不强行造一个。
- **步骤 8 照走**：有受影响的文档就更新 → `git commit`（当前分支）→ 有 remote 则默认停下**询问**要不要提 PR；快车道**没有 proposal.md**，若提 PR，body 用 commit message + 一两句口头描述补充。
- **逃逸口**：动手中发现改动其实不小（碰了可测逻辑、远超几行、牵动多个文件）→ **立即停下回步骤 1** 走完整流程，别在快车道里硬塞。

## 步骤 2 · 在聊天框先出方案

**先把方案写在对话里**给用户看，不要直接落盘。方案要让一个高级程序员看明白整体逻辑即可——写哪几个文件、各自职责、业务线怎么串、有哪些测试案例——不必细到函数签名、数据结构。

两类必须附带的产物：

1. **改页面 / 有视觉版式变化 → 必须出字符图**（ASCII 线框），让用户在写码前就能确认版式。落盘时进 `changes/<id>/wireframes.md`（不进 `design.md`）；基线 MUST 引用 `docs/wireframes/pages/<page>.md`，NEVER 引用其他 change 的 `wireframes.md`。
2. **测试门槛（见下表）触发 → 必须出单元测试用例 + AI 验证流程用例**（AI 验证 = 用 playwright 截图 / 手动跑一遍 / 跑命令看输出这类可由 agent 执行的验证步骤）。

方案出完后, agent 以审稿人视角自审一遍, 没问题就直接进入步骤 3 落盘; 有不确定项才问用户。聊天框方案 + 落盘 + 反思整体合成一个 `方案已定(plan-written)` 状态——落盘和反思不需要用户逐环节拍板，**用户的拍板点在 `方案已定(plan-written)` 之后的默认停**（见步骤 4）。

### 测试门槛

| 改动情形 | 要求 |
|---|---|
| 含**可测逻辑**（纯函数 / 数据转换 / 解析 / 状态机 / 跨文件契约）| **必须有单元测试，与行数无关**——30 行的金额计算也要测 |
| 单文件改动 **diff > 200 行**（任意类型）| **强制停下，逐文件做可测性评估**，在方案里回答每块"要不要测"；若"该测却不方便测"→ 方案**必须**包含拆分 / 重构使其可测的设计 |
| 纯展示 / HTML / CSS / 文案（哪怕 > 200 行且无逻辑）| 记录豁免理由，走 AI 验证流程即可，不强制单测 |

> 为什么"不方便测就改架构"：一段逻辑测不了，几乎总是因为它把业务逻辑和 IO / DOM / 视图耦合在了一起。正确解法是拆开让它可测，而不是给测试开后门。行数（200）只是触发这场评估的警戒线，不是"要不要测"的唯一裁判——别让 250 行的纯 CSS 被逼着写单测。

---

## 步骤 3 · 落盘方案（默认当前分支）

### 3a. 分支策略：默认不新建

落盘前 `git branch --show-current` 看一眼、在进度汇报里带一句当前分支即可。**默认就在当前分支上落盘和写码——包括 `main`**；不新建分支。只有两种情况才建 feature 分支：

| 情况 | 处理 |
|---|---|
| **用户明确要求建分支**（「切个分支做」「开个 branch」）| `git fetch origin && git checkout -b <branch> origin/<基线>` 从最新基线切出；基线取 PR 目标分支（dev 优先，见步骤 8），用户指定则从其指定 |
| **用户明确说了要提 PR，且当前就在 PR 目标分支上**（如在 main 上、PR 要提回 main）| 同上切出 feature 分支——PR 需要一个非目标分支承载提交 |
| 其余（默认）| **不建分支，就地开工** |

**分支命名建议**：用 change-id 当后缀，例如 `change/<change-id>` 或 `feat/<change-id>`——按项目既有约定优先。已有本 change 的 feature 分支（分支名含 change-id，或分支上已有 `openspec/changes/<change-id>/` 痕迹）→ **复用**，别再切一个。

两个仍要停下问的边缘态：**detached HEAD** → 问用户怎么处理，别擅自切；**工作区有大量不相关的未提交改动** → 问用户是先收拾（stash / commit）还是无视继续。

> 为什么默认不建分支：很多仓库本来就是单人 / worktree / 主干直接开发的工作流，强制切分支只会制造一堆没人记得的分支垃圾。分支的意义是承载 PR——而 PR 本身默认就是询问制（步骤 8），所以分支也跟着"明确说才建"。用户真在 main 上又真要 PR 时，commit 还没 push，事后把提交挪到新分支毫无成本（见步骤 8）。

### 3b. 落盘 `openspec/changes/<change-name>/`

按 openspec 规范写：

- 变更提案（proposal）+ 设计权衡（design）+ 任务拆解（tasks）+ spec-delta。
- **方案里有字符图 → 写进 `wireframes.md`**（本项目扩展，按需新建；多页用 `## pages/<page>.md` 小节区分）。**NEVER 塞进 `design.md`**——design 管"怎么实现/权衡/风险"，字符图是版式制品，类型不同。
- **NEVER 在 `tasks.md` 里写「归档时回流 wireframes」这种项目级动作**——归档约定写在 `openspec/changes/AGENTS.md`，每个 change 不重复。

保持和项目已有 openspec 约定一致的文件命名与结构。

---

## 步骤 4 · 反思方案 → 方案已定(plan-written) → 默认停下等口令

落盘后，**以审稿人视角重读方案**找遗漏：

- 有遗漏 → 直接改方案。
- 有不确定项 → 问用户，别自己拍板。
- 没问题 / 问完了 → 整个步骤 2-4 合成的 `方案已定(plan-written)` 状态视为完成。

**`方案已定(plan-written)` 默认停**：把方案要点（含落盘位置）摆给用户，等用户明确说**「开始写代码」**（或等价指令）再进入步骤 5。用户提了别的事按新指令做，该排期的新需求记一条**临时插入的 TODO**别丢。

**除非用户已指定自动**（推进计划里放行了 `方案已定(plan-written)`，如「方案写好后自动开始写代码」「后面全自动」）→ 不停，自动进入步骤 5。

> 为什么默认停：方案是写码前用户唯一的纠偏窗口，方向错了在这儿改是几句话的事，写完码再改是几百行的事。停一下的代价远小于跑偏的代价；真嫌烦，一句「后面全自动」就能放行。

---

## 步骤 5 · 按 spec 写代码

> **若从步骤 0 直接进来（「实施 openspec/changes/X」）**：先确认 `openspec/changes/<X>/` 确实存在且方案完整——它存在意味着方案此前已被确认过，可直接实现；**不存在或残缺 → 回退步骤 1/2** 先把方案补全、确认，再实现。

严格按 `openspec/changes/<change>/` 里的方案写。方案里定义了的测试用例，这一步要真正落地成可运行的测试。

---

## 步骤 6 · 反思代码符合度

> **若从步骤 0 直接进来（「实施完了，检查是否符合方案」）**：先定位对应的 change 方案文档（在 `openspec/changes/` 或已归档的 `archive/`）作为对照基准；**找不到基准 → 问用户指明是哪个 change**，别凭空判符合度。

写完后对照方案逐条核：**代码是否完整实现了方案、有没有偷工或跑偏**。

- 有偏差 / 遗漏 → 回步骤 5 补齐。
- 完全符合、测试通过（单测绿 / AI 验证流程跑通）→ 进入步骤 7。

---

## 步骤 7 · 归档（事实源回流）

闭环的收尾，**不能省**。本步是 `openspec/changes/AGENTS.md` 的「归档」节的执行口径——**MUST 完成下面三步，缺一不算归档**（具体顺序以项目内该文件为准）：

1. **移动 change 目录**：`openspec/changes/<change-id>/` → `openspec/changes/archive/<YYYY-MM-DD>-<change-id>/`（按项目既有归档约定）。
2. **合并 spec-delta → specs**：把 `spec-delta/` 下对应业务域的增删改合并进 `openspec/specs/<domain>/spec.md`——让**行为事实源**反映改完之后的现状。
3. **回流 `wireframes.md` → `docs/wireframes/`**（仅当本 change 有 `wireframes.md` 时）：把字符图回填到 `docs/wireframes/pages/<page>.md`；流转变化同步进 `docs/wireframes/flow.md`——让**版式事实源**反映改完之后的现状。**与第 2 步同等地位，NEVER 省略**。

> 为什么必须做：`openspec/specs` 和 `docs/wireframes/` 分别是项目的**行为事实源**与**版式事实源**。只改代码不更新它们，下一次开发就会基于过时的事实源做决策（画字符图找不到当前基线、读 spec 看到失真规则）。归档是让闭环真正闭上的那一步。
> 项目没有 `docs/wireframes/` 或本 change 没有 `wireframes.md` → 第 3 步天然跳过；不要为不需要回流的 change 强造一个 `wireframes.md`。

---

## 步骤 8 · 更新文档 + commit → 询问是否提 PR

### 8a. commit（到达 已提交(committed)）

- 更新 **AGENTS.md** 里受这次改动影响的部分（模块职责、约定、入口等）。
- `git add` + `git commit` 到**当前分支**（commit message 讲清这次变更做了什么、对应哪个 change）→ ✓ `已提交(committed)`。
- **没有 remote**：commit 后即结束，不问 PR。

### 8b. PR 询问（默认）或直接提（用户明确说过）

有 remote 时，先定**默认目标分支**：`git branch -r` 里有 `origin/dev` → 默认提到 `dev`；没有 → 默认提到 `main`（或项目主干）。**除非用户指定了其他目标分支**。

- **默认：停下询问**。把三件事一起摆给用户：「commit 已完成（在 `<分支>` 上）。要不要提 PR？默认目标 `<dev|main>`。也可以只 push 不开 PR，或先不动。」等用户答复再动。
- **用户此前已明确说要提 PR**（开场「做完提个 PR」、中途点名）：不再问，直接提。**注意：「后面全自动」不算明确要提 PR**——到这儿照样停下问。

**提 PR 的动作**（询问得同意、或明确指示后）：

1. **分支承载**：若当前就在 PR 目标分支上（如提交落在了本地 `main`）→ `git checkout -b change/<change-id>`（新分支带走提交）→ `git branch -f main origin/main`（把本地 main 退回 remote 基线）；若已在 feature 分支 → 跳过。
2. `git push -u origin <branch>` 推分支。
3. 在写 body 之前，**扫一遍当前对话上下文里有没有 issue 编号**（用户消息出现 `#123`、"issue 123"、"修复/关闭/解决 issue 123"、或粘过 GitHub issue 链接）：
   - **有** → PR body **第一行**写 `Closes #<编号>`（多个用逗号分隔或多行），空一行后再接 proposal.md 正文。命令示例：
     ```
     gh pr create --base <dev|main> --title "<commit message 第一行>" \
       --body "$(printf 'Closes #<编号>\n\n'; cat openspec/changes/archive/<YYYY-MM-DD>-<change-id>/proposal.md)"
     ```
   - **没有** → `gh pr create --base <dev|main> --title "<commit message 第一行>" --body "$(cat openspec/changes/archive/<YYYY-MM-DD>-<change-id>/proposal.md)"`。
   **PR body 主体直接粘 proposal.md**，因为 proposal 本来就是 why / what / impact，正好是 PR 描述，无需二次编辑。
4. 把 PR URL 回报给用户。
5. **NEVER 自己合并 PR**——合并由 reviewer 拍板。

> 为什么 PR 默认询问而不是默认提：提 PR 是外向动作——推到 remote、进团队的 review 队列、可能触发 CI 和通知。默认替用户做外向动作，做对了省一句话，做错了要撤 PR、删分支、跟队友解释。询问一句的成本远低于此；真想全托管的用户，开场一句「最后提 PR」就能跳过询问。
> 为什么 dev 优先：有 `dev` 分支的仓库通常走「feature → dev → main」的集成流，直接对 main 提 PR 会绕过集成层。没有 dev 就退回 main。用户指定压倒一切。
> 为什么 body 直接粘 proposal.md：proposal 已经讲清了 why（动机）、what changes（改了什么）、impact（影响范围），这就是 PR description 该写的内容。再让 agent 二次概括只会失真，不如直接粘。

## 失败路径

- **不确定是不是开发任务 / 进场点判不准** → 先问用户一句确认意图，别默认进流程或猜进场点。
- **openspec/ 缺失** → 停在步骤 0，主动提议并（经同意后）引导初始化，NEVER 擅自搭骨架改仓库结构。
- **没出方案就被要求写码** → 没有 `方案已定(plan-written)` 就不可能写码。提醒用户先出方案、回步骤 2；除非用户明确说"跳过方案直接写"。
- **拿不准用户的"自动"覆盖到哪** → 按窄的算：只解除 `方案已定(plan-written)` 停点，PR 照问、分支照旧不建。PR 和分支只认"明确点名"。
- **测试该写却"不方便测"** → 不是跳过测试的理由，而是回到方案改架构使其可测（步骤 2 测试门槛）。
- **写到一半发现方案有硬伤** → 停下回步骤 2 修方案、重新确认，别将错就错继续写。
- **detached HEAD / 工作区有大量不相关未提交改动** → 停下问用户怎么处理，别擅自切分支或 stash。
- **commit 落在 main、用户在 PR 询问时才说要提 PR** → 不慌：提交还没 push，按步骤 8b 的「分支承载」把提交挪到新分支、本地 main 退回 `origin/main`，再 push 开 PR。
- **没有 GitHub remote、`gh` 用不了**（用 GitLab / Gitea / 自建）→ 用对应平台的 PR / MR 命令（如 `glab mr create`），或把分支推上去后让用户去网页开 PR；NEVER 因此跳过询问自作主张。

## 防错红线

- NEVER 跳过方案直接写代码——`方案已定(plan-written)` 必经，方案必须落盘到 `openspec/changes/`。
- NEVER 在 `方案已定(plan-written)` 默认停时，用户没说"开始写代码"（且没提前放行自动）就动代码。
- NEVER 在没有 openspec 的项目里自己搭骨架——停下引导 project-init-docs。
- NEVER 因"不方便单测"就跳过测试——改架构使其可测。
- NEVER 漏掉闭环收尾：写完代码必须按步骤 7 完成归档——移动 change、合并 spec-delta、**有 `wireframes.md` 时必须回流到 `docs/wireframes/`**——再走步骤 8 更新 AGENTS.md。漏回流 wireframes 等于让版式事实源失真，与漏合并 spec-delta 同等严重。
- **NEVER 未经用户明确要求就新建分支**——「后面全自动」不含建分支；只有用户点名建分支、或明确要提 PR 且必须有分支承载时才建。
- **NEVER 未经询问同意或用户明确指示就 push / 开 PR**——「后面全自动」不含 PR；`已提交(committed)` 后有 remote 的默认动作是**问**，不是提。
- NEVER 自己合并 PR——合并由 reviewer 拍板。
- NEVER 在步骤 0 不明示推进计划就闷头进入步骤 1——用户没机会指定"在哪儿停 / 放行哪儿 / 加事"，默认停点和自动衔接都成了黑盒。
- NEVER 把"自动推进"理解成"连步骤 1 采访也跳过"——`采访中(interviewing)` 是 HARD STOP，对话往返本身是 `目标确认(interview-confirmed)` 状态的兑现过程，没法替用户答。

<example>
用户：「给卡片导出加一个『一键导出全系列』的功能。」

流程（默认形态，两处默认停）：
1. 步骤0：是开发任务；仓库有 `openspec/` → 继续。**明示推进计划（A 形态）**：
   > 推进计划（步骤 1 采访会自然停下等你回答；方案落盘后停下等你说「开始写代码」；commit 后停下问要不要提 PR，若提默认目标 dev）：
   > `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) ⏹停⏹ 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) ⏹问PR⏹`
2. 步骤1（`采访中(interviewing)`）：采访——「全系列指当前预览里的所有卡片还是某个目录下的？导出命名规则？失败的卡片是跳过还是中断？」最后问「还有补充吗？」用户确认无补充 → ✓ `目标确认(interview-confirmed)`。
3. 步骤2：聊天框出方案——新增 `scripts/export-all.mjs`（遍历系列、复用单卡导出逻辑），改 `app.html` 加一个按钮（**有视觉变化 → 附字符图，基线引用 `docs/wireframes/pages/app.md`**）。导出逻辑含可测的文件名生成 / 系列遍历（**可测逻辑 → 附单测用例**：文件名生成、空系列、部分失败），再附 AI 验证流程（playwright 跑一遍导出、截图核对产物数量）。
4. 步骤3：`git branch --show-current` 显示在 `main`——用户没要求建分支也没说要提 PR → **就在 main 上落盘**，写入 `openspec/changes/add-export-all/`（proposal + design + tasks + spec-delta + **wireframes.md 含字符图**）。
5. 步骤4：反思方案，发现没定义"部分失败时是否继续"——这是不确定项，**回去问用户**、得到回答后补进方案 → ✓ `方案已定(plan-written)`。**默认停**：把方案要点摆给用户，等口令。用户说「开始写代码」→ 进步骤 5。
6. 步骤5-6：按 spec 实现 → ✓ `代码写完(code-written)`；单测绿、playwright 验证导出数量正确 → ✓ `验证通过(code-verified)`。
7. 步骤7：按 `openspec/changes/AGENTS.md` 的「归档」节三步并列——① `add-export-all/` 移到 `archive/2026-07-20-add-export-all/`；② 导出能力的 delta spec 合进 `openspec/specs/export/`；③ `wireframes.md` 的字符图回流到 `docs/wireframes/pages/app.md`（按钮变更同步进 `flow.md` 相关流程节）。
8. 步骤8：更新 AGENTS.md 的脚本清单 → `git commit`（当前分支 main）→ ✓ `已提交(committed)`。有 remote、`git branch -r` 里有 `origin/dev` → **停下问**：「commit 已完成（在 main 上）。要不要提 PR？默认目标 dev。也可以只 push 或先不动。」用户说「提吧」→ `git checkout -b change/add-export-all`（提交挪到新分支）→ `git branch -f main origin/main` → push → `gh pr create --base dev …`（body 粘 proposal.md）→ ✓ `已开PR(pr-opened)`，回报 PR URL。**不自己合并**。

用户：「为什么我删一个操作员，下面删除预览里却列出了一大串操作员？我们先讨论清楚再决定动不动手——别急着写码。」

流程（问题型开场，可能不改代码）：
1. 步骤0：是 bug 讨论、仓库有 `openspec/` → 路由进 **步骤 1**（不是从需求起步）。用户 prompt 里说「先讨论清楚」≈ 跑到 `目标确认(interview-confirmed)` 就停，**明示推进计划**：
   > 推进计划（按你要求在 `目标确认(interview-confirmed)` 后停下，先把诊断结果摆给你再决定要不要改）：
   > `采访中(interviewing) → 目标确认(interview-confirmed) ⏹停⏹ 方案已定(plan-written) ⏹停⏹ 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) ⏹问PR⏹`
2. 步骤1（`采访中(interviewing)`）：诊断——读删除预览的查询逻辑，定位到"预览把同组关联操作员一并列出"。澄清期望结果：「你是觉得根本不该显示这些，还是显示但要标清它们是关联项？」
3. 分叉（→ ✓ `目标确认(interview-confirmed)`）：
   - 用户说「哦原来是关联项，预期的，不用改」→ **给出解释、干净结束**，不写 openspec、不改代码。
   - 用户说「应该只显示目标本人」→ 结论=需要改 → 进 **步骤 2** 出方案，走正常闭环（`方案已定(plan-written)` 和 PR 两处默认停照旧）。
</example>

<example>
用户：「加个批量删除，方案我就不看了，后面全自动跑完。」

流程（B 形态——放行 `方案已定(plan-written)`，但 PR 不在自动范围）：
1. 步骤0：**明示推进计划（B 形态）**：
   > 推进计划（按你要求方案落盘后自动开始写码，一路跑到 commit；提 PR 不在自动范围，commit 后仍会停下问你）：
   > `采访中(interviewing) → 目标确认(interview-confirmed) → 方案已定(plan-written) → 代码写完(code-written) → 验证通过(code-verified) → 已提交(committed) ⏹问PR⏹`
2. 步骤1 采访照停（HARD STOP，「全自动」不覆盖采访）→ 步骤2-7 自动衔接，不建分支、就在当前分支干 → `已提交(committed)`。
3. 有 remote → **仍然停下问要不要提 PR**——「后面全自动」不等于「要提 PR」。用户若一开始就说「全自动跑完最后提 PR 到 dev」，这一步才不问直接提。
</example>

<bad-example>
用户：「给卡片导出加一个『一键导出全系列』的功能。」（没说自动、没说提 PR、没说建分支）

WRONG 的做法：
- 直接开始写 `export-all.mjs` 和改 `app.html`，跳过采访和方案。
- 方案落盘后不停，自顾自开始写代码——用户没说「开始写代码」也没放行自动。
- 自作主张 `git checkout -b feat/export-all` 切了个分支——用户没要求建分支。
- commit 后不问，直接 `git push` + `gh pr create`，甚至自己 merge——PR 询问被跳过。

为什么错：跳过方案 = 用户没机会在写码前纠偏；`方案已定(plan-written)` 默认就是停点，抢跑等于把用户唯一的纠偏窗口焊死；建分支和提 PR 都是"明确说才做"的动作，默认做会制造分支垃圾、把没经用户同意的东西推进团队的 review 队列。这个 skill 的默认形态是「两处停：方案确认一次、PR 询问一次」，抢跑任何一处都是越权。
</bad-example>

<bad-example>
用户：「加个导出功能，后面全自动。」

WRONG 的做法：
- 把「全自动」理解成连采访也跳过，一上来就出方案写码。
- 把「全自动」理解成含 PR：commit 后不问，直接 push + 开 PR。
- 把「全自动」理解成含建分支：先切了个 `feat/export` 再开工。
- 反过来矫枉过正：明明用户说了全自动，`方案已定(plan-written)` 后还停下问「要开始写码吗？」。

为什么错：「全自动」解除的**只有** `方案已定(plan-written)` 这一个默认停点——采访是 HARD STOP（对话往返没法替答），PR 和建分支是"明确点名才做"（外向动作 / 改仓库拓扑，不因一句概括性的"全自动"就被授权）。反方向同理：用户已明确放行的停点还硬停，是把控制权设计降级回"逐步问"。**自动的范围，宽了越权、窄了烦人，边界就在状态枚举表里，照表执行。**
</bad-example>
