前段时间，我们在做一个内部工具时，让我们用的大模型帮我们生成一段数据处理的脚本。模型给出了一个不太对的回复，导致我们后续花了不少时间在排查上。这篇文章记录一下整个过程。

## 背景

最近我们内部有一个数据归档相关的需求，团队里有人提出可以试试用大模型来生成 SQL 脚本。我们之前在别的场景里用过类似做法，效果一般，但这次任务比较小，就决定试一下。任务的大致内容是把某个表里旧的记录搬到归档库去。

## 第一反应：以为是模型的语法问题

不久前的某天上午，我们让模型生成了第一版脚本。脚本看起来语法没什么问题，跑了一遍 staging 也没报错，但同事在 review 的时候觉得好像 WHERE 条件写得有点怪。我让模型自己再看一眼，它表示没有问题。我又让另一个同事看，他也说看起来 OK。

但事后回头看，那段 SQL 里 OR 和 AND 的优先级处理是有问题的——只是当时数据分布让 bug 没有暴露出来。

## 再查：怀疑是数据分布问题

后来我们又跑了几次，曾经一度以为是数据分布问题。在某次重新构造测试数据的过程中，我们才意识到原来真正的问题不在数据，而在 SQL 本身。模型在生成的时候没有把那个括号显式打出来，但它自己 review 的时候也没看出来。

我们之后又把 prompt 重新整理了一次，让模型在第二次生成时同时给出 EXPLAIN 的预期结果。结果它给的 EXPLAIN 也是凭空瞎编的——模型并没有真的去执行那段 SQL，只是按它的训练数据里类似查询的统计推测了一个数字。

## 第三次：怀疑是 prompt 模板问题

我又把 prompt 模板调了几个版本。从前我们一直用一个比较短的模板，这次我把它改长了大概一倍，加了很多约束。模型按新模板给的结果在某些 case 上正确率提升了一点，但在边界情况下还是会写错。

我们让模型把它认为最容易出错的几种 case 列出来，它列了一段，我们对照看，发现里面有些是对的，有些是它编的——它把 PostgreSQL 的语义和 MySQL 的语义混在一起讲。

## 真相

排查到后面，我们才反应过来真正的问题是：模型对 SQL 操作符优先级的内部表示和我们以为的不一样。它不是不知道，是在没有显式约束的情况下，它会优先选择更"自然"的写法，而那个写法在边界 case 上恰好是错的。

我们把这个问题提了一个 issue，标题大概是"模型在多条件 WHERE 上的括号缺失"，目前还没有被很好地解决。

## 结论与思考

这次踩坑让我们更深刻地理解了，AI 大模型在写 SQL 这件事上虽然看起来很厉害，但在涉及到优先级、边界条件这些细节时，仍然需要工程师做兜底。最终我们还是回到了"模型生成 + 人工 review + EXPLAIN 校验"的传统组合。

我们后续会持续探索更好的工程实践，希望能在保留模型生成效率的同时，把这类隐藏 bug 的概率压到最低。
