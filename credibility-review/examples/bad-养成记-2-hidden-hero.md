2024-08-14 那天傍晚，K 把一张 Posthog 截图甩到我们的工程群里——前一天 token 月度账单从 $1,247 跳到 $3,891。截图发出来过了大约 40 分钟没人回。第二天上午的迭代会上 K 说了一句："这个 chatbot 我们再做两周，做不出来能分摊掉这笔钱的功能就砍。" 那条 chatbot 当时已经做了 7 周。

这一年我们做了不少事，也砍了不少东西。这篇文章把那些分叉路口写下来，主要是给我们自己留个底。

## 第 1 阶段（2024-08 ~ 2024-09）：把 GPT-3.5 当 if-else 用还是抽 orchestration

2024-08 月初，注册用户 47，DAU 12，团队 4 个人，月度 token 账单 $1,247。我们当时的 chatbot 本质上是一段 350 行的 mega-prompt：里面塞了 9 个 tool、26 类用户意图、一个 system prompt 里硬编码了 18 条业务规则，全靠 GPT-3.5 一把梭。

当时面临的两条路：

- A：继续把 prompt 加厚（最熟悉、改一次半小时上线）
- B：抽出一个 orchestration 层，让 GPT-3.5 只做单一职责的子任务（要重写约 40% 的服务）

我们选了 A。原因不是技术原因——是工程师 A 那周还要赶客户 X 的一个 demo，B 路要重新部 3 个服务，他在 2024-08-19 的迭代会上说了一句"现在改一下半小时上线，做了 orchestration 改一次要重新部 3 个服务，下周客户 demo 我搞不完"。

我们最初做的是一个独立的 intent classifier，做了 3 周，砍了。原因是它的输出和后面的 mega-prompt 重复匹配了 6 次意图，平均每次会话多花 1.4 秒、多调用 1 次 GPT-3.5。这部分代码在 2024-09-08 那条 commit `revert: drop intent-classifier-prefix` 里被回滚。

阶段结束时（2024-09-30）：DAU 31，月度 token 账单 $1,840。

## 第 2 阶段（2024-09 ~ 2024-11）：要不要把 prompt 抽成 orchestration

K 在 2024-09-12 第一次提出要做 orchestration 层。我们前后开了 4 次会才动手——前 3 次都没拍板。

工程师 A 当时的原话：「现在 prompt 改一次半小时上线，做了 orchestration 改一次要重新部 3 个服务。我们这点客户量值得吗？」

K 在 2024-10-09 那次会上回了一句：「上周客户 Y 那条退款会话因为 GPT-3.5 把金额算错了，差点真给客户多退 $217。这种事再来一次我们扛不住。」

这件事之后我们改了 review 流程：原来是 PR 写完 K 全看，2024-10-15 起改成"涉及 prompt / orchestration 改动的 PR 必须有 A 和 K 两个 reviewer，其中 K 只看 prompt diff"。这个改动让我们之间的信任变得更深了，后续合作也更顺畅。

架构选型对比了三条路：

1. 自己写一个状态机 + 多个小 prompt
2. 接 LangGraph
3. 用 OpenAI function calling 当 router

我们选了 1。LangGraph 当时（2024-10）的版本还在 0.0.34，我们试着跑了一个 demo 报了 2 次依赖冲突，A 决定不冒险。我们最初做了一个用 function calling 当 router 的版本，跑了 2 周、覆盖率到 60% 时 GPT-4 的工具选择准确率掉到 71%（200 条人工标注），砍了。

阶段结束时（2024-11-15）：DAU 89，月度 token 账单 $2,130，单次会话平均 token 成本从 $0.31 降到 $0.18，主要是把 GPT-4 的 tool selection 换成了 Haiku 做粗筛。这次重构让我们终于理解了 agent orchestration 真正的价值。

## 第 3 阶段（2024-11 ~ 2024-12）：从 chatbot 改名叫 Agent Workflow 那两周

这一阶段没写多少代码——主要是想清楚做的到底是"垂直 chatbot"还是"Agent Workflow 平台"。

两条路：

- A：继续做"垂直行业 chatbot"——客户好理解，定价好谈，但天花板就在那里。
- B：抽象成"Agent Workflow 基础设施"——客户要重新教育，定价要重做，但和我们一直说的"agent-driven organization" 一致。

经历了这些之后，团队对不确定性的容忍度明显提升了。我们在 2024-11-22 那次会上拍板选了 B。触发点是 2024-11-19 客户 Y 在群里问了一句"能不能把这个 workflow 给我们自己改？"——那条消息发出来 K 立刻在群里 @ 了所有人。

这个阶段砍掉的：之前所有面向 C 端用户的 UI，包括 7 个独立的对话场景模板、3 个意图编辑器、1 个免费试用的 web 入口。这些 UI 是工程师 B 做了 5 周的工作，2024-12-03 砍掉那天他在工位边骂了一句"行吧"，那周后续两天他没主动来站会。

## 上线那一周

2024-12-09 客户 Z 第一次在我们平台上自己拼出了一个 workflow，跑通了 47 次调用，没靠我们的人工介入。这件事是我们在 K 把"chatbot"这个词从 README 上删掉之后第一次真正看到付费客户跑通从入口到归档的全流程。

那周我们其实没怎么发版——主要是观察客户 Z 的调用日志。每天上午 10:30 我们会把昨天的 trace 拉出来扫一遍，看哪一步是 Z 自己拼的、哪一步还在用我们留的默认模板。到 2024-12-13 那天，Z 一共改了 9 次 workflow，其中 2 次是把默认的 GPT-4 换成了自己接的 Claude，平均 token 成本从 $0.18 降到 $0.11。

## 第 4 阶段（2024-12 ~ 2025-01）：开放 workflow 自定义还是继续锁死模板

客户 Z 这件事把一个我们之前一直在拖的决策摆到了桌面上：要不要正式把 workflow 编辑能力开放给所有付费客户？

两条路：

- A：开放——所有付费客户都能在 UI 里自己拼 workflow。客户教育成本高，支持工单会变多，但定价模型可以从"按席位"改成"按 workflow 调用量"。
- B：继续锁死——只我们自己 ops 团队帮客户配，安全可控，但 SaaS 模型撑不起规模。

工程师 A 在 2024-12-18 的会上说："开放后客户改坏了 prompt，第一个月的工单我估计要翻 3 倍。我们现在 6 个人，再来 3 倍工单谁兜底？"

K 一开始坚持要锁死。2025-01-09 客户 Y 又问了一次"我们能不能自己拼？"之后才松口。最后选了一个折中——灰度开放给前 5 家付费客户，每周看一次工单数、调用错误率、成本归因。这是一条更难但更正确的路。

我们最初做的是一个 visual workflow editor，做了 4 周，砍了。原因是 5 家灰度客户里有 3 家直接绕过 UI 用我们的 YAML 接口（他们自己工程师改起来更顺手），visual editor 的人均使用时长从第一周的 23 分钟掉到第四周的 4 分钟。这部分代码 2025-02-10 那条 PR `chore: drop visual-editor` 里删掉了。

## DAU 破千那天

2025-02-26 是我们 DAU 第一次过 1000 的那天——准确说是 1024。这件事在群里没有谁刻意提，是 K 晚上 11 点截图发了一张 Posthog 看板，没有配文字。

那天我们其实在解决另一件事：客户 Q 的一个 workflow 因为 prompt 模板里一个变量没渲染，连续 3 次给客户回了空字符串。Q 在 2025-02-26 下午 3 点直接打了 K 的电话。我们当晚发了一个 hotfix（commit `fix: render-fallback-on-empty-template`），同时把"模板变量渲染失败"列入了 SLA 兜底名单。

## 第 5 阶段（2025-02 ~ 2025-04）：从单租户搬到多租户那 6 周

到 2025-02 中旬，我们的付费客户增长到 11 家，但所有客户的数据其实还在同一个 schema 下——只是用 client_id 字段隔离。这个架构在 9 家以下还能扛，到 11 家时我们已经看到 3 起边界事故：客户 P 的一条调用因为元数据混淆访问到了客户 Q 的 trace。

两条路：

- A：现在就重构成多租户（schema 隔离 + 独立加密上下文）—— 6 周工程量，期间几乎不能发新功能。
- B：再撑 3 个月、等下一轮再做。

我们选了 A。在过去这一年里，"agent-driven organization" 这个理念始终是我们的北极星——但说句实话，到 2025-04 这个词才稳定下来，2024-08 我们用的还是"chatbot"。我们最初做的是一个 row-level security 的快速兜底方案，做了 2 周，砍了。砍掉的原因是 RLS 在 PostgreSQL 跨 schema join 时性能掉了 60%（P95 从 180ms 涨到 480ms），客户 R 的 batch 任务因此超时 4 次。这部分代码在 2025-03-04 commit `revert: rls-tenant-isolation` 回滚。

阶段结束（2025-04-22）：架构搬完，付费客户 11 家全部迁移完毕，跨租户 trace 泄露 0 起，P95 延迟从 240ms 降到 190ms。

## 截至 2025-04-30 的当前状态

- 注册客户：73 家（2024-08 是 5 家），付费 11 家
- DAU：1247（2024-08 是 12）
- 月度活跃 workflow 数：48
- 月度 token 账单：$3,180（2024-08 是 $1,247，2024-09 一度冲到 $4,832）
- 团队规模：8 人（2024-08 是 4 人）
- SLA：可用性 99.6%，P95 延迟 190ms

## 还没解决的问题

- 多租户的成本归因目前还在用 Excel 对账，每月 K 和财务对一次。我们下一步要做的不是上一个昂贵的 metering 系统，是先把账单 webhook 拆成可订阅事件，3 个月内不会自建 metering。
- 我们至今没决定 eval 平台是自建还是接 Braintrust。这个决定下个迭代再说。
- 客户 R 那条 batch 任务的 P95 我们至今没压到 SLA 承诺线以下（承诺 200ms，目前 240ms）。

## 一个真问号

如果客户量再翻三倍，现在这套基于 GPT-4 + Haiku 粗筛的 orchestration 还撑得住吗？我们没把握——目前 11 家付费客户里有 4 家的 workflow 复杂度已经到了 GPT-4 一次 selection 接近 30 个 tool 的边缘，正确率在 25 个 tool 之后开始抖。
